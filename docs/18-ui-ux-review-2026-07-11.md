## Evidence vocabulary

- **Render-observed** — reproduced in the production Vite build against the real local API and
  `runs/` fixture.
- **Code-confirmed** — the behavior or absence is directly established by the current React/CSS/API
  implementation.
- **Existing-test-confirmed** — covered by a test that was run during this review.
- **Inference / recommendation** — a product or design conclusion, not a claim about current runtime
  behavior.
- **Not verified** — explicitly outside the completed test matrix.

Severity means user/product impact, not implementation difficulty: **P0** is unsafe or materially
misleading control behavior; **P1** blocks a dependable primary journey; **P2** creates recurring
friction or poor comprehension; **P3** is polish.

# PART I — CURRENT EXPERIENCE

## 1. Executive verdict

LoopLab already has an unusually capable control plane: run discovery, a grouped search DAG, a live
timeline, per-node evidence, reports, trust diagnostics, hypotheses, cross-run analysis, global and
per-run settings, and a persistent assistant. The product is no longer missing a UI shell. Its main
problem is that the shell exposes more capability than its interaction and state model can safely
carry.

The current experience has four structural weaknesses:

1. **The UI can promise a state that is not true.** “Read-only” share is a normal writable run URL;
   time travel still leaves live mutations available; Compare can display stale node details under a
   newly selected node ID; network failures are frequently presented as empty or loading states.
2. **Workspace geometry collapses under ordinary pressure.** Inspector, timeline, assistant, two
   navigation rows, and graph compete for the same viewport. At 360×800 the DAG width becomes `0px`
   and the document becomes `683px` wide; at 768×1024 the DAG gets `342px` while Inspector keeps about
   `426px`.
3. **Visual semantics are not reliable across themes or safety states.** Paper makes several report
   labels nearly disappear. Trust uses a positive green treatment for “no scan recorded” and
   “detector off,” visually equating absence of evidence with a pass.
4. **The UI is predominantly mouse-first.** Many visually interactive surfaces are generic `div`,
   `span`, table, or SVG elements; dialogs are not exposed as dialogs; custom tabs, splitters, charts,
   project rows, timeline rows, and run cards do not form a coherent keyboard model.

**Product conclusion:** do not start with a cosmetic reskin. First make read/write state truthful,
make failures explicit, and establish responsive/accessibility primitives. Then simplify the
information architecture and visual hierarchy. Cosmetic polish applied before those foundations will
make an unsafe interface look more authoritative.

## 2. Audit scope and method

### 2.1 Render fixture

The production UI was rebuilt and served through the real LoopLab preview server. The principal
fixtures were:

- the runs home with **57 runs** (55 unassigned, 2 in a project);
- the cross-run Map containing the same 57 runs;
- finished run `demo`: **19 nodes**, 19 evaluated, 0 failed, **71 events** (`seq` 0–70);
- node `#14`, the current champion;
- Dark default and Paper themes;
- bottom, side, and full assistant modes;
- Report, Search/DAG, Inspector, Overview, Queue, Hypotheses, Trust, Compare, and global Settings.

Responsive render checks used **1280×720**, **1024×768**, **768×1024**, and **360×800** CSS
viewports. The 360px pass is a mobile/reflow test, not a claim that a real phone/touch stack was
tested. Actual browser zoom, touch input, and mobile browser chrome were not simulated. Geometry was
captured during an ephemeral manual browser session; screenshots and DOM dumps were not committed.
A follow-up regression suite should therefore keep a reproducible viewport/fixture manifest and
golden screenshots instead of treating these measurements as a permanent baseline.

### 2.2 Code audit

The review covered all files in `ui/src/`, with concentrated inspection of:

- routing and state: `App.jsx`, `hooks.js`, `api.js`;
- runs/projects/map: `RunList.jsx`, `MapView.jsx`;
- run workspace: `RunView.jsx`, `Dag.jsx`, `Inspector.jsx`, `Dock.jsx`;
- assistant: `AssistantBar.jsx`, `AssistantChat.jsx`, `SharedAssistant.jsx`;
- settings: `Settings.jsx`, `SettingsForm.jsx`, `settingsSchema.js`;
- reports/panels/charts: `Report.jsx`, `panels.jsx`, `charts.jsx`;
- visual system: `styles.css`, `format.js`, `icons.jsx`.

Missing functionality was counted as missing only after checking both the rendered navigation and the
implementation/API seam. A feature being hard to discover is recorded separately from a feature that
does not exist.

### 2.3 Build evidence

`npm run build` completed with Vite 5.4.21: 224 modules, approximately 95.4 kB CSS (~18.0 kB gzip),
and a ~636 kB JavaScript chunk (~201 kB gzip). The rendered-baseline build reported duplicate
`resume`/`run_abort` object keys in `Dock.jsx` plus Vite's >500 kB chunk warning. Upstream commit
`13f087c` removed the duplicate keys; the reconciled `369d6a6` build is 636.04 kB (201.22 kB gzip)
and emits only the chunk warning.

The remaining bundle warning is not by itself a performance failure; it justifies measuring first-load
and interaction cost and introducing route/panel splitting. Generated hashes and last-decimal sizes are
evidence, not a budget.

### 2.4 Explicit limitations

Not verified in this pass:

- screen readers, switch control, voice control, or a full keyboard-only completion of every journey;
- axe-core, automated contrast tooling, or browser-native accessibility trees beyond DOM snapshots;
- Firefox, Safari, real iOS/Android, touch gestures, or actual 200%/400% browser zoom;
- shared JupyterHub origin, real viewer credentials, RBAC, or hostile multi-user deployment;
- a live long-running run, approval-blocked run, disconnected SSE session, injected HTTP failures, or
  a 10k/50k-event performance fixture;
- the complete Genesis plan → editable spec → repo-scout/adaptation checklist → validation → launch
  journey; this pass verified `/new` discovery, pending/cancel behavior, and the implementation surface,
  not a successful end-to-end launch;
- print-to-PDF output, A4/Letter pagination, or a real download/share recipient;
- all six themes in screenshots (all six were code-audited; Dark and Paper were rendered).

Claims depending on those conditions are marked code-confirmed or recommended, not render-verified.

## 4. What should be preserved

The redesign should preserve these strengths rather than flattening the product into a generic admin
dashboard:

- **Conclusion-first Report.** The champion, improvement, caveats, evidence, and reproducible solution
  are ordered well.
- **Files/events remain the truth.** The UI is a read/control plane, not a second engine.
- **Graph grouping and semantic zoom.** Theme/operator/metric/niche grouping, group collapse, and LOD
  are useful foundations; the default viewport behavior is the problem.
- **Visible uncertainty.** “Unconfirmed,” caveats, trust mode, and detector state are exposed in text.
- **One persistent assistant across surfaces.** Bottom/side/full continuity is valuable when state,
  geometry, and mode are explicit.
- **Separation of stop/finalize/resume.** The product has the right conceptual distinctions even when
  control placement and confirmation need work.
- **Power-user depth.** Queue, hypotheses, trust, sensitivity, importance, cross-run, registry,
  artifacts, config, and authoring should remain available through progressive disclosure.

# PART II — DETAILED AUDIT

## 5. P0 findings — truth and control safety

### SAFE-01 — “Copy read-only share link” produces a writable control-plane URL

| Field | Value |
|---|---|
| Evidence | **Code-confirmed**, high confidence |
| Surface/journey | Collaboration → share/review |
| Impact | A recipient who can reach the deployment receives the normal run route and its mutations, contrary to the UI promise |
| Effort | L–XL; requires server authorization, not a label change |
| Dependency | Doc 17 authorization/reference-monitor work |

`panels.jsx:795-803` copies the normal `#/run/<id>` route. `App.jsx:22-48` resolves that route to the
ordinary `RunView`, including assistant, config, reset, node actions, and control transport. There is
no viewer credential or read-only route.

**Recommendation:** create an explicit viewer capability/token and route. Hide writes for clarity,
but enforce the boundary at the server; client-side hiding is not authorization.

**Acceptance:** a share-credential E2E may read DAG/report/artifacts and receives a server rejection
for every control/config/assistant mutation. Network recording proves zero successful write requests.

### SAFE-02 — time travel renders history while retaining live mutations

| Field | Value |
|---|---|
| Evidence | **Code-confirmed**, high confidence |
| Surface/journey | Timeline → historical replay → inspect/act |
| Impact | A user can believe an action applies to the viewed historical state while it mutates the current run |
| Effort | M |
| Dependency | Central write-capability state |

`RunView.jsx:206-207` switches the displayed projection to historical state, while live mutation
paths remain mounted through DAG/Inspector/Dock and panels (`RunView.jsx:281-285,290-317,320-339`;
`Report.jsx:86-93`).
The header's `replay` text is not a sufficient mode boundary. There is a second state-integrity gap:
when `viewSeq` changes, the old `hist` snapshot is not cleared before the new fetch
(`RunView.jsx:86-93`), so the slider can say sequence N while the DAG still shows the previous
sequence until the response arrives.

**Recommendation:** make historical mode read-only by construction. Add a persistent “Viewing seq N”
banner and one action, “Return live to act.” Do not let individual components decide independently.

**Acceptance:** replay mode renders no enabled mutation control and issues no mutation request.
Attempting a command either remains read-only or explicitly returns to live before asking for action
confirmation. Historical state is keyed as `{requestedSeq,resolvedSeq,status,data,error}`; changing
sequence immediately clears or marks the prior data stale, and the UI never labels a snapshot whose
`resolvedSeq` differs from `requestedSeq` as the selected sequence.

### CTRL-01 — assistant and Dock can report different outcomes for the same control

| Field | Value |
|---|---|
| Evidence | **Code-confirmed through `369d6a6`** and inherited from doc 16 P1-10, high confidence |
| Surface/journey | Assistant/Dock/TUI → stop/finalize/resume/abort |
| Impact | The UI can announce success although the authoritative engine transition did not occur |
| Effort | M–L |
| Dependency | One server-owned control command/state-transition contract |

`AssistantBar.jsx:32-40,299-302` implements direct `/resume`, `/finalize`, and `/abort` by appending a
control event and immediately flashing success. The generic control route returns after the append
(`looplab/serve/routers/control.py:79-130`). Dock's corresponding paths append the event **and** call
`resumeRun` so an exited engine is actually spawned (`Dock.jsx:558-559`). The same user intent can
therefore say “resumed” or “finalizing” without producing the same runtime effect. Commit `3170d85`
expanded `api.js:67-82` and added a TUI/JS needs-engine registry parity test, but direct
`AssistantBar.runDirect` does not use that batch-plan registry. This is partial remediation, not the
shared operation/postcondition contract required here.

**Recommendation:** route Web buttons, assistant commands, and TUI commands through one server-owned
operation that returns a command ID and observable lifecycle. The client may acknowledge receipt, but
must not present completion until the authoritative state transition is folded/observed.

**Acceptance:** one `ControlSpec`/server command service owns stop/finalize/resume/abort semantics.
Parity tests cover **Assistant, Dock, and TUI × live, paused, stalled/engine-exited, finished, and
finalizing**. Every cell produces the same accepted/rejected event and process postcondition; success
appears only after that postcondition is observed, while timeout/failure remains actionable.

### DATA-01 — Compare can label stale details as the newly selected node; “diff” is unordered

| Field | Value |
|---|---|
| Evidence | **Render-observed** layout + **code-confirmed** race/diff, high confidence |
| Surface/journey | Analysis → Compare nodes |
| Impact | A user can make a scientific conclusion from code/metrics that belong to another node |
| Effort | M |
| Dependency | Shared async resource-state pattern |

`panels.jsx:943-980` neither clears old detail on selection nor binds a response to the selector that
requested it. A late A response can overwrite C after A→B→C selection. `Inspector.jsx:815-819`
compares sets of lines, so order and duplicates disappear and changed lines can be displaced from
their original position.

The rendered Compare modal also has unlabeled selectors, small lower-case `diff`, raw code without
line numbers, and a whole-modal horizontal scrollbar. The command palette can remain above the modal
and obscure its content.

**Acceptance:** detail state is `{nodeId,status,data,error}`; stale responses are ignored; old data is
hidden immediately; selectors are labeled; an ordered line diff preserves duplicate and reordered
lines; race and golden-diff tests cover these cases.

## 6. Detailed priority findings — P1 unless marked P2

### STATE-01 — failures are routinely rendered as empty, missing, or forever loading

**Evidence:** code-confirmed, high confidence. **Effort:** L. **Dependency:** one shared async-state
contract and route-level error boundaries.

Examples:

- Runs converts fetch failure to `[]`, then can say “No runs” (`RunList.jsx:162-164`).
- Map swallows both fetch errors and leaves a blank canvas (`MapView.jsx:125-132`).
- Settings can stay at “Loading settings…” forever (`Settings.jsx:31-36,103`).
- Before any state arrives, an invalid run or initial SSE failure can remain “Connecting…”
  (`RunView.jsx:205`, `hooks.js:54-100`). After a first state, the hook can expose an offline condition;
  this finding is specifically the unclassified pre-state path.
- Inspector, report code, Dock log/trace, cross-run trajectories, and shared chat similarly collapse
  error into blank/loading/no-data states.
- There is no app/route Error Boundary.

**Recommendation:** standardize `idle | loading | success-empty | success | error | offline | stale`.
Every error needs a plain-language cause, retry, retained last-known data when safe, and a diagnostic
detail affordance. Empty must mean a successful empty response.

**Acceptance:** fixture tests inject timeout, 401/403, 404, 409, 500, invalid JSON, SSE disconnect,
and stale cache for every primary surface; none is labeled empty or successful.

### RESP-01 — the interface does not reflow; mobile can eliminate the DAG entirely

**Evidence:** render-observed + code-confirmed, high confidence. **Effort:** XL. **Dependency:** target
layout model and accessible drawers/splitters.

Measured behavior:

| Viewport | Observed geometry/result |
|---|---|
| 1280×720 | Side assistant at ~440px plus Projects at 280px leaves the list main around 560px; filters and header wrap |
| 1024×768 | DAG `598×323`; Inspector about 426px; timeline 260px; assistant bar 45px |
| 768×1024 | DAG width `342px`; Inspector still about `426px`; graph controls wrap and obscure nodes |
| 360×800 run | Document width `683px`; DAG width **`0px`**; Inspector fills the run workspace; horizontal scrollbar appears |
| 360×800 home | Projects remains 280px; run cards shrink to about `30px` content width and grow to 300–400px tall |

The stylesheet has no general responsive system: one hypothesis-board breakpoint and a
reduced-motion block. Sidebar, Inspector, assistant side panel, headers, tables, and splitters keep
desktop assumptions (`styles.css:106-130,460,623-625,1344,1488-1491,1535-1537`;
`RunView.jsx:49-66`).

**Recommendation:** define desktop, compact desktop, tablet, and mobile-read-only compositions.
Below the compact threshold, Inspector/timeline/assistant become mutually exclusive drawers; Projects
collapses; run cards become a single-column list; the graph never receives less than its minimum
functional width. Splitter alternatives must be keyboard operable.

**Acceptance:** 360×800, 768×1024, 1024×768, 1440×900, and real 200%/400% zoom have no page-level
horizontal scroll, zero-width primary content, obscured CTA, or overlapping layer. Mobile read-only
supports runs → run → node → report without a mutation control.

### VIS-01 — cross-run Map's default fit makes the entire view unreadable

**Evidence:** render-observed with 57 real runs, high confidence. **Effort:** M–L.

At 1280×720 the initial Map fits all runs and groups into one almost horizontal line near the center
(`MapView.jsx:22-25,125-140`);
cards become a few pixels wide and labels cannot be read. Most of the canvas is empty. The control
offers zoom and fit but no usable default framing, search, active filter summary, legend, minimap
overview of meaningful scale, or staged disclosure.

**Recommendation:** start at project/task clusters, not all leaf runs. Preserve a readable minimum
node size, collapse large groups, show a list/map selection handshake, and make “fit selected” the
primary action. Reconcile the existing graph-grouping proposal rather than replacing its layout work.

**Acceptance:** the 57/500/5,000-run fixture manifest contains at least two top-level groups. On first
paint the default level shows **10–40** run/cluster targets, each at least **24×24px**, with labels at
least **11px** and no label overlap; their bounding box uses at least **40%** of usable canvas width and
**30%** of height. Lower levels aggregate or hide labels instead of shrinking them. After entering a
name in search, a user can focus/open that run in at most **three interactions** without blind zooming;
0/1-run and error states are explicit.

### VIS-02 — the run workspace allocates space to panels instead of the user's current task

**Evidence:** render-observed + code-confirmed, high confidence. **Effort:** L.

On Search, an empty Inspector consumes roughly one third of the desktop. Selecting a node adds six
mouse-first tabs. Timeline fixes another 260px and the assistant another 45–60px. The graph-grouping
toolbar floats over nodes. At 1280×720 the graph is visually small; at compact widths it becomes
nonfunctional. Side chat does not coordinate with Inspector or Dock and can leave all three open.
The competing fixed/flex allocations are visible in `RunView.jsx:49-66,290-317`,
`AssistantBar.jsx:110`, and the responsive anchors cited in RESP-01.

**Recommendation:** adopt task-priority rules:

1. Graph is the default primary canvas.
2. Inspector opens contextually and returns space when closed or empty.
3. Timeline, assistant, and Inspector cannot all claim permanent desktop columns/rows below a
   threshold.
4. User-resized dimensions persist per surface, with “reset layout” and keyboard resize controls.

**Acceptance:** at 1024×768 the usable DAG remains at least **480×320px**; if that cannot be preserved,
opening side chat converts Inspector/timeline to mutually exclusive drawers or asks for an explicit
layout choice. No primary canvas silently falls below the minimum.

### VIS-03 — light themes fail contrast and semantic consistency

**Evidence:** Paper render-observed; all six themes code-confirmed. **Effort:** M.

The Paper report visibly loses the `/new` header token, the caveat message, several chart labels, and
muted labels. The problem is not the shared `--fg-mut` token by itself: code contains dozens of
specific hard-coded color literals outside the theme-token layer;
the exact count depends on whether SVG decoration, repeated literals, and fallback colors are included.
Examples include:

- report gold `#ffd98a` against Paper surfaces (approximately 1.06:1 in the affected state);
- chart label `#e6e9ef` against Paper/White Sci-Fi (approximately 1.01–1.09:1);
- fixed light text in `.chip.ok/.alarm/.warn` on light surfaces;
- white hover text on Old Green's bright primary (~1.30:1);
- dark fixed chart/grid/minimap colors across light themes.

Trust compounds the issue: “no leakage scan recorded” and “detector off” are rendered as green
positive chips. That is semantically wrong even when contrast passes.

**Recommendation:** make charts and semantic chips token-driven. Separate `pass`, `warn`, `unknown`,
`disabled`, and `not-run`; do not derive them from a binary green/red model.

**Acceptance:** automated contrast matrix for six themes: text ≥4.5:1, large text/UI graphics ≥3:1;
status meaning is available in icon/text, not color alone; not-run/disabled never uses pass styling.

### VIS-04 — undefined `--border` silently removes structure

**Evidence:** code-confirmed, high confidence. **Effort:** S. **Priority:** P2.

`--border` is not defined in the root tokens (`styles.css:2-28`) but is used without fallback for
hypothesis cards/columns, the Why strip, node context menu, and MultiTrajectory axes
(`styles.css:421,426,432,439`; `charts.jsx:322-323`). Those declarations are invalid.

**Acceptance:** define the intended token or replace it with `--line`/`--line-2`; a theme screenshot
test confirms every affected border/axis in all six themes; a CSS custom-property lint prevents
unknown tokens.

### A11Y-01 — interaction semantics are not a coherent accessibility model

**Evidence:** DOM render observations + code census, high confidence. **Effort:** XL.

The static census found dozens of `onClick` handlers on non-button elements (`div`, `span`, `tr`,
`th`, `g`, `circle`, `path`) and very sparse explicit ARIA. The exact count is intentionally omitted:
it changes with multiline JSX and whether a non-native element already has an explicit role. Examples:

- Inspector and Settings tabs are clickable `div`s without `role=tab`, `aria-selected`, or arrow-key
  navigation;
- run-card bodies, projects, timeline rows, chart marks, artifact rows, assistant sessions, and
  disclosures are mouse-first;
- settings labels are not programmatically bound to controls;
- custom checkboxes hide the native input with `display:none`;
- panels/modals are generic containers without `role=dialog`, `aria-modal`, focus trap/restore, or a
  common Escape contract;
- splitters are drag-only;
- toasts/live SSE changes are not announced;
- chart tooltips depend on hover and have no table alternative;
- several targets are 18px and `chipFontSize` can fall to 8px;
- reduced-motion does not cover all pulse/blink/cursor animations.

Representative anchors include Inspector tabs (`Inspector.jsx:97-100`), settings labels and custom
checkboxes (`SettingsForm.jsx:32-63,81-87`), generic modal shells (`panels.jsx:12-20`), and compact
interactive styling (`styles.css:417,771-780`).

**Acceptance:** keyboard-only smoke completes find run → open → inspect node → compare → report;
focus is visible and restored after dialogs; axe has no critical/serious issues; screen-reader testing
confirms names/order/live announcements. Pointer targets meet WCAG 2.2 2.5.8's **24×24 CSS px**
minimum or a documented exception; primary mobile touch actions use **44×44px** targets.

### SAFE-03 — destructive actions use incompatible and sometimes absent safeguards

**Evidence:** code-confirmed, high confidence. **Effort:** M–L.

Immediate/no-undo paths include drag-to-merge, context-menu reset/kill, Finalize, permanent hypothesis
delete, and chat deletion. Other paths use native `confirm/alert`. Approval, reset, resume, finalize,
abort, delete, and merge therefore have different visual language and consequence disclosure.
Representative mutation anchors are `Dag.jsx:372-449`, `Dock.jsx:556,623-628`,
`panels.jsx:871-875,914-915`, and `AssistantBar.jsx:284-289,654-657`.

**Recommendation:** one themed confirmation pattern with object, consequence, destructive primary
label, pending state, double-submit protection, and an undo/compensating action where possible.
Drag-to-merge should preview the merge and require explicit commit.

**Acceptance:** maintain a complete mutation inventory and assign every item one policy tier:
reversible (immediate + undo), consequential (explicit confirm), or irreversible/high-risk (object,
scope, consequence, and re-auth/typed confirmation where warranted). Contract tests prove every
inventory item uses its tier and cover cancel/confirm, double click, network failure, partial success,
and retry. Audit history records who/what initiated the action when identity exists.

### DATA-02 — “best metric” sorting compares incomparable values and reverses minimize tasks

**Evidence:** code-confirmed, high confidence. **Effort:** S–M.

`RunList.jsx:203-214` sorts a raw metric number without task or objective direction. The initial sort
key is time; after the user selects metric, the retained descending direction puts worse results first
for `min` tasks. “All tasks” compares unrelated metrics. Registry also sorts descending and mixes
tasks (`panels.jsx:624-627`).

**Acceptance:** metric sort is task-scoped or grouped by task; direction follows `min/max`; the
toolbar states the objective; missing metrics sort consistently; tests cover mixed tasks and both
directions.

### PERF-01 — timeline reloads and renders the full log on every live sequence

**Evidence:** code-confirmed; performance magnitude not measured. **Effort:** L. **Priority:** P2
until benchmarked.

`Dock.jsx:466-470` refetches the complete log on `liveSeq`; `Dock.jsx:599-604` renders every event;
`Dock.jsx:540-544` forces scroll-to-bottom even while the user may be reading history.

**Recommendation:** cursor/incremental tail, virtualization, stable anchoring, and a “N new events”
badge unless the user was already at the end.

**Acceptance:** on a 50k-event fixture, the rendered event-row DOM stays at **≤400**, p95 filter/input
response is **<100ms**, and live-update work stays **<32ms/frame** on the agreed reference machine.
After initial sync, each update transfers only the cursor delta. Incoming events never steal scroll
from a reader, and reconnect does not duplicate rows.

### UX-SET-01 — Settings exposes engine completeness without progressive disclosure or safe editing

**Evidence:** render-observed + code-confirmed, high confidence. **Effort:** L–XL.

There are 9 groups and 128 fields. At 1280px the tabs already wrap to two rows; help text is dense and
small; the sticky Save/Defaults row plus assistant reduces the working height. Field/group/page dots
do indicate changes, but there is no search, Basic/Advanced view, dependency-driven visibility, or
consolidated cross-group summary of deviations with effective value and source
(`Settings.jsx:38-62,96-123`; `SettingsForm.jsx:24-28,66-93`).

Number inputs lack schema min/max and inline validation; integer parsing can truncate; leaving Back
can discard edits; backdrop-click can dismiss authoring/config; global settings and secret save can
partially succeed while reporting a generic failure.

**Recommendation:** searchable settings with Recommended/Advanced modes, dependency visibility,
effective-value/source display, schema validation, dirty-state guard, and transactional/explicit
partial-save results.

**Acceptance:** every field has label, help, default/effective value, validation, and keyboard path;
invalid combinations cannot save/launch; leaving dirty state requires a choice; partial secret save
states exactly what persisted.

### UX-REPORT-01 — export promises exceed the implemented print experience

**Evidence:** Report render-observed + print implementation code-confirmed; PDF output not rendered.
**Effort:** M. **Priority:** P2 until an actual PDF render establishes higher impact.

`Print / PDF` calls `window.print()`, but `styles.css` has no `@media print`. Navigation, dark/light
chrome, assistant, overflow, tables, and page breaks are uncontrolled. Report also does not visibly
mark an agent narrative stale when `rep.at_node` predates the current node count.

**Acceptance:** A4 and Letter golden PDFs have white print surfaces, hidden controls, repeated table
headers, deliberate page breaks, no clipped chart/code, and report provenance/staleness metadata.
With `currentNodeCount = Object.keys(state.nodes || {}).length`, a narrative is visibly stale exactly
when `rep.at_node < currentNodeCount`; refresh success clears the marker only after an equal/newer
report is observed.

### UX-HITL-01 — blocking approval is hidden behind a command

**Evidence:** code-confirmed. **Effort:** S–M.

The approval banner (`RunView.jsx:249-257`) tells the user to type `/approve #id` or `/ratify`; it
provides no visible Approve/Ratify action for a blocking workflow. Slash commands should accelerate a
visible workflow, not be its only path. Reject/Defer would require explicit server semantics and is a
separate product decision, not claimed here as an existing UI-only gap.

**Acceptance:** the banner identifies request, scope, risk, and consequence and provides explicit
buttons subject to the same permission/confirmation policy as commands.

### UX-NAV-01 — navigation mixes tabs, dropdowns, modals, and overlays without a layer contract

**Evidence:** render-observed, high confidence. **Effort:** L. **Priority:** P1 for layer/focus
contract; P2 for IA naming and tooltip/content mismatch.

The run header has Search/Report/Overview styled as peers, but Overview opens a modal over Search.
The header tooltip says Overview contains a budget bar, while the rendered Overview shows only six
summary cards. A second row contains four dropdown hubs plus Settings. Multiple layers remain open:
Progress and command palettes can coexist; the command palette can render above Compare and obscure
code. Modals are not focus-trapping dialogs (`RunView.jsx:225-277,320-339`; `panels.jsx:12-20`).

**Recommendation:** establish three primitives with distinct appearance and behavior:

- route/tab: replaces the primary work surface;
- popover/menu: transient, closes on outside action/Escape/another menu;
- dialog/drawer: modal or contextual task, traps/restores focus and suppresses unrelated overlays.

**Acceptance:** only one transient menu is open; opening a dialog closes menus and assistant
suggestions; z-index tokens are documented; Overview is either a real tab or visibly a summary
dialog, and its copy matches its content.

### UX-GEN-01 — Genesis proposes a launch card that the user cannot edit

**Evidence:** code-confirmed; successful launch not render-verified. **Effort:** M. **Priority:** P1.

The guide promises an editable spec whose task/backend/settings can be reviewed and changed before
launch. `AssistantChat.jsx:97-125` instead renders name, rationale, and settings as text/chips with a
single Start button. A 409 says the run already exists and tells the user to “rename it,” but there is
no rename control. Repo-scout/adaptation details likewise have no structured correction path in the
card.

**Recommendation:** use a real editable LaunchCard with sections for run identity, task/data/repo,
evaluation contract, backend/model, budget/policy, and adaptation checklist. Validate locally and at
the server, preserve the assistant's proposal as provenance, and let the user ask the assistant to
revise a field without discarding manual edits.

**Acceptance:** plan → edit → validate → launch E2E covers run-name conflict, invalid path, missing
metric, incompatible settings, backend unavailable, pending launch, server error, retry, and cancel.
Every guide claim about editable fields is backed by a control in the card.

### UX-ASSIST-01 — assistant continuity is strong, but mode, focus, and geometry are ambiguous

**Evidence:** render-observed + code-confirmed. **Effort:** L. **Priority:** P1 for context/geometry;
P2 for mode and icon polish.

Observed details:

- Projects' prominent `+ New` means “New project”; a new run is `/new` in the bottom bar.
- On home, the slash menu also offers run-only `/stop`, `/finalize`, `/resume`, `/pause`, `/abort`.
- The bottom bar hides its active Plan/Ask/Auto-edit/Auto mode; modes appear only in side/full.
- The slash menu has no active option/arrow-key navigation; partial commands can do nothing.
- Side view at 1280px narrows the main enough to wrap header/filter rows; it does not negotiate with
  Projects or Inspector (`AssistantBar.jsx:110,590-638`).
- Full view is mostly blank for a short exchange and uses tiny symbolic controls.
- A stopped `/new` exchange correctly exposes a stop control and `(stopped)`; this is user

**Acceptance:** one primary New Run CTA focuses `/new`; commands are context-filtered; bar shows mode;
palette supports arrows/Enter/Escape and accessible option semantics. Draft text, attachments, caret,
and the active session survive bar ↔ side ↔ full transitions and navigation; focus moves to the

responsive workspace rules; stopped turns offer Regenerate, while failed turns show a reason and
retry the same turn without duplicating user input.

# PART III — TARGET EXPERIENCE AND DELIVERY

## 9. Target interaction principles

1. **Truth before reassurance.** Empty, pass, unknown, disabled, stale, and offline are different
   states with different visual semantics.
2. **One surface owns an action.** Every action has a visible canonical home; commands and shortcuts
   accelerate it.
3. **Context earns space.** Empty Inspector, idle timeline, and inactive assistant do not permanently
   tax the graph.
4. **History is a capability boundary.** Viewing the past is read-only unless the user returns live.
5. **Progressive disclosure, not feature deletion.** Common work is visible; specialist panels remain
   one predictable layer away.
6. **Task/objective compatibility is explicit.** Metrics are never ranked across incompatible tasks or
   directions without normalization and disclosure.
7. **Every visual explanation has a text/data equivalent.** Graphs, charts, color, and hover do not
   become the sole carriers of evidence.
8. **The URL describes the review state.** A shared diagnostic link opens the same node, tab, panel,
   and sequence within the viewer's permissions.

## 10. Recommended information architecture

### 10.1 Runs home

- Primary action: **New run**. Secondary action inside Projects: **New project**.
- Collapsible Projects rail; on mobile it becomes a drawer.
- Search + Filters button + active filter chips; advanced filters live in a popover/drawer.
- Saved views/favorites and a task-scoped comparison mode.
- List/Map preserves the same selection and filters.
- Run card: status + name + task, one-line outcome, best metric with objective direction, last update,
  one semantic open target, separate overflow actions.

### 10.2 Run workspace

Recommended hierarchy (names should be validated with users):

- primary: **Explore** (DAG + contextual Inspector), **Timeline**, **Report**;
- contextual Analyze drawer: Compare, sensitivity, importance, cross-run;
- contextual Rigor drawer: Trust, data quality, Pareto/diversity;
- contextual Work drawer: Queue, hypotheses, research, failures, artifacts;
- run actions: status-aware primary action plus one overflow menu;
- assistant: explicit mode, context attachments, and responsive bar/drawer/full behavior.

Overview should be either a true primary surface with budget/progress or a clearly styled summary
dialog. It should not look like a selected route while leaving Search underneath.

### 10.3 Visual hierarchy

- Use one 4/8px spacing scale, one radius scale, and declared elevation/layer tokens.
- Keep primary body text at a durable size/contrast; metadata may be quieter but not 8px.
- Reduce simultaneous accent colors. Reserve green/red/yellow for evaluated semantic states; use
  neutral accent for selection/navigation.
- Standardize icon + label rules. Symbols alone are acceptable only for universal, labeled controls.
- Give charts a shared grammar: title, objective direction, unit, baseline, uncertainty, legend,
  accessible table, loading/empty/error.

## 12. Acceptance metrics

- 0 successful mutation requests from share-viewer and replay fixtures.

- In an unmoderated first-use test, ≥90% of participants start New Run in **≤15 seconds** and
  **≤2 interactions** without learning slash syntax.
- Finished run → champion evidence → code → report is completable keyboard-only.
- Node/panel/sequence diagnostic state survives reload and can be deep-linked.

- WCAG contrast targets across six themes; no axe critical/serious findings.
- 360×800 through 1440×900 and 200%/400% zoom: no page horizontal scroll or hidden primary action.
- Graph/table/chart has an accessible alternative and non-color distinction.
- Reduced-motion removes all nonessential continuous animation.

### Performance

Reference profile: headless Chromium, 4× CPU slowdown, 1.6 Mbps down / 150ms RTT, cold cache; report
the median of five runs.

- Initial-route JavaScript is **≤170 kB gzip**, the largest lazy panel is **≤75 kB gzip**, and the
  primary route is usable in **≤2.5s** with total blocking time **<200ms** on the reference profile.
- At 50k events: ≤400 mounted event rows, p95 filter/input response <100ms, live update <32ms/frame.
- At 5k runs: p95 search/filter input response <100ms on the same reference machine.
- After initial sync, incoming SSE/timeline traffic is cursor-delta only and does not steal reader
  scroll or trigger a full-list rerender.

## Final verdict

LoopLab's UI has enough breadth to be a serious research control plane. The next step is to make that
breadth **truthful, spatially stable, and operable without hidden knowledge**. Ship the P0 truth
boundaries first; establish common async/accessibility/layout primitives second; then reorganize the
navigation and add expert accelerators. Preserve the report, graph grouping, event provenance, and
assistant continuity, but stop allowing panels, colors, and labels to imply stronger guarantees than
the underlying state provides.

# PART IV — 2026-07-12 REMEDIATION DISPOSITION

## 17. How to read this reconciliation

Parts I–III are the evidence ledger for the original rendered baseline. Their observations and line
references are intentionally not rewritten as if they had been made against the remediation branch.
This part is the current-state overlay.

Status vocabulary:

- **Fixed (UI scope)** — the branch closes the user-visible finding and has named code plus test or
  browser evidence. This is not a claim of server authorization unless explicitly stated.
- **Partial** — the main journey improved, but at least one acceptance condition remains open.
- **Code fixed / validation gap** — the implementation is present and unit/build checks pass, but the
  full interaction, scale, platform, or assistive-technology matrix has not been automated.
- **Open** — the branch does not materially close the finding.

The branch was deliberately kept separate from `master`. The disposition below describes the feature
branch only; it is not a release or production-deployment claim.

## 18. What materially changed

### 18.1 Workspace geometry and responsive behavior

- The search canvas is now the primary surface. An empty Inspector no longer consumes a permanent
  desktop column.
- At `<=900px`, Inspector becomes a focus-trapped overlay drawer and Timeline starts collapsed; at
  `<=600px`, the temporary surfaces become full-width sheets. The canvas is not resized to zero.
- Projects becomes a modal drawer on compact layouts, with backdrop, `Escape`, inert closed state,
  focus trap, and focus restoration.
- Side Assistant overlays the workspace below the wide breakpoint rather than subtracting another
  440px from the graph.
- Inspector and Timeline splitters use Pointer Events, keyboard arrows, `role="separator"`, value
  metadata, focus rings, and resize clamping. Saved desktop pane sizes are clamped on resize.
- The home cards, toolbar, Compare columns, Code Viewer, Report, and Trust tables have compact-layout
  fallbacks. `100dvh` is used for temporary mobile surfaces.

Primary files: `RunView.jsx`, `RunList.jsx`, `AssistantBar.jsx`, `hooks.js`, `styles.css`,
`resource-states.css`, and `useDialogFocus.js`.

### 18.2 History is now a capability boundary

- Requesting a sequence immediately enters `loading(requestedSeq)` and hides live/previous content
  until that exact prefix fold resolves. Failed requests cannot retain an older snapshot under a new
  sequence label.
- A persistent banner identifies `seq N of max`, read-only state, and the only exit: **Return to live**.
- A selection that does not exist at the requested sequence is cleared. Merge arms, Compare seed,
  live panels, and stale compact Inspector state are cleared when History starts.
- Central `api.js` mutation guards reject every `/api/runs/:id` non-GET request while that run is in
  History. Dock, Inspector, Report, and Assistant also remove their local mutation affordances.
- Historical node detail uses `/nodes/:id?seq=N`. The server returns a real prefix fold, reports the
  live `max_seq` separately, and does not stamp present-time `engine_running` into the old state.
- Unversioned sidecars — trace, operation trace, streaming metrics, logs-as-live-detail, refresh, and
  Assistant run tools — are hidden or disabled rather than presented as historical evidence.
- Scrubber timers and the client access registry are cleaned on live return and unmount, preventing a
  delayed callback from re-locking a later live route.

Primary files: `runMode.js`, `api.js`, `RunView.jsx`, `Dock.jsx`, `Inspector.jsx`, `Report.jsx`,
`AssistantBar.jsx`, `looplab/serve/appstate.py`, and `looplab/serve/routers/runs.py`.

This closes SAFE-02 in the current UI scope. It is a client safety boundary, not a substitute for the
still-missing server-side read-only share credential in SAFE-01.

### 18.3 List and Map now describe the same set

- List owns one polled run/project resource; Map no longer independently fetches a second, stale set.
- Project membership is authoritative from `run.project_id`. Query, project, task, status, and
  super-task filters remain visible and produce the same count in both representations.
- Metric sorting is disabled until one task is selected, then follows the task's `min`/`max`
  direction with explicit **best/worst first** semantics and missing values last.
- Status now distinguishes `running`, intentional `paused`, `approval`, unexpected `stalled`, and
  `finished` instead of treating every stopped engine as stalled.
- Project regions use bounded grid packing. Large groups become super-nodes; Unassigned is an explicit
  synthetic region; expanding a cluster preserves the user's zoom/pan. A MiniMap and scope summary
  expose framing and counts.
- Run labels lead; technical IDs are secondary. Run, region, project-tree, and cluster targets are
  keyboard operable. React Flow's wrapper focus is disabled where an inner semantic target exists.
- If project metadata fails, Map shows an error with Retry instead of silently moving every run into
  Unassigned.

Primary files: `runIndex.js`, `runMapModel.js`, `RunList.jsx`, `MapView.jsx`, and `groupnodes.jsx`.

### 18.4 Inspector, Compare, and code comprehension

- Node detail resources carry `{nodeId,status,data,error}`; selection clears stale detail immediately,
  ignores late responses, and shows loading/error/Retry rather than interpreting a trimmed summary as
  “no code.”
- Code Viewer adds line numbers, search, Wrap, Copy, helper-file views, accessible region labels, and
  ordered positional diff. Duplicate lines and insert/delete positions are preserved.
- Compare is a labelled modal dialog with focus trap/restoration and labelled Left/Right selectors.
  A failed side produces an explicit error/Retry in both column and diff modes.
- LCS diff work is memoized by the two source strings so unrelated SSE frames do not allocate the
  matrix again.
- Inspector tabs are real tabs. Historical Inspector exposes only Overview, Code, Trust, and Cost;
  pipeline stages become static historical evidence instead of promising a no-op re-run.
- DAG context menus are clamped to the viewport.

Primary files: `CodeViewer.jsx`, `lineDiff.js`, `Inspector.jsx`, `panels.jsx`, and `Dag.jsx`.

### 18.5 Assistant and New Run

- The home has one primary **New run** CTA. It opens the persistent Assistant's proposal/launch flow;
  `/new` remains an accelerator rather than required prior knowledge.
- Slash suggestions are a combobox/listbox with Arrow Up/Down, Enter, Escape, active option, and
  `aria-activedescendant`. Run-only direct commands are absent when no run is open.
- Provider exceptions render as fixed, allow-listed error cards with safe Retry/Open Settings actions.
  URLs, routed model/provider names, account IDs, and credential fragments are never interpolated into
  the card or preview.
- New server responses persist `error_kind` plus safe copy. Reads also sanitize legacy persisted
  exception-shaped messages, so reload/share is as safe as the live stream.
- Retry preserves the exact in-memory attachment snapshot. A reloaded turn whose file content is no
  longer available does not silently retry a different request.
- History disables the composer, permission cards, direct commands, file attachment, launch cards,
  edit undo, and Retry; defensive action guards remain behind the hidden controls.

Primary files: `AssistantBar.jsx`, `AssistantChat.jsx`, `assistantErrors.js`,
`looplab/serve/assistant.py`, and `looplab/serve/routers/assistant.py`.

### 18.6 Settings, Report, Trust, and system states

- Settings defaults to a curated 17-field Essential cross-section, with All and catalogue-wide search.
  It reports matches, unsaved edits, customized values, and hidden edits; help is progressive rather
  than always expanded.
- Every settings field now has stable `id`, associated `label`, described help, accessible native
  checkbox, secret state, and keyboard tab navigation. Loading and connection-test failures are
  explicit and retryable without exposing raw provider detail.
- Report and Trust use `passed`, `warning`, `unknown`, and `failed` semantics. “Not scanned,” detector
  off, absent drift evidence, and single-evaluation winners are neutral/unknown — never green success.
- Report no longer says “trustworthy” when detector coverage is missing. Undefined feasibility is
  `unknown — not established`, not `yes`. `manual · manual-probe` is separated correctly.
- Winning-code, report-refresh, Trust-config, run-list, invalid-run, Inspector-detail, Compare-detail,
  and Timeline resources now have explicit loading/error/empty/Retry states.
- Paper/White Sci-Fi foreground tokens and scoped Report/Trust surfaces were strengthened. Print mode
  removes app chrome and Assistant, releases fixed scroll containers, wraps code, and restores table
  layout. `--border` is now a defined design token.

Primary files: `Settings.jsx`, `SettingsForm.jsx`, `settingsModel.js`, `Report.jsx`, `report.js`,
`trustSemantics.js`, `panels.jsx`, `styles.css`, `settings-polish.css`, and
`report-trust-polish.css`.

## 21. Remaining functionality, dependency ordered

1. **True review/share capability** — viewer credential, server-enforced read-only routes/tools,
   expiry/revocation, and a read-only E2E (SAFE-01).
2. **One command service** — stop/finalize/resume/reset/approval must have one server-owned lifecycle,
   idempotency key, postcondition, and Assistant/Dock/TUI parity tests (CTRL-01).
3. **Destructive-action contract** — consistent risk tiers, confirmation, progress, result, undo where
   possible, and a generated mutation inventory (SAFE-03).
4. **Editable Genesis** — validate and edit run name, source/task, settings, readiness, and conflicts
   before spending tokens; preserve the New Run Assistant entry point (UX-GEN-01).
5. **Timeline at scale** — cursor/delta API, virtualization, stable scroll ownership, and a 50k-event
   performance fixture (PERF-01).
6. **Deep-linkable review state** — run/node/tab/panel/filter/sequence in the URL so a diagnostic link
   opens the exact evidence under discussion.
7. **Accessible component contract** — shared DataTable/Chart alternatives, axe plus keyboard smoke,
   screen-reader pass, ≥44px touch targets where appropriate, reduced motion, and real zoom/reflow.
8. **Notification and approval inbox** — opt-in finish/approval/failure alerts with deep links and a
   visible in-product pending-actions destination.
9. **Collaboration semantics** — authenticated author/time/edit/resolve history rather than anonymous
   notes and a normal control-plane URL.
10. **Performance budgets** — code-split heavy panels/React Flow, measure first-load/interaction cost,
    and keep scale fixtures in CI before claiming portfolio-scale Map/Timeline.

## 22. Reconciled verdict

The branch materially changes the product's visual perception because it changes what owns space,
what colors are allowed to mean, and whether a visible control is truthful. The largest gains are not
decorative: History is isolated from live actions, the graph keeps the workspace, Map remains readable,
Compare and code are inspectable, Settings has a novice path, Trust no longer turns missing evidence
green, and failures explain how to recover.

It is still not correct to mark the whole audit “fixed.” True read-only sharing, unified run-control
postconditions, destructive-action policy, editable Genesis, Timeline scale, deep links, collaboration,
and formal accessibility/visual regression remain substantial product work. Those are now the next
layer rather than being obscured by broken geometry and misleading state semantics.
