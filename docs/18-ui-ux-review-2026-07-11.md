## Evidence vocabulary

- **Render-observed** — reproduced in the production Vite build against the real local API and
  `runs/` fixture.
- **Code-confirmed** — the behavior or absence is directly established by the current React/CSS/API
  implementation.
- **Existing-test-confirmed** — covered by a test that was run during this review.
- **Inference / recommendation** — a product or design conclusion, not a claim about current runtime
  behavior.
- **Not verified** — explicitly outside the completed test matrix.

Severity means user/product impact, not implementation difficulty: **P0** is unsafe or materially
misleading control behavior; **P1** blocks a dependable primary journey; **P2** creates recurring
friction or poor comprehension; **P3** is polish.

# PART I — CURRENT EXPERIENCE

## 1. Executive verdict

LoopLab already has an unusually capable control plane: run discovery, a grouped search DAG, a live
timeline, per-node evidence, reports, trust diagnostics, hypotheses, cross-run analysis, global and
per-run settings, and a persistent assistant. The product is no longer missing a UI shell. Its main
problem is that the shell exposes more capability than its interaction and state model can safely
carry.

The current experience has four structural weaknesses:

1. **The UI can promise a state that is not true.** “Read-only” share is a normal writable run URL;
   time travel still leaves live mutations available; Compare can display stale node details under a
   newly selected node ID; network failures are frequently presented as empty or loading states.
2. **Workspace geometry collapses under ordinary pressure.** Inspector, timeline, assistant, two
   navigation rows, and graph compete for the same viewport. At 360×800 the DAG width becomes `0px`
   and the document becomes `683px` wide; at 768×1024 the DAG gets `342px` while Inspector keeps about
   `426px`.
3. **Visual semantics are not reliable across themes or safety states.** Paper makes several report
   labels nearly disappear. Trust uses a positive green treatment for “no scan recorded” and
   “detector off,” visually equating absence of evidence with a pass.
4. **The UI is predominantly mouse-first.** Many visually interactive surfaces are generic `div`,
   `span`, table, or SVG elements; dialogs are not exposed as dialogs; custom tabs, splitters, charts,
   project rows, timeline rows, and run cards do not form a coherent keyboard model.

**Product conclusion:** do not start with a cosmetic reskin. First make read/write state truthful,
make failures explicit, and establish responsive/accessibility primitives. Then simplify the
information architecture and visual hierarchy. Cosmetic polish applied before those foundations will
make an unsafe interface look more authoritative.

## 2. Audit scope and method

### 2.1 Render fixture

The production UI was rebuilt and served through the real LoopLab preview server. The principal
fixtures were:

- the runs home with **57 runs** (55 unassigned, 2 in a project);
- the cross-run Map containing the same 57 runs;
- finished run `demo`: **19 nodes**, 19 evaluated, 0 failed, **71 events** (`seq` 0–70);
- node `#14`, the current champion;
- Dark default and Paper themes;
- bottom, side, and full assistant modes;
- Report, Search/DAG, Inspector, Overview, Queue, Hypotheses, Trust, Compare, and global Settings.

Responsive render checks used **1280×720**, **1024×768**, **768×1024**, and **360×800** CSS
viewports. The 360px pass is a mobile/reflow test, not a claim that a real phone/touch stack was
tested. Actual browser zoom, touch input, and mobile browser chrome were not simulated. Geometry was
captured during an ephemeral manual browser session; screenshots and DOM dumps were not committed.
A follow-up regression suite should therefore keep a reproducible viewport/fixture manifest and
golden screenshots instead of treating these measurements as a permanent baseline.

### 2.2 Code audit

The review covered all files in `ui/src/`, with concentrated inspection of:

- routing and state: `App.jsx`, `hooks.js`, `api.js`;
- runs/projects/map: `RunList.jsx`, `MapView.jsx`;
- run workspace: `RunView.jsx`, `Dag.jsx`, `Inspector.jsx`, `Dock.jsx`;
- assistant: `AssistantBar.jsx`, `AssistantChat.jsx`, `SharedAssistant.jsx`;
- settings: `Settings.jsx`, `SettingsForm.jsx`, `settingsSchema.js`;
- reports/panels/charts: `Report.jsx`, `panels.jsx`, `charts.jsx`;
- visual system: `styles.css`, `format.js`, `icons.jsx`.

Missing functionality was counted as missing only after checking both the rendered navigation and the
implementation/API seam. A feature being hard to discover is recorded separately from a feature that
does not exist.

### 2.3 Build evidence

`npm run build` completed with Vite 5.4.21: 224 modules, approximately 95.4 kB CSS (~18.0 kB gzip),
and a ~636 kB JavaScript chunk (~201 kB gzip). The rendered-baseline build reported duplicate
`resume`/`run_abort` object keys in `Dock.jsx` plus Vite's >500 kB chunk warning. Upstream commit
`13f087c` removed the duplicate keys; the reconciled `369d6a6` build is 636.04 kB (201.22 kB gzip)
and emits only the chunk warning.

The remaining bundle warning is not by itself a performance failure; it justifies measuring first-load
and interaction cost and introducing route/panel splitting. Generated hashes and last-decimal sizes are
evidence, not a budget.

### 2.4 Explicit limitations

Not verified in this pass:

- screen readers, switch control, voice control, or a full keyboard-only completion of every journey;
- axe-core, automated contrast tooling, or browser-native accessibility trees beyond DOM snapshots;
- Firefox, Safari, real iOS/Android, touch gestures, or actual 200%/400% browser zoom;
- shared JupyterHub origin, real viewer credentials, RBAC, or hostile multi-user deployment;
- a live long-running run, approval-blocked run, disconnected SSE session, injected HTTP failures, or
  a 10k/50k-event performance fixture;
- the complete Genesis plan → editable spec → repo-scout/adaptation checklist → validation → launch
  journey; this pass verified `/new` discovery, pending/cancel behavior, and the implementation surface,
  not a successful end-to-end launch;
- print-to-PDF output, A4/Letter pagination, or a real download/share recipient;
- all six themes in screenshots (all six were code-audited; Dark and Paper were rendered).

Claims depending on those conditions are marked code-confirmed or recommended, not render-verified.

## 4. What should be preserved

The redesign should preserve these strengths rather than flattening the product into a generic admin
dashboard:

- **Conclusion-first Report.** The champion, improvement, caveats, evidence, and reproducible solution
  are ordered well.
- **Files/events remain the truth.** The UI is a read/control plane, not a second engine.
- **Graph grouping and semantic zoom.** Theme/operator/metric/niche grouping, group collapse, and LOD
  are useful foundations; the default viewport behavior is the problem.
- **Visible uncertainty.** “Unconfirmed,” caveats, trust mode, and detector state are exposed in text.
- **One persistent assistant across surfaces.** Bottom/side/full continuity is valuable when state,
  geometry, and mode are explicit.
- **Separation of stop/finalize/resume.** The product has the right conceptual distinctions even when
  control placement and confirmation need work.
- **Power-user depth.** Queue, hypotheses, trust, sensitivity, importance, cross-run, registry,
  artifacts, config, and authoring should remain available through progressive disclosure.

# PART II — DETAILED AUDIT

## 5. P0 findings — truth and control safety

### SAFE-01 — “Copy read-only share link” produces a writable control-plane URL

| Field | Value |
|---|---|
| Evidence | **Code-confirmed**, high confidence |
| Surface/journey | Collaboration → share/review |
| Impact | A recipient who can reach the deployment receives the normal run route and its mutations, contrary to the UI promise |
| Effort | L–XL; requires server authorization, not a label change |
| Dependency | Doc 17 authorization/reference-monitor work |

`panels.jsx:795-803` copies the normal `#/run/<id>` route. `App.jsx:22-48` resolves that route to the
ordinary `RunView`, including assistant, config, reset, node actions, and control transport. There is
no viewer credential or read-only route.

**Recommendation:** create an explicit viewer capability/token and route. Hide writes for clarity,
but enforce the boundary at the server; client-side hiding is not authorization.

**Acceptance:** a share-credential E2E may read DAG/report/artifacts and receives a server rejection
for every control/config/assistant mutation. Network recording proves zero successful write requests.

### SAFE-02 — time travel renders history while retaining live mutations

| Field | Value |
|---|---|
| Evidence | **Code-confirmed**, high confidence |
| Surface/journey | Timeline → historical replay → inspect/act |
| Impact | A user can believe an action applies to the viewed historical state while it mutates the current run |
| Effort | M |
| Dependency | Central write-capability state |

`RunView.jsx:206-207` switches the displayed projection to historical state, while live mutation
paths remain mounted through DAG/Inspector/Dock and panels (`RunView.jsx:281-285,290-317,320-339`;
`Report.jsx:86-93`).
The header's `replay` text is not a sufficient mode boundary. There is a second state-integrity gap:
when `viewSeq` changes, the old `hist` snapshot is not cleared before the new fetch
(`RunView.jsx:86-93`), so the slider can say sequence N while the DAG still shows the previous
sequence until the response arrives.

**Recommendation:** make historical mode read-only by construction. Add a persistent “Viewing seq N”
banner and one action, “Return live to act.” Do not let individual components decide independently.

**Acceptance:** replay mode renders no enabled mutation control and issues no mutation request.
Attempting a command either remains read-only or explicitly returns to live before asking for action
confirmation. Historical state is keyed as `{requestedSeq,resolvedSeq,status,data,error}`; changing
sequence immediately clears or marks the prior data stale, and the UI never labels a snapshot whose
`resolvedSeq` differs from `requestedSeq` as the selected sequence.

### CTRL-01 — assistant and Dock can report different outcomes for the same control

| Field | Value |
|---|---|
| Evidence | **Code-confirmed through `369d6a6`** and inherited from doc 16 P1-10, high confidence |
| Surface/journey | Assistant/Dock/TUI → stop/finalize/resume/abort |
| Impact | The UI can announce success although the authoritative engine transition did not occur |
| Effort | M–L |
| Dependency | One server-owned control command/state-transition contract |

`AssistantBar.jsx:32-40,299-302` implements direct `/resume`, `/finalize`, and `/abort` by appending a
control event and immediately flashing success. The generic control route returns after the append
(`looplab/serve/routers/control.py:79-130`). Dock's corresponding paths append the event **and** call
`resumeRun` so an exited engine is actually spawned (`Dock.jsx:558-559`). The same user intent can
therefore say “resumed” or “finalizing” without producing the same runtime effect. Commit `3170d85`
expanded `api.js:67-82` and added a TUI/JS needs-engine registry parity test, but direct
`AssistantBar.runDirect` does not use that batch-plan registry. This is partial remediation, not the
shared operation/postcondition contract required here.

**Recommendation:** route Web buttons, assistant commands, and TUI commands through one server-owned
operation that returns a command ID and observable lifecycle. The client may acknowledge receipt, but
must not present completion until the authoritative state transition is folded/observed.

**Acceptance:** one `ControlSpec`/server command service owns stop/finalize/resume/abort semantics.
Parity tests cover **Assistant, Dock, and TUI × live, paused, stalled/engine-exited, finished, and
finalizing**. Every cell produces the same accepted/rejected event and process postcondition; success
appears only after that postcondition is observed, while timeout/failure remains actionable.

### DATA-01 — Compare can label stale details as the newly selected node; “diff” is unordered

| Field | Value |
|---|---|
| Evidence | **Render-observed** layout + **code-confirmed** race/diff, high confidence |
| Surface/journey | Analysis → Compare nodes |
| Impact | A user can make a scientific conclusion from code/metrics that belong to another node |
| Effort | M |
| Dependency | Shared async resource-state pattern |

`panels.jsx:943-980` neither clears old detail on selection nor binds a response to the selector that
requested it. A late A response can overwrite C after A→B→C selection. `Inspector.jsx:815-819`
compares sets of lines, so order and duplicates disappear and changed lines can be displaced from
their original position.

The rendered Compare modal also has unlabeled selectors, small lower-case `diff`, raw code without
line numbers, and a whole-modal horizontal scrollbar. The command palette can remain above the modal
and obscure its content.

**Acceptance:** detail state is `{nodeId,status,data,error}`; stale responses are ignored; old data is
hidden immediately; selectors are labeled; an ordered line diff preserves duplicate and reordered
lines; race and golden-diff tests cover these cases.

## 6. Detailed priority findings — P1 unless marked P2

### STATE-01 — failures are routinely rendered as empty, missing, or forever loading

**Evidence:** code-confirmed, high confidence. **Effort:** L. **Dependency:** one shared async-state
contract and route-level error boundaries.

Examples:

- Runs converts fetch failure to `[]`, then can say “No runs” (`RunList.jsx:162-164`).
- Map swallows both fetch errors and leaves a blank canvas (`MapView.jsx:125-132`).
- Settings can stay at “Loading settings…” forever (`Settings.jsx:31-36,103`).
- Before any state arrives, an invalid run or initial SSE failure can remain “Connecting…”
  (`RunView.jsx:205`, `hooks.js:54-100`). After a first state, the hook can expose an offline condition;
  this finding is specifically the unclassified pre-state path.
- Inspector, report code, Dock log/trace, cross-run trajectories, and shared chat similarly collapse
  error into blank/loading/no-data states.
- There is no app/route Error Boundary.

**Recommendation:** standardize `idle | loading | success-empty | success | error | offline | stale`.
Every error needs a plain-language cause, retry, retained last-known data when safe, and a diagnostic
detail affordance. Empty must mean a successful empty response.

**Acceptance:** fixture tests inject timeout, 401/403, 404, 409, 500, invalid JSON, SSE disconnect,
and stale cache for every primary surface; none is labeled empty or successful.

### RESP-01 — the interface does not reflow; mobile can eliminate the DAG entirely

**Evidence:** render-observed + code-confirmed, high confidence. **Effort:** XL. **Dependency:** target
layout model and accessible drawers/splitters.

Measured behavior:

| Viewport | Observed geometry/result |
|---|---|
| 1280×720 | Side assistant at ~440px plus Projects at 280px leaves the list main around 560px; filters and header wrap |
| 1024×768 | DAG `598×323`; Inspector about 426px; timeline 260px; assistant bar 45px |
| 768×1024 | DAG width `342px`; Inspector still about `426px`; graph controls wrap and obscure nodes |
| 360×800 run | Document width `683px`; DAG width **`0px`**; Inspector fills the run workspace; horizontal scrollbar appears |
| 360×800 home | Projects remains 280px; run cards shrink to about `30px` content width and grow to 300–400px tall |

The stylesheet has no general responsive system: one hypothesis-board breakpoint and a
reduced-motion block. Sidebar, Inspector, assistant side panel, headers, tables, and splitters keep
desktop assumptions (`styles.css:106-130,460,623-625,1344,1488-1491,1535-1537`;
`RunView.jsx:49-66`).

**Recommendation:** define desktop, compact desktop, tablet, and mobile-read-only compositions.
Below the compact threshold, Inspector/timeline/assistant become mutually exclusive drawers; Projects
collapses; run cards become a single-column list; the graph never receives less than its minimum
functional width. Splitter alternatives must be keyboard operable.

**Acceptance:** 360×800, 768×1024, 1024×768, 1440×900, and real 200%/400% zoom have no page-level
horizontal scroll, zero-width primary content, obscured CTA, or overlapping layer. Mobile read-only
supports runs → run → node → report without a mutation control.

### VIS-01 — cross-run Map's default fit makes the entire view unreadable

**Evidence:** render-observed with 57 real runs, high confidence. **Effort:** M–L.

At 1280×720 the initial Map fits all runs and groups into one almost horizontal line near the center
(`MapView.jsx:22-25,125-140`);
cards become a few pixels wide and labels cannot be read. Most of the canvas is empty. The control
offers zoom and fit but no usable default framing, search, active filter summary, legend, minimap
overview of meaningful scale, or staged disclosure.

**Recommendation:** start at project/task clusters, not all leaf runs. Preserve a readable minimum
node size, collapse large groups, show a list/map selection handshake, and make “fit selected” the
primary action. Reconcile the existing graph-grouping proposal rather than replacing its layout work.

**Acceptance:** the 57/500/5,000-run fixture manifest contains at least two top-level groups. On first
paint the default level shows **10–40** run/cluster targets, each at least **24×24px**, with labels at
least **11px** and no label overlap; their bounding box uses at least **40%** of usable canvas width and
**30%** of height. Lower levels aggregate or hide labels instead of shrinking them. After entering a
name in search, a user can focus/open that run in at most **three interactions** without blind zooming;
0/1-run and error states are explicit.

### VIS-02 — the run workspace allocates space to panels instead of the user's current task

**Evidence:** render-observed + code-confirmed, high confidence. **Effort:** L.

On Search, an empty Inspector consumes roughly one third of the desktop. Selecting a node adds six
mouse-first tabs. Timeline fixes another 260px and the assistant another 45–60px. The graph-grouping
toolbar floats over nodes. At 1280×720 the graph is visually small; at compact widths it becomes
nonfunctional. Side chat does not coordinate with Inspector or Dock and can leave all three open.
The competing fixed/flex allocations are visible in `RunView.jsx:49-66,290-317`,
`AssistantBar.jsx:110`, and the responsive anchors cited in RESP-01.

**Recommendation:** adopt task-priority rules:

1. Graph is the default primary canvas.
2. Inspector opens contextually and returns space when closed or empty.
3. Timeline, assistant, and Inspector cannot all claim permanent desktop columns/rows below a
   threshold.
4. User-resized dimensions persist per surface, with “reset layout” and keyboard resize controls.

**Acceptance:** at 1024×768 the usable DAG remains at least **480×320px**; if that cannot be preserved,
opening side chat converts Inspector/timeline to mutually exclusive drawers or asks for an explicit
layout choice. No primary canvas silently falls below the minimum.

### VIS-03 — light themes fail contrast and semantic consistency

**Evidence:** Paper render-observed; all six themes code-confirmed. **Effort:** M.

The Paper report visibly loses the `/new` header token, the caveat message, several chart labels, and
muted labels. The problem is not the shared `--fg-mut` token by itself: code contains dozens of
specific hard-coded color literals outside the theme-token layer;
the exact count depends on whether SVG decoration, repeated literals, and fallback colors are included.
Examples include:

- report gold `#ffd98a` against Paper surfaces (approximately 1.06:1 in the affected state);
- chart label `#e6e9ef` against Paper/White Sci-Fi (approximately 1.01–1.09:1);
- fixed light text in `.chip.ok/.alarm/.warn` on light surfaces;
- white hover text on Old Green's bright primary (~1.30:1);
- dark fixed chart/grid/minimap colors across light themes.

Trust compounds the issue: “no leakage scan recorded” and “detector off” are rendered as green
positive chips. That is semantically wrong even when contrast passes.

**Recommendation:** make charts and semantic chips token-driven. Separate `pass`, `warn`, `unknown`,
`disabled`, and `not-run`; do not derive them from a binary green/red model.

**Acceptance:** automated contrast matrix for six themes: text ≥4.5:1, large text/UI graphics ≥3:1;
status meaning is available in icon/text, not color alone; not-run/disabled never uses pass styling.

### VIS-04 — undefined `--border` silently removes structure

**Evidence:** code-confirmed, high confidence. **Effort:** S. **Priority:** P2.

`--border` is not defined in the root tokens (`styles.css:2-28`) but is used without fallback for
hypothesis cards/columns, the Why strip, node context menu, and MultiTrajectory axes
(`styles.css:421,426,432,439`; `charts.jsx:322-323`). Those declarations are invalid.

**Acceptance:** define the intended token or replace it with `--line`/`--line-2`; a theme screenshot
test confirms every affected border/axis in all six themes; a CSS custom-property lint prevents
unknown tokens.

### A11Y-01 — interaction semantics are not a coherent accessibility model

**Evidence:** DOM render observations + code census, high confidence. **Effort:** XL.

The static census found dozens of `onClick` handlers on non-button elements (`div`, `span`, `tr`,
`th`, `g`, `circle`, `path`) and very sparse explicit ARIA. The exact count is intentionally omitted:
it changes with multiline JSX and whether a non-native element already has an explicit role. Examples:

- Inspector and Settings tabs are clickable `div`s without `role=tab`, `aria-selected`, or arrow-key
  navigation;
- run-card bodies, projects, timeline rows, chart marks, artifact rows, assistant sessions, and
  disclosures are mouse-first;
- settings labels are not programmatically bound to controls;
- custom checkboxes hide the native input with `display:none`;
- panels/modals are generic containers without `role=dialog`, `aria-modal`, focus trap/restore, or a
  common Escape contract;
- splitters are drag-only;
- toasts/live SSE changes are not announced;
- chart tooltips depend on hover and have no table alternative;
- several targets are 18px and `chipFontSize` can fall to 8px;
- reduced-motion does not cover all pulse/blink/cursor animations.

Representative anchors include Inspector tabs (`Inspector.jsx:97-100`), settings labels and custom
checkboxes (`SettingsForm.jsx:32-63,81-87`), generic modal shells (`panels.jsx:12-20`), and compact
interactive styling (`styles.css:417,771-780`).

**Acceptance:** keyboard-only smoke completes find run → open → inspect node → compare → report;
focus is visible and restored after dialogs; axe has no critical/serious issues; screen-reader testing
confirms names/order/live announcements. Pointer targets meet WCAG 2.2 2.5.8's **24×24 CSS px**
minimum or a documented exception; primary mobile touch actions use **44×44px** targets.

### SAFE-03 — destructive actions use incompatible and sometimes absent safeguards

**Evidence:** code-confirmed, high confidence. **Effort:** M–L.

The visual inconsistency is backed by a deeper authorization inconsistency:

- drag-to-merge, context-menu reset/kill, Finalize, permanent hypothesis deletion, and chat deletion
  can execute immediately; other paths use native `confirm/alert`;
- `remember` is mounted before the Plan-mode mutation boundary and can write the shared knowledge
  base from a nominally read-only session/subagent;
- Assistant “Always” is an unbounded `session → tool_kind` grant. It carries no mode, turn/cancel
  epoch, action, arguments/scope, or TTL. MCP calls are labelled `shell`, so one remembered MCP grant
  can authorize unrelated shell/MCP effects;
- Auto makes shell, delete, and unknown MCP effects inline even though these are high-risk. Cancel can
  race after approval because providers do not re-check an immutable authority epoch immediately
  before the effect;
- file edits prepare a diff, wait for approval, then write the precomputed result without verifying
  the file version again. Backup is best-effort, while undo accepts an absolute path rather than an
  originating change receipt and can overwrite a newer manual/agent edit;
- the server generally enforces only owner authentication. Browser `confirm()` is therefore not an
  authorization boundary: a direct HTTP owner client can bypass it. Shared Assistant reads also retain
  `applied.abs_path`/diff metadata because the share sanitizer removes only `raw`.

Representative anchors are `serve/assistant.py:335-352`, `serve/routers/assistant.py:51-98,173-181,
268-272`, `tools/perm_modes.py:32-43`, `tools/write_tools.py:40-72,188-231`,
`tools/mcp_tools.py:149-174`, `tools/knowledge_tools.py:137-190`, `Dag.jsx:372-449`,
`Dock.jsx:556,623-628`, `panels.jsx:871-875,914-915`, and `AssistantBar.jsx:284-289,654-657`.

**Recommendation:** one deny-by-default action registry with four risk classes: READ, REVERSIBLE,
CONSEQUENTIAL, HIGH. Plan denies every durable mutation. HIGH always asks, including Auto, and can
never be remembered. A remembered grant is exact `(session, mode, cancel epoch, action, scope digest)`,
expires after ten minutes without sliding, and is revoked by cancel/mode change/fork/delete. Providers
follow `prepare → approve → lock → revalidate version/scope → effect`; reversible writes require a
fail-closed backup and return a session-bound opaque change receipt. Direct consequential/high HTTP
mutations use a one-use 60-second server challenge bound to principal, method/path, normalized body,
version, and Idempotency-Key; the confirmed replay reuses the exact body/key and runs at most once.
One accessible confirmation host renders the server-authored object, scope, risk, consequence, expiry,
pending state, and recovery. Drag-to-merge becomes an explicit preview/commit action.

**Acceptance:** a complete tool/route mutation inventory fails closed on unknown actions. Contract
tests cover the 4×4 mode/risk matrix, exact-grant mismatch and expiry, cancel/mode/fork revocation,
allow-vs-cancel races, MCP/shell scope isolation, HIGH non-rememberability, Plan `remember` denial,
file/git/run-scope TOCTOU, backup failure, stale/forked undo, and secret-free cards/stores. HTTP tests
prove first request = 428/effect zero; wrong/reused/expired or different-body challenges fail; version
drift = 409/effect zero; concurrent/exact lost-response replay produces one effect/result; legacy
control/resume and TUI cannot bypass the gate. UI tests cover every current immediate/native-confirm
entry point, one POST under double click, focus trap/restore, live status, and History/Review denial.
Audit history records the action/risk/scope/source and actor when identity exists.

# PART III — TARGET EXPERIENCE AND DELIVERY

## 9. Target interaction principles

1. **Truth before reassurance.** Empty, pass, unknown, disabled, stale, and offline are different
   states with different visual semantics.
2. **One surface owns an action.** Every action has a visible canonical home; commands and shortcuts
   accelerate it.
3. **Context earns space.** Empty Inspector, idle timeline, and inactive assistant do not permanently
   tax the graph.
4. **History is a capability boundary.** Viewing the past is read-only unless the user returns live.
5. **Progressive disclosure, not feature deletion.** Common work is visible; specialist panels remain
   one predictable layer away.
6. **Task/objective compatibility is explicit.** Metrics are never ranked across incompatible tasks or
   directions without normalization and disclosure.
7. **Every visual explanation has a text/data equivalent.** Graphs, charts, color, and hover do not
   become the sole carriers of evidence.
8. **The URL describes the review state.** A shared diagnostic link opens the same node, tab, panel,
   and sequence within the viewer's permissions.

## 10. Recommended information architecture

### 10.1 Runs home

- Primary action: **New run**. Secondary action inside Projects: **New project**.
- Collapsible Projects rail; on mobile it becomes a drawer.
- Search + Filters button + active filter chips; advanced filters live in a popover/drawer.
- Saved views/favorites and a task-scoped comparison mode.
- List/Map preserves the same selection and filters.
- Run card: status + name + task, one-line outcome, best metric with objective direction, last update,
  one semantic open target, separate overflow actions.

### 10.2 Run workspace

Recommended hierarchy (names should be validated with users):

- primary: **Explore** (DAG + contextual Inspector), **Timeline**, **Report**;
- contextual Analyze drawer: Compare, sensitivity, importance, cross-run;
- contextual Rigor drawer: Trust, data quality, Pareto/diversity;
- contextual Work drawer: Queue, hypotheses, research, failures, artifacts;
- run actions: status-aware primary action plus one overflow menu;
- assistant: explicit mode, context attachments, and responsive bar/drawer/full behavior.

Overview should be either a true primary surface with budget/progress or a clearly styled summary
dialog. It should not look like a selected route while leaving Search underneath.

### 10.3 Visual hierarchy

- Use one 4/8px spacing scale, one radius scale, and declared elevation/layer tokens.
- Keep primary body text at a durable size/contrast; metadata may be quieter but not 8px.
- Reduce simultaneous accent colors. Reserve green/red/yellow for evaluated semantic states; use
  neutral accent for selection/navigation.
- Standardize icon + label rules. Symbols alone are acceptable only for universal, labeled controls.
- Give charts a shared grammar: title, objective direction, unit, baseline, uncertainty, legend,
  accessible table, loading/empty/error.

### 10.4 Semantic token taxonomy (concrete target for VIS-03/VIS-04)

"Make it token-driven" is under-specified on its own. The following is the proposed token layer that the
VIS-03/VIS-04 acceptance criteria should be measured against. Every token is defined in **all six themes**;
a CSS custom-property lint fails the build on any `var(--x)` whose `--x` is undefined — exactly the check
that would have caught the baseline VIS-04 defect. Part IV records the branch's concrete `--border` fix;
the complete taxonomy and lint remain target-state acceptance criteria.

- **Status (never a binary green/red):** `--status-pass`, `--status-warn`, `--status-unknown`,
  `--status-disabled`, `--status-notrun`, each with `-fg`/`-bg` pairs. `unknown`, `disabled`, and
  `notrun` are neutral/amber, **never** the `pass` hue (TRUST-01, VIS-03). Each status also carries a
  paired icon + text label so meaning survives grayscale and color-blindness.
- **Structure:** keep the now-defined `--border`, plus `--line`/`--line-2` for hairlines vs. dividers
  (VIS-04). No component reads an undefined token.
- **Charts:** `--series-1..N`, each bound to a **dash/marker pattern** so a series never relies on hue
  alone (GRAPH-01), plus `--axis`, `--grid`, `--baseline`, `--uncertainty`. No hard-coded hex in
  `charts.jsx` — VIS-03 cited dark fixed chart/grid/minimap colors bleeding across light themes.
- **Elevation & layer:** documented `--z-base < --z-menu < --z-dialog < --z-toast` and `--elev-1..3`.
  This is the machine-readable form of the UX-NAV-01 layer contract — the same tokens that keep the
  command palette from rendering above Compare.
- **Density & type:** a spacing step set on a 4/8px scale and a minimum-legible type token
  (`--fs-meta` ≥12px) so nothing compresses meaning to 8px (TYPE-01).

### 16.1 New directions (2026-07-13 review pass)

The 07-11 audit is deliberately conservative: it grades the *current* control plane against truthfulness,
layout, accessibility, and gap criteria. This addendum records seven **product directions** that the
conservative frame under-weighted. They are recommendations, not runtime claims; each is anchored to code
that already exists, so it reads as an extension of shipped foundations rather than a green-field wish.
Each maps onto the §11 delivery stages (see "New-directions placement" there).

The unifying observation: **LoopLab's UI is a cockpit for a semi-autonomous process, not a CRUD admin
panel.** Several of the sharpest opportunities follow from taking that seriously — the human is
supervising an engine that acts on its own, over hours, while they are often not looking.

#### ND-1 — Agency legibility: always disclose *who* acted and whether it is committed

**Evidence:** code-confirmed foundations; recommendation. **Connects:** CTRL-01, SAFE-02, UX-ASSIST-01.

At any instant a state change can originate from the autonomous engine loop, the assistant acting on a
human instruction, a direct human control, or the background concurrent-research task (the only non-main
writer, restricted to the `BACKGROUND_APPENDABLE` types). That origin is present in the event log but
flattened in the UI, and "proposed" is not visually distinct from "folded/committed." At the audit
baseline, CTRL-01 was the acute symptom: the Assistant could flash success before the engine transitioned.
Part IV records the branch's shared command/postcondition fix for that seam; the broader actor + lifecycle
model remains an open product direction.

**Acceptance:** every mutation and annotation surfaces its actor (`loop | assistant | human | research`)
and lifecycle (`proposed | accepted | folded | rejected/failed`); the timeline attributes each control
event to its origin; optimistic UI never shows a terminal state before the corresponding domain event is
folded (this subsumes CTRL-01's postcondition rather than duplicating it).

#### ND-2 — Provenance affordance: trace any number to its evidence in ≤1 interaction

**Evidence:** recommendation; builds on shipped WhyStrip + Report caveat deep-links. **Connects:**
TRUST-01, UX-REPORT-01, DATA-02, CONTENT-01.

Every headline value — a champion metric, a trust chip, a badge, a leaderboard row — should be one gesture
from "which event/node produced this, confirmed or not, which gates ran, as of which node." Today
staleness (UX-REPORT-01), trust state (TRUST-01), champion disclosure, and metric provenance each solve a
slice with a *different* gesture. LoopLab's core credibility claim is that any number is defensible from
the log; a single "trace this" affordance is the UI expression of "files/events are the truth."

**Acceptance:** a shared provenance popover on any evaluated value shows source `seq`, node,
confirmed/unconfirmed, trust-gate states, and `at_node`/staleness; it is keyboard-invokable; every
headline value in Report/Inspector/Compare resolves to its evidence in ≤1 hop.

#### ND-3 — An attention model for unattended runs (wire the dormant notifications)

**Evidence:** code-confirmed dead hook; recommendation. **Connects:** NOTIFY-01 (elevate from P2),
UX-HITL-01, crash-repair, stall detection.

The product's implicit promise is "launch and walk away," which makes *interruption* the central journey,
not a P2 footnote: approval-blocked, spec ratification, stalled, budget-exhausted, crash-repair looping,
champion changed, finished, failed. `hooks.js:104 useNotifications` already encodes
finish/approval/spec-approval/failure-spike — and is **never called** anywhere in `ui/`. The raw material
is shipped and dead.

**Acceptance:** an in-app attention inbox plus opt-in browser notification (pluggable external routing
deferred to UI-R5) fires on a defined event set, each item deep-linking to the exact surface/action,
dismissible/acknowledged and surviving reload; `useNotifications` is wired and extended with
stall/budget/champion-change triggers; every notification corresponds to a domain event so replay/resume
never double-fires it (engine invariant 3).

**Feature-branch resolution:** §18.16 removes the unsafe dead hook rather than wiring its guessed
state-transition logic. Desktop alerts now use durable event-derived signals; derived stall remains
in-app only. Champion-change, service worker/Web Push, external routing, cross-device sync, and a direct
visible run/spec approval control remain open, so this historical acceptance criterion is only partial.

#### ND-4 — Live-connection resilience and missed-window recovery

**Evidence:** code-confirmed; recommendation. **Connects:** STATE-01, SAFE-02, PERF-01.

`hooks.js` auto-reconnects with backoff and RunView shows an `offline` label, while §18.13 adds the
generation-fenced cursor transport needed for correct backfill. A *gap* is otherwise still invisible: no
"N events while you were disconnected," no per-surface stale marking, no backfill confirmation, and the
fate of a control issued during a drop is undefined. For a live event stream that researchers keep open
for hours across suspends and network blips, the reconnection *UX* — not merely the transport — is a
first-class journey.

**Acceptance:** on reconnect the UI backfills by cursor and shows "resynced (+N events)"; each live surface
marks itself **stale**, not empty, while disconnected and clears on resync; a control issued while offline
is explicitly queued-or-rejected, never silently dropped; reconnect never duplicates timeline rows (it
shares the cursor tail with PERF-01).

#### ND-5 — The assistant as universal explainer and diagnosis→action bridge

**Evidence:** recommendation; extends shipped WhyStrip and Report deep-links. **Connects:** CONTENT-01,
UX-ASSIST-01, TRUST-01.

LoopLab already narrates autonomous decisions (the U6 WhyStrip) and deep-links Report caveats to the
panel that explains them. Two extensions turn the persistent assistant from a *parallel command channel*
into connective tissue: (a) an "explain this" affordance on any term/chip/metric that returns a
context-scoped, glossary-aware definition — directly answering CONTENT-01's mixed vocabulary — and (b)
binding the top diagnostics to a follow-up control ("this node tripped the reward-hack gate → open
confirmation / spawn ablation"), routed through the one authoritative control path from ND-1.

**Acceptance:** "explain this" is available on trust chips, metrics, phases, and operators and returns
plain-language, source-linked copy; at least the weak-node, failed-node, reward-hack-flag, and stall
diagnostics offer a bound action executed through the ND-1/CTRL-01 control path.

#### ND-6 — Keyboard-first as a power mode, not only accessibility compliance

**Evidence:** recommendation. **Connects:** A11Y-01, §8 command palette, INSPECT-02.

A11Y-01 frames keyboard support as accessibility debt. Reframed for the actual audience —
keyboard-heavy ML researchers moving between queue, compare, trust, and deep links — a documented command
grammar, a global palette, and a consistent shortcut model become a *velocity* feature that delivers the
accessibility baseline as a by-product. Design it as first-class navigation (Linear/Superhuman-style),
not a retrofit of `role=` attributes onto mouse-first `div`s.

**Acceptance:** a published shortcut map covers navigate-runs, open node, cycle Inspector tabs, compare,
guarded approve/finalize, toggle assistant, and command palette; each shortcut is discoverable
in-product; the same primitives satisfy A11Y-01's axe/keyboard baseline.

#### ND-7 — Post-ship UX telemetry closes the acceptance loop

**Evidence:** recommendation. **Connects:** §12 acceptance metrics, STATE-01, §2.1 fixture manifest.

§12's metrics are all *pre-ship* gates; nothing measures whether the redesign works in the wild — journey
completion, where users stall, error-state frequency by surface, panel usage, notification engagement.
LoopLab's single-tenant / self-hosted posture makes privacy-respecting, local-first instrumentation
feasible, and the project's event-log discipline extends naturally to a UI interaction log.

**Acceptance:** an opt-in, local-first UX event log records journey funnels and error-state occurrences;
a small internal view surfaces top drop-offs and most-frequent error states; the §12 targets are
re-measured against real usage, not only fixtures.

## Part III baseline verdict

LoopLab's UI has enough breadth to be a serious research control plane. The next step is to make that
breadth **truthful, spatially stable, and operable without hidden knowledge**. Ship the P0 truth
boundaries first; establish common async/accessibility/layout primitives second; then reorganize the
navigation and add expert accelerators. Preserve the report, graph grouping, event provenance, and
assistant continuity, but stop allowing panels, colors, and labels to imply stronger guarantees than
the underlying state provides.

The §16.1 directions do **not** reorder that sequence — they sharpen its endgame. Read the whole
product through one lens: the UI is a cockpit for a semi-autonomous engine the user leaves running.
That lens turns four scattered findings into one program — *who acted* (ND-1), *where a number came
from* (ND-2), *when the run needs you* (ND-3), and *what happened while you were away* (ND-4) — and it
is exactly the program the P0 truth boundaries already begin. Build the truth first; then make that
truth legible, traceable, and able to reach the user who walked away.

- Unversioned sidecars — trace, operation trace, streaming metrics, logs-as-live-detail, refresh, and
  Assistant run tools — are hidden or disabled rather than presented as historical evidence.

**Authorization architecture**

- `LOOPLAB_UI_TOKEN` is no longer embedded in owner or review HTML. When it is configured, the owner
  shell presents an explicit unlock screen and keeps the accepted credential in that owner tab's
  `sessionStorage`; API requests continue to send it as `X-LoopLab-Token`. A deployment without this
  owner credential cannot create a review link and receives `409`, because an anonymous owner control
  plane cannot honestly issue a lower-privilege share capability.
- A link contains a high-entropy bearer only after the URL fragment on the tokenless `/review` shell.
  The fragment is not part of the HTTP path. Reviewer requests send it as `X-LoopLab-Review`, and
  every success and denial uses `Cache-Control: no-store`, `Referrer-Policy: no-referrer`, and
  `Vary: X-LoopLab-Review`. The client also requests review resources with `cache: 'no-store'`; a
  cached revoked/expired `401`/`410` therefore cannot poison a later bearer that reuses the same
  `/api/review/*` URL.
- The server accepts a reviewer principal only in the dedicated `/api/review/*` read namespace. The
  resolved capability supplies the run ID; reviewer input cannot select another run. Any owner route,
  any non-GET request, and any unsupported review resource returns a server-side rejection even if a
  future client accidentally renders a writable control; an unknown `/api/review/*` path is a JSON
  `404`, never a misleading owner-SPA `200`.
- Capabilities are persisted one atomic file per link under `.reviews/`. Only the SHA-256 digest is
  stored; the bearer is returned once. Each record binds one run, its exact generation, scopes,
  creation/expiry, and revocation. TTL is bounded from five minutes through 30 days (the UI offers one
  hour, one day, seven days, or 30 days), and expiry/revocation/generation is rechecked on every
  request. A reset or replacement makes the old capability return `410` rather than retargeting it.
  Capturing and rechecking that generation reads only the first durable event record. The short
  sequencer section is therefore **O(first durable event)** with respect to log length, not a full-log
  scan; state folds, metrics adapters, and response assembly remain outside it.

**Scope and UX**

- The default `summary` scope exposes the folded DAG/report state, masked run configuration, cost, and
  derived node metrics. `evidence` is an explicit opt-in that adds redacted node source/results and
  parent diff through an explicit field allow-list. Known credential patterns are recursively scrubbed
  from nested JSON keys and values, source filenames, and metric-series tags before disclosure.
  Redacted-name collisions retain every entry through deterministic `[redacted N]` suffixes rather
  than silently overwriting one another; adding a future Node field cannot silently make it
  reviewer-visible.
- Raw logs and captured process output including `stdout_tail`, model prompts, traces, live sidecars,
  artifacts, authoring, memory, collaboration writes/history, and owner settings are intentionally
  not shareable. A reviewer may read the recursively redacted *current* comment projection described
  in §18.17; no reviewer mutation or comment-history endpoint exists. This remains a narrower and
  safer contract than copying the entire owner workspace.
- Collaboration now creates and copies a time-bounded link, shows the newly minted URL once, lists
  active/stale/expired/revoked links without their bearers, lets the owner revoke an active or stale
  link, and exposes the scoped current comment feed/deep links. `stale` means the link's original
  generation is gone; it is retained for explicit cleanup and audit visibility, not treated as usable.
- Review renders the normal DAG/Report comprehension surfaces under a persistent **Read-only review**
  banner with scope and expiry. Assistant and Timeline are not mounted; node mutation menus and
  report refresh are absent; unsafe hub entries are disabled; the visible Settings control is
  disabled. Inspector explains the evidence boundary and hides live traces/actions.
- Review state polls through the capability namespace because native `EventSource` cannot carry the
  review header. A `401`/`410` clears the previously rendered run and replaces it with the terminal
  **Review access ended** state rather than leaving stale evidence on screen.

**Boundary and residual limits**

- This capability guarantees one-run, server-enforced read-only access inside `/api/review/*`; it is
  not an identity system. Existing lightweight owner projections remain governed by the deployment's
  broader auth policy, and anonymous/local mode remains unsuitable for sharing.
- Evidence redaction is defence in depth, not formal data-loss prevention. Owners must treat source
  disclosure as sensitive. Revocation prevents subsequent reads but cannot erase data a recipient
  already downloaded or copied.
- Shared-origin co-tenants, authenticated human/reviewer attribution, role-based policy, replies,
  mentions, and a cross-principal audit/RBAC contract remain open. Block 8 supplies deployment-level
  actor labels plus append-only comment history, not human identity. A private origin per principal is
  still required for hostile multi-user deployment; see the [deployment guide](guide/deployment.md).

### 18.8 CTRL-01: one observable run-command lifecycle

Web controls, Assistant tools, and the TUI no longer infer completion from an appended control event
or maintain separate “needs engine” registries. They submit the same server-owned command and report
the command record's state.

**Authoritative contract**

- `RunCommandService` owns one `ControlSpec` for every event in `CONTROL_EVENTS`. Each spec names its
  engine policy and observable postcondition: a foldable intent, paused with no driver, finished with
  no driver, or an exact engine acknowledgement.
- `POST /api/runs/{run_id}/commands` accepts `{type,data,expected_generation}` and requires an
  `Idempotency-Key`;
  `GET /api/runs/{run_id}/commands/{command_id}` exposes the durable record, and
  `POST /api/runs/{run_id}/commands/{command_id}/retry` re-arms that same record. Records move through
  `accepted`/`executing` to `succeeded`, `noop`, `failed`, `rejected`, or `timed_out`. Only
  `succeeded` and `noop` are presented as completion.
- The service persists one atomic record per command under the run's `.commands/` directory and
  stores only the SHA-256 digest of the idempotency key. Retrying the same key and payload returns the
  same command and cannot append or spawn twice; reusing the key for a different payload returns
  `409`. Web, Assistant, and TUI transient-submit retries reuse the original key.
- For engine-driving work, the engine emits `command_ack` only after it has folded the marked control
  event, including the exact command ID and event sequence. This proves that the intended engine
  observed that intent; it does **not** claim that downstream domain work such as a fork, reset,
  evaluation, or research stage has finished. Stop and finalize retain their stronger paused/dead and
  finished/dead postconditions.
- Command and engine acknowledgement monitors now share one process-local immutable observation from
  an eight-log LRU. The scanner parses only the recoverable suffix and indexes intents,
  acknowledgements, finish, and abort facts; the engine acknowledgement writer keeps a process-local
  cursor. Unchanged observations still perform a bounded content probe, and a state/finalize fold is
  O(total prefix) once per changed revision, so this removes repeated disk parsing without claiming
  end-to-end O(delta) command-volume scalability. See §18.18 for the exact boundary.
- Terminal failures carry a structured safe error with code, message, retryability, and remediation.
  Command responses are `Cache-Control: no-store` and vary by owner authorization so a cached
  `accepted` record cannot freeze polling or cross principals.
- Every control type has an explicit payload-field allowlist and strict normalization. Fractional
  node IDs, unknown/ignored fields, conflicting parent selectors, unsafe workspace paths, and a
  null finalize reason are rejected before an intent can become durable.
- `GET /state` and the state SSE expose a stable lowercase 64-hex generation token derived from the
  first durable event. A new command must echo the exact generation on which its intent was formed.
  Missing/malformed tokens fail with structured `400 invalid_run_generation`; an empty event log
  returns no token and rejects commands; a valid stale token fails with structured
  `409 run_generation_changed` before record creation, event append, execution claim, or spawn.
  Generation is intentionally outside the semantic payload digest: a valid same-key/same-payload
  lost-response replay still resolves its already-durable old record after Replay, but a fresh key
  can never redirect generation-A intent into B. Old-generation retry and nonterminal observation
  are inert, including unbound pre-upgrade records.

**Sequencing, retry, and destructive coordination**

- A cross-process per-run sequencer covers decision, normalized event append, and driver start. A
  different active command returns structured `409 command_in_progress` with its
  `existing_command_id`; an unresolved identical intent returns `409 retry_existing_command`. Web
  reattaches only to the latter, so success for one command can never be presented as success for a
  different requested action.
- The driver uses a pre-`Popen` spawn lease that covers the gap before `engine.lock` appears. Command,
  UI start, compatibility-resume, and reset starts all participate, preventing two processes from both
  deciding that no engine exists and spawning one. An observation timeout alone never releases a cold
  child: the lease enters fail-closed quarantine until `engine.lock` appears or the recorded PID is
  definitively dead; retry, another command, and destructive mutation remain blocked meanwhile. If a
  crash left ownership unknowable, an operator can use `POST /api/start/{run_id}/resolve-claim` with
  the exact verification phrase after the recovery delay. A claim whose PID and creation identity
  still match a live process cannot be forced away.
- Worker execution claims also carry their owner PID and process-creation identity. Missing heartbeat
  age alone never permits takeover: a stale claim is reclaimed only after definitive owner death or
  PID reuse, preventing a suspended worker from later racing a replacement worker's terminal write.
  Positive child/worker death is now acted on immediately rather than waiting through the nominal
  lease/command timeout; a failed `Thread.start()` releases its claim. A new execution owner is fully
  written before one exclusive hard-link publishes it, closing the normal create-then-write crash
  window; the compatibility fallback remains explicitly recoverable. Windows obtains a native
  creation-FILETIME identity without requiring optional `psutil`. Creation identities are tagged by
  their observation source (`psutil`, Windows FILETIME, or `/proc` start time): unequal tokens prove
  PID reuse only within the same source scheme. Cross-scheme and tagged/legacy comparisons stay
  inconclusive and block automatic takeover instead of evicting a possibly live owner; the delayed,
  exact-phrase operator recovery remains available. Pre-hardening/malformed/unknown
  execution or activity claims have a delayed, exact-phrase operator recovery endpoint that still
  refuses a provably live owner. Command records are also atomic, and stale empty pre-hardening
  reservations are healed.
- Failed or timed-out commands with a durable intent are recovered through the same-ID retry endpoint;
  the marked event is not appended again. A later GET also reconciles a terminal record to
  `succeeded` if its exact postcondition arrived after the observer's deadline.
- A terminal attempt writes `finalize_step:begun` with its exact terminal payload before the scoped
  `run_finished`. It writes `finalize_step:complete` only after the read-model build attempt (success
  or an explicit best-effort skip) and successful trace/tree projections. A hard kill on either side
  of `run_finished` stays `finalizing`; recovery republishes the begun payload and completes the same
  scope without reopening setup/search. The currently effective terminal event, not merely any
  historical success in that scope, controls recovery, so a later outer `reason=error` cannot become
  the permanent result after a projection failure. A configured natural-finish report is staged as
  `finalize_step(begun, finish_report_planned=true) -> finalize_step(report_begun) ->` scoped
  `report_generated -> finalize_step(report, outcome=...)` inside that same scope. A
  crash before `report_begun` is safely retried once; a durable report is reused; and an ambiguous
  begun attempt with no durable report is explicitly marked incomplete and is not billed again.
  Event-backed
  budget/diversity/cost gates are idempotent, while reflection records a begun marker and deliberately
  recovers at-most-once. This is not a blanket exactly-once claim for arbitrary external file writes.
- Reset and delete acquire the same sequencer, reject active command/execution/finalization/activity
  claims, and re-check canonical path/liveness after approval. Run-scoped boss/chat/report LLM work
  holds a generation activity lease; report and numeric usage append before that lease releases. If
  the usage append remains unavailable, the server retains the exact same-ID delta and keeps the
  activity lease. The next metered-route entry flushes that delta before constructing any provider
  client; a reset/delete guard performs the same non-paid flush before taking the destructive
  sequencer. Persistent failure stays blocked, so a replacement generation cannot inherit the usage.
  Engine and server clients also first attempt to persist the exact numeric delta in a run-local
  atomic `.llm-usage-outbox` before the event append; either successful write is the first durable
  boundary. The next reconciliation/metered/destructive boundary in a fresh process drains a
  persisted record with the same ID; reset/delete perform a second store-only drain under the destructive sequence, and reset
  archives the old outbox with the old event generation. Malformed, conflicting, symlink, or
  unacknowledgeable evidence blocks the destructive action rather than drifting into the replacement.
  Terminal `.commands` records survive an in-place reset, so an already-accepted same-key replay cannot
  apply to the new generation. SAFE-03's confirmation/grant/undo gaps remain separate.

**State and surface parity**

- Pending or projection-incomplete finalization is a canonical `finalizing` phase. Repeated finalize attaches to the in-flight
  operation; finalize on an already successfully finished run is a no-op; stop/resume and other
  engine-driving work are rejected while wrap-up is pending. `run_finished(reason=error)` while the
  stop intent remains set is recovery state, not successful finalization. The run list, header, and
  Dock share one lifecycle helper, keep finalizing/terminal-write-out visible, and suppress conflicting
  Resume/Replay controls until the driver is actually gone.
- Every Web `CONTROL` wrapper, including reset, approval, fork, merge/inject, strategy, and budget
  operations, uses the command endpoint. Report regeneration remains a deliberately separate job
  endpoint because it is not a `CONTROL_EVENTS` transition.
- Dock and Assistant each persist a strictly allowlisted, sanitized transport envelope and acquire one
  exact per-run lock in tab-scoped `sessionStorage` **before** POST. The lock binds source, action,
  displayed run generation, idempotency key, and any learned command ID; a known ID cannot be
  downgraded or replaced. `useRunState` publishes a generation to the command registry only after
  React commits that state, so a click on visible A cannot be silently rebound by an SSE frame for B
  that has not rendered yet. If storage
  is unavailable, blocked, or fails its write, the envelope is rolled back and the command is not sent.
- Non-critical `localStorage` preferences now use a shared exception-safe boundary in RunView,
  AssistantBar, Dock, and DAG. A browser `SecurityError` therefore falls back to defaults instead of
  blanking the React tree before the storage-unavailable command recovery can render.
- Stored state is treated as untrusted protocol input. Loaders require the expected action/event pair,
  exact record shape, and valid command-ID format; unsafe text is bounded and redacted. Malformed,
  tampered, or mismatched state is quarantined as a non-resubmittable protocol failure rather than
  replayed. A known durable ID may still be checked or dismissed, but an unknown intent is never
  reconstructed from suspicious storage.
- The shared lock is exact and tab-scoped: Dock exposes an Assistant command pending for that run, and
  Assistant refuses a competing Dock/Assistant intent until the same identity resolves. Commands for
  other runs remain independent. The in-page invalidation signal carries only the run ID, not the
  idempotency key, command ID, or payload.
- Reload, a lost POST response, 408/425/429, network/5xx status loss, or invalid JSON/protocol state
  becomes explicit **status unavailable** recovery with **Check same command**. Both surfaces preserve
  one logical identity: with a known durable ID they GET/retry that ID; if the POST response was lost
  before the ID arrived, they resubmit the same idempotency key plus deterministic payload and accept
  the server's same record. Assistant also restores a durable failure after reload and offers **Retry
  same command** only where the record permits it. `accepted` remains pending rather than fabricated
  into `executing`.
- Assistant pending/failure moves focus to its recovery status, uses polite/assertive live regions for
  status/error feedback, and returns focus to the composer after same-run success. Recovery actions
  wrap and retain touch-sized targets on narrow screens. A direct result remains attributed to its
  originating run and is never flashed into a run selected while the request was in flight.
- Assistant run tools receive the same server service through dependency injection. The TUI uses the
  same HTTP records and persists the predicted ID, exact idempotency key, displayed generation, and
  immutable nested intent **before** POST. Ambiguous network/408/425/429/5xx results stay pending. If
  an early GET returns 404,
  the TUI replays that exact key/payload with a bounded observation instead of declaring the command
  absent; a delayed original and the recovery POST therefore converge on one event. Corrupt recovery
  envelopes fail closed, and an ordered plan stops at its first pending step.
- Assistant command-backed run tools use an atomic per-session/turn mutation journal, not merely a
  deterministic key convention. Each normalized run-command intent and its pre-approval generation
  are staged before its side effect.
  Recovery of a dangling user turn may replay only the same ordered entries with their journaled
  keys; a changed/new mutation is blocked, and a direct storage mutation is treated as outcome-
  uncertain rather than repeated. Recovery pins the exact persisted raw instruction and permission
  mode and rejects a mismatch with `409`. Its reconstructed toolset is fail-closed: read/Todo tools
  plus the journal-backed run-control provider only; file/shell/git, knowledge, MCP, proposal, and
  subagent capabilities are absent. A different new user message receives `409` until the dangling
  turn is recovered or its session is explicitly abandoned. Stop marks the active turn cancelling
  but retains its single-turn slot until the worker exits, so a second turn cannot overtake an
  already-issued mutation or lose the first reply.
- Web reload/retry uses that recovery contract directly: it rechecks that the same dangling `turn_id`
  is still last, submits its persisted raw/display/mode once, and polls the existing transcript. A
  late original reply wins without a duplicate bubble/POST; corrupt or mismatched identity renders a
  blocked alert instead of falling through to a clean-content/current-mode retry.
- Both TUI and Assistant say requested/pending when bounded observation ends; neither advances a plan
  or claims success before `succeeded`/`noop`.
- Legacy `/control` and `/resume` remain for older clients. `/control` cannot overtake an active,
  retryable, or incomplete-finalize intent; the mutation-free stop-aware `/resume` may attach a
  recovery driver without clearing the terminal scope. New Web/Assistant/TUI controls do not depend
  on the old append-then-spawn chain.
- Standalone legacy CLI `stop`, `finalize`, `resume`, and `approve` commands remain outside the server
  sequencer. They must not be run concurrently with an active server-owned command; bringing them
  behind the same lifecycle is an explicit compatibility boundary, not implied CTRL-01 coverage.

This closes CTRL-01 at the shared command/postcondition and cross-generation admission seam. A fresh
request formed for generation A and first delivered after an in-place reset to B is rejected with
zero effect; an already-accepted same-key A record remains observable without being re-applied.
The automated suites cover the service
and all three adapters, and targeted production-Browser scenarios cover the highest-risk identity,
conflict, recovery, and lifecycle paths. A complete matrix of every surface and command across live,
paused, stalled, finished, finalizing, error-finalize, and terminal-write-out runs has not yet been
executed; this remains a validation boundary, not a claim of exhaustive end-to-end coverage.

### 18.9 COST-01: observed paid OpenAI-compatible usage is no longer rendered as free

A real MiniMax M3 run exposed a cross-surface truth defect: OpenRouter returned provider-reported
cost and token usage,
but `OpenAICompatibleClient` always added `$0.0`. Dock, Report, Inspector, exports, and review links all
consume that durable roll-up, so the UI truthfully showed the recorded value but the recording itself
was false.

**Provider and stream normalization**

- The client accepts only finite non-negative numeric `usage.cost`; strings, booleans, negatives,
  NaN/infinity, and overflow degrade to zero without partially mutating counters. Token fields are
  bounded integers. A contradictory `total_tokens < prompt_tokens + completion_tokens` uses the
  saturated component sum rather than preserving an impossible total.
- Every parsed provider response is accounted before semantic `choices` validation. Thus an HTTP 200
  envelope with known usage but no choices remains a paid call even though the caller receives an
  error. A successful usage-less response still increments calls once; a cache hit increments
  nothing. Local/OpenAI-compatible servers that omit price retain zero-dollar, measured-call behavior.
- A completed stream is one call. Usage received before a yielded chunk is retained if the consumer
  closes the generator. If cancellation happens after content but before the provider's final usage
  chunk, the ledger records one known call and honestly leaves tokens/cost unknown instead of
  inventing precision. A blocking fallback is recorded as its own observed call; if the abandoned
  outer attempt was accepted but never returned usage, that hidden invoice charge remains ambiguous
  as described below.
- Accountant mutation is atomic and saturating. A post-provider ledger-sink failure is telemetry
  failure, never a reason to repeat a paid request or subtract an already-observed call.

**Durable observed run ledger**

- Engine Researcher/Developer/Strategist/deep-research/report/reflection clients and attributable
  run-scoped boss/chat/per-run-report clients bind to the run ledger. Each observed delta appends
  numeric-only
  `llm_usage` (`cost`, call/token counters, and an opaque `usage_id`; never prompt, output, URL, model
  credential, or API key).
- Binding and accountant transfer use one lock-protected ownership boundary. A delayed sink cannot
  race aggregate reconciliation into a duplicate, and reusing an accountant transfers future calls
  to the new owner without letting the old Engine infer them later.
- Ambiguous append recovery retries the same `usage_id`; replay is first-write-wins, so a write that
  committed and then raised is counted once. The latest old-style `llm_cost` before the ledger is a
  compatibility base; later summaries are presentation snapshots, not accounting input.
- Before an event append, the ledger first attempts to atomically rename each exact sanitized delta
  into `.llm-usage-outbox/<usage_id>.json`; either a successful rename or event append is the first
  durable boundary. On the next reconciliation, metered-route, or destructive boundary, a normal
  finalizer exit/server restart/fresh Engine can therefore recover a persisted delta without the
  original accountant object. Exact stale records are
  acknowledged after their matching first-write event; malformed, conflicting, non-regular,
  symlinked, or undeletable evidence fails closed and prevents a summary/finalize marker.
- A short-lived server client whose first append and final reconcile both fail is retained with its
  run-generation activity lease. The next metered-route entry first retries only the same delta/ID;
  a reset/delete guard makes the same non-paid retry before destructive sequencing. Another provider
  call or destructive mutation is refused while that known usage remains nondurable. Reset/delete
  also scan the disk outbox after process-local state is gone, before crossing the run-generation
  boundary.
- Finalization does not mark its cost step complete while a known in-memory or outbox delta is still pending.
  Its summary is emitted after reflection because reflection may itself call the model.

**Paid live evidence and boundaries**

- The accepted harness explicitly pinned this worktree with `PYTHONPATH`; a preliminary invocation
  that imported the installed older package was rejected as evidence. Phase 1 reached **10 calls /
  62,286 tokens / `$0.00800382`**. A fresh process resumed the same run to **20 calls / 143,164 tokens /
  `$0.05811504`**. A later no-new-call re-entry stayed at **20 / 143,164 / `$0.05811504`**: the durable
  total survived restart and already-built work was not billed again. All 20 generation spans carried
  positive provider-reported cost; Report displayed `143 164 tokens · $0.05812`. The snapshot masked
  the key and an exact-value recursive scan found it in zero artifacts.
- That bounded live run proves the core engine restart/no-rebill path. Strategist, deep research,
  report generation, reflection, and cross-run memory were deliberately disabled there; their ledger
  routing is automated-test evidence, not mislabeled live coverage.
- Five honest boundary classes remain. An OS/server-process kill after provider success and in-memory
  counter commit but before either the atomic outbox rename or event append completes can lose the
  exact charge. A timeout,
  reset, decode/empty-body failure, or other ambiguous transport retry can hide an accepted/billed
  attempt A and issue attempt B without provider-side idempotency; only usage actually returned to
  the client can enter this ledger, so it is not invoice reconciliation. Cancellation or any missing
  terminal usage/result likewise cannot recover tokens or price the provider never delivered.
  Non-run-scoped spend is intentionally excluded: global Assistant, Genesis/new-run planning,
  `/api/research`, cross-run scope reports, `/api/llm/health`, and analogous CLI/global calls have no
  unambiguous run attribution. Finally, a custom `CostAccountant(limit=...)` is not seeded from the
  already-durable total on resume and is not a shared multi-role budget guard; the current product
  exposes no configured run-dollar limit, so this is an enforcement boundary rather than a false UI
  roll-up claim. The durable outbox removes restart recovery's dependence on the process-local
  retained-client registry; LoopLab's overall command/activity service is still validated as a
  single-UI-server deployment, not as a general multi-worker server architecture.

### 18.10 SAFE-03 phase 1: risk is now a server-owned permission contract

The earlier Assistant permission UX asked about a provider-supplied tool kind and could remember that
entire kind for the session. That made the button visually simple but semantically broad: a harmless
write could authorize a later write to another target, the grant had no visible expiry, and `auto`
could bypass destructive or unclassified actions. Phase 1 replaces that behavior at the Assistant
tool-provider and permission-registry boundary.

**One deny-by-default risk matrix**

| Classified risk | Plan | Default | Accept edits | Auto |
|---|---:|---:|---:|---:|
| `READ` | inline | inline | inline | inline |
| `REVERSIBLE` | deny | ask | inline | inline |
| `CONSEQUENTIAL` | deny | ask | ask | inline |
| `HIGH` | deny | ask | ask | ask |
| `UNKNOWN` | deny | ask | ask | ask |

- `classify_action()` is the shared server-side registry. An absent, malformed, or unregistered
  `(tool_kind, tool)` identity becomes `UNKNOWN`; provider labels cannot promote it to a safer class.
  Arbitrary shell commands and test execution are `HIGH`, because executing repository code is not
  inherently read-only. Delete/revert, checkout, process kill, run/node deletion, and trust-gate
  weakening also remain explicit every-time approvals even in Auto.
- Plan omits the shared-knowledge `remember` mutator entirely. In mutating modes, `remember` is
  `CONSEQUENTIAL` and uses the same provider gate as file, shell, git, run-control, and external MCP
  tools. MCP remains `UNKNOWN` until a concrete external capability has its own classification.
- A remembered grant is bounded to the exact session, normalized mode, current turn/cancel epoch,
  action identity, and canonical scope digest, with a fixed maximum TTL of ten minutes. A changed
  target, command arguments, content/diff digest, run budget, directive, node subtree, mode, turn, or
  expiry misses the grant. `HIGH` and `UNKNOWN` cannot be stored or consumed as remembered grants,
  including through a forged `allow_always` POST.
- Cancel, turn release, and session deletion invalidate the relevant grants. Permission resolution is
  a pending-to-resolved compare-and-set: a stale click cannot overwrite a cancel/previous decision.
  An expired card is denied and unblocked instead of remaining executable.
- Only explicit registered `(kind, tool)` pairs can be `READ`; a new tool cannot self-label its kind
  as read and bypass the registry. Providers accept only the exact `allow_once`/`allow_always` wire
  values, not an `allow*` prefix. Cancel publishes its flag and denies the currently owned turn's
  requests in one registry-lock transaction, preventing a stale cancel from invalidating a successor
  turn.
- Write/edit/patch scope now includes the full desired payload and current preimage digests. In the
  Assistant path they are `REVERSIBLE` only when a recovery-snapshot provider exists, and the effect
  aborts if that snapshot cannot be written. A manual file change during the approval wait still

### 18.13 PERF-01 implementation checkpoint: Timeline is paged, fenced, and virtualized

The owner Timeline no longer uses `liveSeq` to download and render the complete JSONL file. The new
`GET /api/runs/{run_id}/log-page` contract has four explicit directions: `tail` for initial/latest
sync, opaque-cursor `older` and `newer` reads, and `direction=around&anchor_seq=N` for a direct
scrubber jump.
`liveSeq` is only a wake-up hint; the server-issued cursor remains the pagination authority even at
the current live edge. Responses carry the run generation, both outer cursors, both `has_more`
directions, total/range metadata, and torn-tail state.

The generation is the same canonical first-event token used by run state/commands. Each opaque v2
cursor binds a row boundary to that generation **and** to a per-index revision. Append-only growth
preserves the revision. Atomic replacement (including a rewrite that deliberately preserves the
first event), shrink, same-size in-place rewrite detected from file metadata, LRU re-indexing/server
restart, a different first-event generation, an out-of-range boundary, or replacement during response
materialization invalidates it and the next pager read fails with `409 run_generation_changed`.
The browser never splices a delta from that response into the displayed rows. Its shared controller
discards old rows/cursors and reloads an exact-generation tail; while the SSE display still names the
old generation, it holds an empty fenced error state instead of alternating old and newly discovered
runs. Aborted/out-of-order requests are also fenced by request identity and run identity.

One discovery boundary remains explicit: a manual same-first-event rewrite that also preserves
count, last sequence, liveness, and folded generation has no changed state/SSE wake-up field. It is
fenced as soon as the next pager request occurs (or a later append/interaction wakes the controller),
but is not promised as an immediate passive refresh. Supporting that repair workflow immediately
would require a state/SSE file-revision signal or periodic pager polling.

The server builds a byte-boundary index once and extends it only from the last verified prefix as the
file grows. `around` bisects a cached sequence index rather than reparsing the 50k-row source on each
scrubber move. The per-process index cache is an eight-run LRU, so inspecting many historical runs
does not retain every index indefinitely. A final incomplete, malformed, non-event, duplicate, or
decreasing row ends the recoverable prefix and is never returned as a valid event; an in-place
completion can be picked up on the next read.

**Resource bounds are part of the transport contract:**

- pages default to 200 events and reject more than 500;
- event JSON inside a page defaults to 256 KiB and is capped at 512 KiB (minimum request 1 KiB); the
  reported `bytes` covers the compact event objects, not the surrounding HTTP/response envelope;
- one event larger than the page byte budget advances the cursor through a bounded metadata row with
  its sequence/type/timestamp, source byte count, and `_log_page.truncated`, rather than transferring
  the large body or creating an unpageable gap;
- an individual source line has an independent 8 MiB parser-memory ceiling. Crossing it stops the
  verified prefix and sets `source_tail_limited`; the UI says that the source tail was omitted;
- the browser retains at most 5,000 loaded rows under the normal server contract. Eviction removes
  complete server-page segments, keeping each opaque cursor paired with the exact retained boundary
  instead of slicing through a page and later skipping or repeating events.

RunView owns one `useTimeline` controller and passes the same state/actions to Dock and Raw event
explorer. Opening the explorer therefore creates no second full-log fetch, cursor stream, or reset
race. `tail`/`newer` catch-up is serialized; older paging and `around` explicitly park the reader.
The scrubber previews locally while dragging and requests the chosen historical window only on
commit. Directional errors leave the already loaded window visible with a matching retry, while a
successful jump back to live refreshes the tail before clearing unread evidence.

Both surfaces use the same variable-height virtual list. It measures only visible/overscan rows,
restores a stable event-and-pixel anchor after prepending older pages or remeasurement, pins the
bottom only for a user who is actually following, and scopes measurements to run generation.
Expanded Dock-row state is owned above the virtual row, so scrolling it out of the mounted range does
not silently collapse the user's detail when it returns. `windowAtTail` and `followingTail` are
separate: scrolling away parks a frontier, and the state/SSE payload's generation-scoped
`event_count` updates an exact **N new** indicator even when repaired logs have sequence gaps. An old
server without that additive field falls back to an unknown-activity wake-up. So does count growth
without sequence progress, or a count/pager mismatch at a malformed, duplicate, or decreasing tail:
`event_count` counts the full recoverable folded projection, while the Timeline pager deliberately
accepts only its stricter canonical prefix. Only an explicit/actual return to the live bottom resumes
following, and new events do not steal scroll from a reader of history.

History is fenced by the same generation identity. A requested and resolved historical snapshot
stores both sequence and generation, refetches when the live generation changes, rejects a delayed
generation-A payload or error after generation B is displayed, and is not rendered unless both
identities still match. A reset in another tab therefore cannot place an A-era DAG/state snapshot
beside a B-era live Timeline.

Filtering is deliberately honest and bounded. Dock and Explorer search only the retained loaded
window and say so when `total_events` is larger; paging is the way to inspect more source history.
Dock includes at most 500 serialized data characters in its projection so an unknown event's visible
fallback text remains searchable without multiplying retained payload memory. The raw explorer
renders at most a 500-character data preview and retains at most the first 4,000
serialized characters per loaded event in its lowercase search projection. Oversized source rows
are searchable/rendered only through their omission metadata, never by reconstructing the omitted
payload in the browser.

`/log-page` remains a raw owner surface. When owner auth is configured it requires the owner token;
a read-only review bearer is denied even when that share includes redacted evidence, and review mode
keeps the owner Timeline controller disabled and sends no raw-log request. Success, validation error,
generation conflict, auth failure, and review denial responses are all `no-store` with credential-
aware `Vary`. The route also rejects a symlinked `events.jsonl` or a resolved path outside the run
directory.

### 18.14 INSPECT-02: generation-fenced diagnostic route state

Block 5 makes the URL fragment the canonical store for shareable run context. An owner view can now
encode `gen`, Search/Report `view`, `node`, Inspector `tab`, overlay `panel`, direction `focus`,
historical `seq`, Timeline text `q`, and Timeline `kinds`, for example
`#/run/demo?gen=<64-lowercase-hex>&view=report&node=14&tab=code&panel=trust&seq=29&q=timeout&kinds=eval,trust`.
Defaults are omitted. Values are bounded and allow-listed; duplicate, unknown, malformed, control-
character, dependent, or generation-less diagnostic state is discarded and canonicalized with a
visible notice rather than guessed.

RunView reads this state directly. Discrete navigation uses history entries, Timeline typing uses
replacement, and both `popstate` and `hashchange` restore the exact view. A routed node reveals
Inspector, a routed filter reveals Timeline controls, and a deep link is not overwritten by the
finished-run auto-Report landing. Historical sequence remains snapshot-only and is valid only with
the exact run generation. Even a default **Copy view** URL keeps its generation-only fence through
canonical hydration rather than briefly becoming an unfenced live route. A semantic no-op such as
selecting the already-active default view returns the existing route state and preserves that explicit
generation-only fence.

Historical Inspector and Report detail requests send both `seq` and `expected_generation`. The server
requires that generation, checks it under short sequencer fences before and after folding the prefix,
and returns structured `409 run_generation_changed` if reset/replacement wins. The potentially
expensive fold runs without the exclusive command lock, so an old DAG cannot be paired with new node
code and a reader cannot starve owner commands.

The generation is also the mutation fence. Until current generation is known, and whenever the link
generation differs, Timeline/config reads are suppressed and the workspace shows a dedicated
verification/stale-link state. Assistant, Dock transport, and the shared API mutation guard remain
read-only behind the same `stale-link` access mode. LoopLab never reinterprets old node ids or sequence
numbers against a replacement log; **Open current generation** explicitly clears the stale context.
**Copy view** copies the canonical exact context, with a selectable fallback when clipboard access is
denied; an owner link still requires owner access.

Review URLs keep both the bearer and safe diagnostic context after `#`, for example
`/review#/rv_…?gen=…&node=4&tab=code`; neither reaches the HTTP request target. Minted links sanitize
the current view to the capability scope. Summary links expose only summary-safe tabs and panels;
evidence links may add Code and Compare. Both remove historical sequence and raw Timeline text/kind
filters. The review bearer parser reads only the fragment route portion, so suffix state cannot extend
the credential or fall back to owner auth.

Every new review capability now stores the exact generation captured while the run sequencer is held.
Every `/api/review/*` projection uses two short sequencer fences to validate paths and generation
immediately before and after assembling the response; state folds and metrics reads run outside the
exclusive lock. Each fence reads only the first durable event, so its critical section is
**O(first durable event)** rather than O(total log). A reset/replacement may therefore finish during a
slow review read, but the second fence discards the raced projection with **410 Gone** before it is
returned. Missing or malformed
legacy bindings and reset, replacement, or deletion also return 410; an old link is never silently
retargeted. The owner list labels it `stale`, keeps it revocable, and can mint a fresh link for the new
generation. Evidence detail is built from an explicit allow-list and excludes `stdout_tail`/captured
process output. Its recursive scrub covers credential-shaped nested keys, source filenames, and
metric tags without losing colliding redacted entries; unsupported `/api/review/*` paths return 404.

Block 6 turns the earlier one-off accessibility fixes into shared UI contracts. It materially closes
the implementation side of A11Y-01 for the existing Web surfaces, but it does **not** claim full WCAG
conformance or replace testing with assistive technology.

**Tables and charts.** `accessibility.jsx` now owns two reusable primitives:

- `DataTable` gives every migrated table a programmatic caption, a separately named and keyboard-
  focusable scroll region, column and row-header semantics, numeric alignment, explicit empty state,
  narrow-screen card labels, and optional CSV export. All remaining user-visible legacy tables in
  Dock, Inspector, Report, and the panel catalogue are inside this contract. Sort headers and row
  actions remain native buttons rather than clickable rows or headers.
- `ChartFrame` gives every analytical chart a persistent title and description, one named focusable
  visual viewport, a native **View data** disclosure, an exact-value table, CSV export, and an explicit
  no-data path. SVG points are no longer individual tab stops; keyboard users act through the native
  table/legend controls instead of traversing hundreds of marks.
- The visible encoding is redundant rather than color-only. Trajectory combines up to twelve point
  shapes with solid/outline/dot variants and constraint rings; Scatter combines shape and color;
  MultiTrajectory uses dash and color; failed Gantt spans use a dashed outline. Labels, grids,
  underlays, and hover strokes follow the active theme. Transparent SVG hit geometry is at least
  24 CSS px where a mark is actionable, while the visual mark can remain compact.
- CSV generation quotes embedded commas, quotes, and newlines and prefixes untrusted text beginning
  with a spreadsheet formula/control marker. Real numeric values stay numeric. The accessibility
  export therefore does not introduce a formula-injection path.

**Keyboard, routes, and layers.** DAG actions, the run-module hubs, Theme, Energy, and run menus now
share a roving-focus contract with Arrow keys, Home/End, Escape, disabled-item skipping, and focus
return. Merge destination selection has a native keyboard form; the earlier pointer-only final step
no longer applies. Primary run targets in List and Map are real links, and the SPA helper preserves
Ctrl/Cmd/Shift/Alt/middle-click browser behavior. Route changes update the document title and move
focus to a named `main` landmark without stealing focus on ordinary in-page updates.

`useDialogFocus` now maintains a topmost-dialog stack, traps forward and reverse Tab, contains stray
focus, closes only the active layer on an unconsumed Escape, and restores the invoking control.
Nested project creation, merge/reset flows, compact Inspector, project drawer, Assistant, and delete/
rename fallbacks use that ownership model. Compact Assistant is a true modal side drawer with an
inert/pointer-blocking backdrop and focuses the composer rather than an earlier editable launch card;
its desktop side mode remains nonmodal. Temporary controls keep a connected focus destination after
the underlying row, project, run, or supertask is renamed or removed.

**Responsive, target, and motion behavior.** At compact widths, popup menus become bounded bottom
sheets, tables become single-column labelled cards, the Inspector and project/Assistant surfaces use
one active modal layer, project actions remain discoverable without hover, and primary mobile actions
use the 44 px target contract. Splitters support keyboard focus, cancellation, and touch/pointer
cleanup. Metric grids reflow to one column, chart viewports scroll rather than forcing page overflow,
and reduced-motion rules suppress the remaining live control, Assistant, pulse, blink, and cursor
effects covered by the source regression.

**Six-theme contrast.** The theme layer now separates decorative status color from text ink through
`--working-text`, `--ok-text`, `--fail-text`, `--best-text`, `--accent-text`, and `--alarm-text`, plus
filled-control foreground tokens. Report/Trust, Assistant errors, resource states, chart labels, and
semantic icons consume those tokens rather than light-theme-hostile literals. An automated WCAG
contrast calculation evaluates all six selectable palettes on `bg`, `bg-1`, `bg-2`, `bg-3`, filled
controls, and the actual tinted Report/Trust chip recipes at **4.5:1 or greater**.

The adversarial pass caught implementation defects that a source-only checklist would have missed:
runtime references in Bars/Gantt, unnamed SVG output, nested interactive controls, focus on the
DataTable wrapper instead of its scroll region, CSV formula injection, disabled menu items receiving
initial focus, one Escape closing two nested layers, a hidden modal remaining semantically active,
focus loss after deletion, compact menus clipping off-screen, undefined semantic variables, a white
chart hover stroke on light themes, stale cross-tab Energy state, and Report/Trust-scoped ink values
that overrode the otherwise safe light-theme tokens. Each was corrected before this disposition was
updated.

Automated evidence for the final Block 6 tree is **250/250 UI tests passed**. It includes real SSR
rendering of every non-empty analytical chart, migration checks for every legacy table, an actual
jsdom nested-dialog focus/restore test, route/layer/source regressions, CSV safety, native-link
gestures, the six-theme contrast matrix, and a focused axe-core WCAG 2 A/AA fixture covering
`DataTable`, `ChartFrame`, and representative charts with **zero serious or critical violations**.
Because jsdom has no layout or Canvas implementation, axe color contrast is disabled in that fixture;
the independent computed contrast matrix covers the declared token/surface recipes instead. The
production build also passes; exact current asset evidence is recorded in §20.1.

The boundary remains explicit. There is no full mounted-app or production-Browser axe project, no
screen-reader, switch, voice, real-touch, Firefox/Safari, safe-area/mobile-browser-chrome, or real
200%/400% zoom pass. The six-theme result is computed token evidence, not a six-theme screenshot or

§20.2 remains earlier production-Browser evidence and is not relabelled as validation of this build.

### 18.16 NOTIFY-01 / UX-HITL-01 / Block 7: owner Attention Center and exact approval identity

Block 7 adds an owner-plane Attention Center for the loaded owner application. A fixed bell exposes a
badge, **Needs action** and **Recent** groups, unread state, dismiss, mark-all-read, partial/stale source
status, and bounded cursor pagination. Run attention and Assistant permissions are polled as independent
sources: a failure of one retains that source's last safe snapshot without erasing a healthy result from
the other. The Review route returns before this component is mounted, and the shared/review Assistant
surfaces do not mount it. This is an active-owner-tab inbox, not a background notification service.

The feed has the following deliberately narrow truth table:

| Signal | Authoritative condition | In-app | Desktop | Destination |
|---|---|---:|---:|---|
| Result approval | accepted request still names the exact live node id and attempt | Yes | Yes | generation-fenced node/run |
| Incomplete approval | a pending request has no safe current lifecycle identity | Yes | No | Events; never a guessed approval |
| Spec approval | accepted ratification request exists and the spec is not confirmed | Yes | Yes | generation-fenced spec context |
| Failure spike | a causal upward crossing to at least three current, non-ignored failed lifecycles | Yes | Yes | Failures and one current lifecycle |
| Developer-crash pause | the accepted system pause names an exact node attempt | Yes | Yes | Failures; an ordinary user pause stays quiet |
| Terminal run failure | durable `run_finished(error/leakage/no_eligible_candidate/stuck:*)` | Yes | Yes | failure evidence and recovery |
| Budget reached | required finalization marker is durable and the engine lock is definitively released | Yes | Yes | Report |
| Finished | required finalization marker is durable and the engine lock is definitively released | Yes | Yes | Report |
| Intentionally stopped | aborted/finalized wrap-up is durable and the engine lock is released | Yes | No | Report |
| Stalled / finalization stalled | derived liveness with a definitively absent engine after a grace interval | Yes | No | Events |
| Assistant permission | unexpired opaque request/session identity | Yes | Yes | exact Assistant session, opened observe-only |

#### Exact approval identity

The previous shortcut could substitute `best_node_id` even when the outstanding decision concerned a
different experiment or an earlier attempt of a reused node id. `pendingApprovalTarget`, `RunView`, the
empty-DAG action, `AssistantBar`, bare server control, and bare CLI/Boss approval now require the exact
pending subject and node generation. An explicit CLI/Boss node override may select a different existing
lifecycle, but only while an approval gate is active; it is bound to that selected attempt, the run
generation, and the admitted gate sequence rather than falling back to current best. Missing, mismatched,
reset, tombstoned, or aborted lifecycles fail closed to Events. Spec ratification likewise requires an
accepted request and a proposal frozen at the human-review boundary. The Assistant rereads current state
before submit and binds both node attempt and run generation; durable command admission records the
approval-gate sequence, and the worker rejects a request if that gate changed before execution or before
the compare-and-append decision.

Attention actions are observational. A run card reconstructs its own allow-listed, generation-fenced
fragment destination; a permission card opens only the exact opaque Assistant session. Neither path
approves, ratifies, replays, resumes, or calls a mutation endpoint. A direct visible run/spec
**Approve / Ratify** control and unified non-Assistant confirmation component are still missing, so
UX-HITL-01 is narrowed rather than completely closed.

#### Privacy, consistency, and cross-tab invariants

- `GET /api/attention` is owner-plane, `no-store`, cursor-paged, and bounded to `limit=1..200`; its
  projection reads append-only logs directly and does not call run-list reconciliation, start a process,
  acquire a command slot, or append an event. A bounded 8,192-entry cache covers the documented 5k-run
  operating target without becoming unbounded.
- The response is a fixed allow-list. It never forwards raw event `data`, goal, error, path, code,
  prompt, provider response, or Assistant action/scope/preview. The server emits only fixed,
  kind-derived title/detail text; the client deliberately ignores those fields, owns all rendered copy,
  and reconstructs destinations from validated identity fields rather than accepting a server link.
- An opaque run item id hashes `run_id + run generation + accepted causal seq + kind`. Unrelated appends
  cannot rotate it, a copied run directory cannot alias it, and a replacement generation cannot reuse it.
  Folded request/terminal/finalization anchors prevent duplicate or rejected events from minting alerts.
- A moving or corrupt log marks the feed partial. If a prior verified projection exists, only its
  event-derived cards remain visible as **Last verified**, with desktop delivery and derived liveness
  disabled; otherwise that run is omitted. An unreadable run root returns 503 so the client keeps its
  last safe snapshot instead of accepting an authoritative empty inbox.
- Engine liveness is tri-state. Terminal success and derived recovery appear only after an exact
  `False`; permission errors, unsupported locking, symlinks/reparse points, and uncertain metadata are
  unknown rather than “engine stopped.” Mutations, deletion, reset, and spawn also fail closed on
  unknown liveness.
- `localStorage` contains only schema version, enabled/armed timestamps, and opaque id/timestamp pairs.
  Each acknowledged/dismissed/notified set is bounded to 8,192 identities and 90 days, namespaced by API
  prefix, strictly parsed, and verified by read-after-write. It contains no run/session/action payload.
- All storage mutations and notification claims use the same per-prefix Web Lock when available, so
  concurrent owner tabs cannot overwrite a dismiss/read with a notification claim. The claim is saved
  before presentation: constructor failure may lose one visual alert but cannot produce a duplicate.
  Without Web Locks, desktop delivery and enabling fail closed; in-app read/dismiss remains a
  best-effort local preference.
- `BroadcastChannel` accepts and emits only `{type:"invalidate",v:1}`. Receiving tabs reload and validate
  their own storage rather than trusting cross-document payload.
- Notification permission is requested only by the explicit **Enable** click. Enabling baselines all
  currently loaded eligible ids and records a current-time delivery boundary, preventing a backlog
  burst when older pages are loaded. Desktop title/body are generic, and click handling only focuses
  and navigates/opens the inbox.

#### Accessibility and presentation boundary

The bell and compact actions retain 44 px targets. The trigger exposes `aria-expanded`,
`aria-controls`, and `aria-haspopup="dialog"`; the modal drawer uses the shared focus trap, Escape and
focus restoration, native links/buttons, named sections/lists, a polite baselined live region,
reduced-motion rules, mobile full-width layout, and safe-area padding. Focused component automation is
recorded in §20.1. It is not a claim of full-app WCAG conformance.

At least one owner page must stay loaded. Polling is approximately every four seconds and background
tabs may be throttled; there is no service worker, Web Push, post-close delivery, email/chat routing,
or cross-device sync. Champion-change is not yet a signal. Read/dismiss/preferences are local to one
browser profile, origin, and API-prefix namespace. “Owner” is a deployment plane, not human identity:
token-enabled deployments default-deny this endpoint to review credentials, but a shared deployment
token still supplies neither attribution nor RBAC. No fresh production-Browser, full-app axe,
screen-reader, switch/voice, real-touch, zoom, Safari/Firefox, or two-real-tab pass is claimed for the
final Block 7 asset.

The server cache removes repeat folds for unchanged logs, but a changed active log still requires a
full read/fold plus corruption scan. Cursor pagination bounds each response/increment, not that
per-change scan cost. Loaded pages currently accumulate without a hard client-retention/DOM cap;
indexed/cursor server reads plus bounded or virtualized client retention remain PERF-02/scale work.

Tri-state liveness is observational, not an atomic destructive lease. Current owner-plane
delete/reset/trace-rewrite paths refuse inconclusive locking and serialize server workers, but a
standalone legacy CLI does not join that sequencer/lifecycle protocol: it can acquire the in-run
`engine.lock` after the server's final free probe and before the destructive filesystem mutation.
Machine-run tools also reuse a best-effort lock helper that can degrade unlocked. Closing this P1
requires a permanent strict sibling lease outside the run directory, held by every current CLI engine
for its lifetime and by each destructive transaction, with the legacy in-run lock retained for
mixed-version detection. Windows deletion must keep that sibling lease while closing in-directory
handles and removing the directory. `LOOPLAB_ALLOW_UNLOCKED_WRITER` may remain an explicit engine-start
escape hatch but must never authorize destructive UI/Assistant mutations. This lease protocol and its
cross-process/unsupported-lock/Windows tests are explicitly not implemented in Block 7.

### 18.17 COLLAB-01 / Block 8: exact-attempt, event-sourced comments

The collaboration surface now supports a deliberately bounded decision log rather than treating a
share link or legacy annotation as a complete collaboration model.

#### Durable lifecycle and authority

- Create, edit, resolve, and reopen are append-only `comment_created`, `comment_edited`, and
  `comment_resolution_changed` events. Edits never replace prior text; owner history pages project
  every retained version in reverse chronological order.
- The server stamps the actor. An authenticated deployment owner is labelled **Deployment owner**;
  tokenless/local operation is **Local operator**. Client-supplied actor or comment identifiers are
  rejected. These labels identify a deployment plane, not a human principal.
- Every mutation is fenced by the exact run generation, experiment ID, experiment attempt
  (`node_generation`), and current comment version. A stale run returns `run_generation_changed`; a
  recycled node lifecycle returns `comment_subject_changed`; a lost update returns
  `comment_version_changed` with only the bounded safe current version. Modern IDs remain strict
  `cmt_<32 lowercase hex>` values. Synthetic `legacy_<seq>` annotations are lookup-only and can never
  enter a persisted mutation.
- Mutations use the same durable idempotent command service as run control. Cross-process append uses
  the strict event-log lock; unrelated tail growth is reprojected and retried under the lock, while a
  changed subject/version fails closed. Ambiguous or retryable failures re-arm the same command ID;
  the UI never creates a second logical edit merely because transport status is unknown.
- The contract is intentionally bounded: valid non-empty plain text is at most **8 KiB UTF-8**;
  projection admits at most **500 comments per run**, **100 per exact node attempt**, and **50
  versions per comment**. Owner current/history pages are cursor-bound to generation and filter scope,
  expose at most **100 rows**, and reject stale/tampered cursors. At version 50 mutation controls
  disappear but owner audit history remains visible.

#### Read models and interaction

- Owner APIs expose the current run-wide or exact-attempt feed and per-comment history. Projection is
  fenced before and after folding so a reset cannot relabel generation-A comments as B. Responses are
  `no-store` and vary across owner/review credentials.
- A summary or evidence review capability receives only the recursively redacted current comment
  projection through `/api/review/comments`. It has no history endpoint, authoring controls, or
  mutation route. Reviewer data is always immutable even if a future caller forgets a redundant
  `readOnly` prop.
- Normal state/SSE does not broadcast comment text or history; it carries only `comments_revision` as
  a refresh signal. The client replaces a same-run feed immediately when generation changes rather
  than showing an old page while the replacement request settles.
- Inspector has an exact-attempt **Comments** tab; the independently lazy **Comments & sharing** panel
  provides a run-wide open/resolved/all feed and link management. Deep links carry exact generation,
  node, node attempt, Comments tab, and comment ID, and reject a different current node attempt. Drafts survive CAS
  conflict and generation-safe refresh; hostile HTML is rendered as text; keyboard submission,
  byte-count feedback, resolution/reopen, paged history, retry-same-command, and accessible resource
  states have mounted contracts.

The remaining gap is identity and threaded collaboration, not the basic lifecycle: there is no human
login/RBAC, reviewer attribution, replies, mentions, Markdown, comment export, external notification,
or archival/compaction UI. Legacy annotations retain unknown actor/attempt identity and are read-only.
The 50-version cap is an explicit abuse/storage bound, not an audit-retention policy for a regulated
multi-user service.

Primary files: `looplab/events/comment_projection.py`, `looplab/serve/routers/collaboration.py`,
`looplab/serve/routers/reviews.py`, `looplab/serve/run_commands.py`, `CommentsThread.jsx`,
`CollabPanel.jsx`, `useComments.js`, `commentsModel.js`, `Inspector.jsx`, and `RunView.jsx`.

  closures, subtracts only proven baseline assets for interactions, rejects missing selectors/imports
  and out-of-`dist` paths, and prints the exact forbidden import chain. The checked-in Node 22 CI job

The reconciled production build transformed **270 modules**. Measured totals are **934.1 KiB JS raw /
307.5 KiB gzip** and **157.2 KiB CSS raw / 31.9 KiB gzip**. The initial shell is **204.1 KiB raw /
64.7 KiB gzip**, satisfying the stricter §12 target of 170 KiB gzip as measured, not merely by its

is **102.7 KiB gzip**. The total JS gate has only about **2.5 KiB** headroom, so it
is a regression tripwire, not a licence to keep adding chunks.

#### Command observation

- `CommandObservationIndex` keeps a process-local LRU for eight event logs. One locked refresh scans
  only a recoverable suffix, copy-on-write indexes exact command intents/acknowledgements plus finish
  and abort facts, and hands every helper in a monitor iteration the same immutable revision. Torn
  tails resume from their last valid boundary; shrink, replacement, metadata change, and a matching
  bounded content-probe change rebuild the index. Engine ACK emission likewise retains a
  process-local exact event cursor.
- An unchanged observation reports **0 event-scanner bytes**. Physical validation still reads the
  whole file up to 12 KiB or three 4-KiB start/middle/end sentinels for a larger file. Event-byte
  parsing is therefore `O(delta)` plus a bounded probe, and indexed fact lookup is `O(1)`; this is not
  a claim that the entire refresh is `O(delta)` or that polling performs zero I/O.
- Folded state and incomplete-finalize projection are cached within a revision, but a new revision
  still materializes/folds the complete valid prefix once. Copy-on-write intent/ack/finish maps and
  the chunk tuple can also copy `O(total indexed facts/chunks)` pointers on an append, so one-event-
  per-poll workloads can accumulate quadratic in-memory bookkeeping even though disk parsing remains
  suffix-only. A single process-wide lock serializes cold scans across the eight cached runs.
  Process restart or LRU eviction is a cold scan. A same-size rewrite on a large file that preserves
  metadata and misses all three sentinel windows is theoretically invisible; rewrite-plus-growth of
  an old prefix is also outside the append-only writer contract. These limits are explicit; the
  implementation does not claim arbitrary in-place snapshot safety or end-to-end `O(delta)` work.

The split/budget result is build- and manifest-validated, not a runtime Web Vitals result. No network
waterfall, parse/TBT/interaction p95, slow-link cache test, or fresh Blocks 8–9 production-Browser
journey was completed. Portfolio-scale Map/home and incremental state folding remain separate work.

`vite.config.js`, `ui/scripts/check-bundle.mjs`, `.github/workflows/tests.yml`,
`looplab/serve/command_observation.py`, and `looplab/engine/orchestrator.py`.

open rather than hidden behind that narrower claim. The shared table/chart, keyboard/layer/focus,
responsive-target, reduced-motion, and computed six-theme semantic-contrast contracts are also now
implemented and regression-tested.

The owner attention loop is now visible, redacted, generation-safe, and observational: its links open
the exact evidence or Assistant session and never execute a decision. That materially improves the
“launch and walk away” journey, but it is not a background notification service or a multi-user
collaboration/RBAC system, and it does not replace the still-missing direct run/spec decision control.

Collaboration itself is no longer only a bearer-link builder or an unversioned annotation: exact
experiment attempts now have append-only create/edit/resolve/reopen comments, version conflict
handling, bounded owner history, review-safe current projection, deep links, preserved drafts, and
generic deployment actor labels. This materially improves review and decision traceability. It still
does not identify a human, authorize roles, provide threaded discussion, or define regulated
retention, so the identity/RBAC conclusion above remains unchanged rather than being papered over.

Delivery is no longer one monolithic route payload. The initial shell, owner routes, run surfaces,
panel hub, comments, and React Flow have explicit lazy boundaries and enforceable gzip/reachability
budgets; the current initial shell is 64.7 KiB gzip. Command monitoring likewise avoids reparsing the

## 23. The gap: Part IV shipped, the UI never saw it

Part IV shipped a whole concept-graph subsystem — per-node/hypothesis concept tags, concept concentration +
uncovered-region signals, action-space lock-in, capability-expansion (`expand` operator), graded-dedup
overrides (levels 4/5), the R1-c verifier tie-break, opt-in R1-d leader-CI-band ties, and concept consolidation — plus the offline diagnostics
(`concept-coverage`/`lock-in`/`board-dedup`/`research-targets`/`novelty-recall`). A three-agent audit
(2026-07-15), followed by a domain-model inventory and adversarial UX pass, found the React UI renders
**none** of it. A naming collision hides the gap: the UI narrates the legacy `coverage_snapshot` but not the
Part-IV `concept_coverage_snapshot`.

The UI's current “theme” is the flat, single-value `idea.theme` string (DAG grouping,
`DirectionsOverview`, run-summary chips, and the `coverage — N themes` Dock line). The concept subsystem was
built to supersede that flat vocabulary **for concentration and coverage analytics**; it is not a drop-in UI
grouping replacement. A raw theme still expresses an agent-authored primary direction and partitions a run
into one lane per experiment. A concept assignment is derived, multi-label, may be multi-parent, and answers
a different question. Treating either as the other would recreate the false single-parent taxonomy Part IV
was designed to remove.

**Three-tier data reality (decides scope of every fix).** `GET /api/runs/{id}/state` + the SSE stream serialize
the full `RunState` and its drop-list (`serve/appstate.py`) excludes no folded Part-IV field, so the client
already receives `node_concepts`, `node_concepts_at_vocab`, `concept_coverage_snapshots`,
`hypothesis_concepts`, `hypothesis_concepts_at_vocab`, `concept_consolidation`, `novelty_grades`,
`Node.verifier_score`, `Node.operator=="expand"`, and `select_verifier_tiebreak` — and ignores them. This is
enough for a **qualitative per-run observability layer**: settings, event summaries, raw concept-id chips,
tag-based highlighting, verifier/dedup caveats, and exact latest-snapshot facts.

It is **not** enough for a faithful per-run Concept Map. The folded state does not serialize the graph
manifest: canonical labels/definitions, all multi-parent edges, key/reference flags, taxonomy revision or
digest, assignment source/tagger/version/confidence/rationale, importance receipts, or explicit coverage

semantics and silently flatten curated multi-parent concepts. A full per-run graph therefore needs a small
read-only `ConceptFrame` projection (§30); it is not a pure UI fix.

The third tier is the *derived cross-run* analytics (portfolio coverage/lock-in/dedup/targets). A **lean**
projection now exists in engine/CLI and at `GET /api/cross-run/atlas` / `claims`, with operator overlays at

Neither is the Research Atlas contract specified here: there is no scope/snapshot/filter/cursor request contract, no `CoverageFrame`,
no comparison or evidence-family receipts, no index-health/degraded states, and no stable `concept_uid` or

the alias overlay, but production responses do not carry a release-pinned concept entity UID and raw per-run

shipped, while the faithful B1 Atlas backend **and** React surface

The detailed concept review resolves and refines the earlier sketches:

- **Do not overload the Map or Cross-run Sweep with concepts.** Run List / Map stay *operational navigation +

- **Naming.** Relabel current “Project” → **Collection/Folder** (an owning Project = ACL/budget/data boundary

- **No anonymous graph edges.** Lineage, taxonomy hierarchy, typed relation, consolidation, assignment,

backend exists.

**Still open before implementation.** Choose the assignment rationale/confidence enrichment level; define

## 26. Truth model, terminology, and primary information architecture

### 26.1 Three maps, three kinds of edge

The product needs three coordinated representations, not one universal graph:

| Representation | Question answered | Canonical objects / edges | What position may mean |
|---|---|---|---|
| **Experiment Lineage** | What was derived from what, and what happened next? | experiment attempts; parent and merge lineage | search progression / deterministic DAG layout |
| **Direction lanes** | How did the run move between its agent-authored primary directions? | one raw `idea.theme` label per experiment; lineage bridges across lanes | lane membership + experiment order; **not** semantic similarity |
| **Concept Map** | Which methods/assets/constraints were assigned, how are they related, and where is evidence? | versioned concepts, multi-parent/typed concept edges, assignments | explicit topology only; proximity is never evidence unless labelled |

Lineage edges, taxonomy edges, assignment edges and inferred similarity must never become anonymous lines on
the same canvas. A concept selection may highlight experiments in Lineage; it does not redraw their parents.
A multi-tag experiment appears once. A multi-parent concept has one identity and exposes every parent in the
relationship table; a bounded canvas may omit edges deterministically and disclose `+N omitted`. This avoids
both node cloning and the arbitrary “primary concept” shortcut without breaking the graph budget.

### 26.2 User-facing vocabulary

| Term | UI meaning | Explicit non-meaning |
|---|---|---|
| **Direction** | human-legible compatibility label backed by `idea.theme` | not a governed concept or cross-run identity |
| **Axis** | top-level concept family such as loss/data/evaluation | not a metric axis and not necessarily single-parent |
| **Concept** | precise node in a versioned vocabulary | not automatically a claim, hypothesis, or result |
| **Assignment** | recorded relation from an exact experiment/hypothesis to a concept | not proof that the concept caused the outcome |
| **Study** | owner-defined many-to-many research programme across runs | not the legacy theme or single folder |
| **Candidate gap** | no observed assignment for a suggested/reference region within a disclosed frame | not proof that nobody tried it or that it will win |
| **Concentration** | share/frequency of observed tags in a named denominator | not breadth or coverage |

During migration, headings should say **Directions (legacy labels)** once per surface, then use “Direction”
locally. Raw ids remain visible in provenance. “Topic,” “theme,” “direction,” and “concept” must not rotate as
stylistic synonyms.

### 26.3 Per-run navigation

Keep Report as the conclusion surface and make one compact lens selector own the research canvas:

```text
Run workspace
  Research map:  [Lineage] [Directions] [Concepts]
  Report

Lineage    → existing experiment DAG; default on every run
Directions → stable legacy-direction lanes + Direction×Concept crosswalk
Concepts   → Landscape | Intersections | Journey | Focused map
```

This is one “research map” with coordinated selection, not four more global navigation tabs. The current
`DirectionsOverview` chips remain a compact Lineage filter and shortcut into Direction detail. Concepts is
enabled only when assignments or an honest feature-status explanation exists; old logs retain Directions as
a first-class fallback.

Canonical route state includes run generation, folded sequence, lens/layout, selected experiment attempt,
direction, concept UID/raw id, axis, axis membership (`exact` or `descendants` at a pinned release), concept
filter operator (OR/AND), and ConceptFrame/taxonomy revision.
History mode uses the same prefix for Lineage, assignments and snapshots. It must never combine a historical
DAG with current tags or taxonomy. Live data arriving after the displayed frame produces **Updates available**;
the user refreshes explicitly so the map and focus do not jump underneath them.

## 27. Per-run Research Space interaction concept

### 27.1 Signal strip: summary and doorway, not the whole feature

The strip answers “is there anything research-space-related that needs attention?” and opens the full lens.
Its stable order is freshness → observed focus → gap → current narrowing:

```text
Research signals · as of experiment 48 · LLM tags
Most frequent tag: loss/decoupled-contrastive
3 suggested gaps · no observed assignment at this snapshot
Recent focus: loss · current streak 7     [Open research space]
```

With a complete `ConceptFrame`, the first line may add exact facts such as `61/67 eligible experiments
tagged` or `34 of 61 tagged experiments touch DCL`. The current compact snapshot cannot supply those
denominators, so Horizon A omits the percentage rather than presenting `top_concept_frac` as coverage.

The strip uses a neutral/info tone for incomplete or suggested gaps. Alarm styling is allowed only when a
fresh server projection supplies the governing rule, threshold, complete frame and fired decision. It renders
current `recent_axis/current_streak` separately from longest-ever `locked_axis/streak`. “No data” opens a
status explanation; it never silently disappears or claims the feature is off.

### 27.2 Lineage lens: causality remains untouched

The existing DAG remains the default and sole owner of parent/merge causality, experiment status colour,
champion, work-in-progress and exact attempt actions. Concept integration is deliberately light:

- a selected concept adds an outline/glyph to every assigned experiment and dims non-matches; it does not
  replace status colour or change the edge set;
- multi-select defaults to explicit OR; AND is a labelled toggle with a live result count;
- “Show in Concepts” opens every assignment for the exact selected attempt;
- “Show experiments in Lineage” returns from a concept/gap with generation, sequence and selection intact;
- tombstoned/historical assignments are excluded by default and available through one labelled history
  filter, never mixed into current totals silently.

Concepts must not be added to the existing `GROUP_MODES`. Its `id → group` contract is single-membership;
forcing a concept into it would undo the core Part-IV model.

### 27.3 Directions lens: a stable temporal graph, not a semantic force layout

Directions is a compatibility and narrative view for `idea.theme`. Use horizontal experiment progression
across deterministic swimlanes whose order freezes at first discovery:

```text
Direction                 experiment order →
baseline        ● #0 ────────────────┐
loss/DCL               ● #4 ─ ● #7 ─┼─ ● #12
hard negatives              ● #6 ───┘
data                                     ● #15 ─ ● #18
unassigned                         ◇ merge / bridge
```

Each lane header contains the raw label, distinct experiment count, evaluated/failed/pending mix, first/last
touch, and best **associated** direction-aware result within this run. Counts of related concepts are marked
“overlap”; a
multi-label experiment contributes once to the lane total but may contribute to several concept counts.
Champion, selected, active and merge junctions never auto-collapse. Cross-lane parentage remains an explicit
bridge. Lane proximity and order do not imply semantic similarity.

Lane selection filters Lineage and opens a Direction Inspector with its experiments, hypotheses, associated
concepts and “Open crosswalk” action. New runs may place this under a “Raw labels” disclosure after the
ConceptFrame is established; old logs keep it prominent. Do not invent edges between directions from label
similarity.

### 27.4 Direction × Concept crosswalk: the migration view

A matrix is more truthful than a theme force graph for explaining how one legacy direction decomposes into
many concepts:

| | Concept A | Concept B | Concept C | Assignment status |
|---|---:|---:|---:|---|
| Direction 1 | 8 | 3 | — | 10/12 experiments tagged; partial |
| Direction 2 | 1 | 7 | 4 | 9/9 tagged; current frame |
| Unassigned | — | 2 | 1 | 2/5 tagged; partial |

Each cell is the number of **distinct co-tagged experiments**, never causal effect or a metric sum. Row and
column totals disclose overlap; cells distinguish zero in a complete frame from unknown/partial. Selecting a
cell filters exact experiments and hypotheses, then offers Lineage and Concept Inspector. Columns initially
roll up by axis; an explicit **Expand** control replaces an axis with its concepts. Wheel or pinch zoom changes
geometry only and never loads or changes the data set. Every axis selection says either `exact` or
`descendants · release R`, and cells deduplicate by experiment/assignment UID across multi-parent paths. This
crosswalk is also the clean migration audit: it
shows whether raw directions are too fragmented, too broad, or simply untagged without pretending they are a
taxonomy.

### 27.5 Concept Landscape: structured overview is the default

The Concepts lens opens to a stable, searchable axis board / tree-list rather than a force graph. Initial
content is about 10–40 relevant axes/concepts; everything else is behind search, filters and `+N related`.

Each concept row/card shows:

- canonical label + raw id, kind/status and every parent-axis chip;
- distinct assigned experiments and active hypotheses;
- evaluated/failed/pending **associations**, not a causal “concept score”;
- first/last observed touch and assignment freshness;
- reference/suggested gap source and completeness, when applicable;
- exact snapshot/revision and whether assignments are heuristic, model-derived, operator-confirmed or
  unavailable.

Priority ordering is deterministic: selected/current-focus → referenced candidate gaps → touched concepts →
untouched concepts → stale/unknown. Metric ranking is never the global sort because a concept can occur with
other concepts and cannot inherit the whole experiment outcome causally. Filters cover observed, candidate
gap, stale/partial, failed-associated, champion-associated and hypothesis-linked states.

### 27.6 Intersections: make Part IV's axis-clique insight visible

An axis×axis co-occurrence matrix is an expert subview because Part IV's decisive concentration signal was a
`loss × regularization` clique that flat themes hid. A cell reports distinct experiments assigned to both
axes, share of a named tagged denominator, first/last touch and associated outcome distribution. Its sum is
not the number of experiments because multi-label membership overlaps. Each axis independently declares
`exact` or `descendants · release R`; the intersection deduplicates by experiment/assignment UID across
multi-parent paths. Selecting a cell opens the exact experiments, constituent concepts and “Show in Lineage.”
Unknown/partial assignments never become zero.

### 27.7 Focused Concept Map: topology on demand

The visual graph opens only after concept selection. Its topology node set contains **concepts only** and its
edges are typed concept relations. Experiments stay as count badges plus list/drawer evidence; one selected
experiment may appear as a non-topological context card outside the layout, never as fan-out. Default is one
hop and about 30 visible nodes; the hard budget is 40 visible nodes and 80 edges. Deterministic per-relation
quotas decide which edges fit; each truncated relation exposes `+N omitted`, and the synchronized relationship
table remains logically complete through server filtering/cursor paging, row/byte caps and virtualization — it
is never preloaded in full. A deterministic left-to-right layout places hierarchy parents/axes, the selected
concept, then children/typed neighbours. The **hierarchy layer** is a DAG; the full typed-relation layer is a
general graph and may contain cycles. Existing coordinates remain stable until the user refreshes the pinned
frame.

| Relation | Visual treatment | Default |
|---|---|---|
| hierarchy / parent | solid directed edge + text label | on |
| `uses` / `requires` | labelled directed edge with distinct dash/marker | on when present in focused hop |
| `alternative_to` / `conflicts_with` | labelled symmetric treatment with different marker/pattern | on when present |
| consolidation / alias | thin alias link in Changes/provenance, not normal topology | off |
| assignment | evidence count badge + drawer/list; do not fan hundreds of experiment nodes into the map | off as edges |
| co-occurrence | dashed analytical overlay with denominator and “observed together” label | off |
| embedding similarity | “suggested similarity” overlay with model/version and Why; never taxonomy truth | off |

Every edge has “Why related?” and exact provenance. A concept node never uses one green/red fill to summarize
mixed outcomes. Identity comes from label/kind; selection from focus ring; topology from labelled edge; state
from icon/pattern/border; exact evidence stays in the Inspector. Click or Enter only inspects a neighbour;
**Focus here** explicitly re-roots the map. Re-rooting preserves a breadcrumb/back stack, camera, filters and
the pinned frame. Expand/Collapse changes the fetched topology explicitly; geometric zoom never does.

Before `ConceptFrame` exists, this tab is replaced by “Canonical structure unavailable” plus the honest
assignment table. Prefix-derived axis lanes may appear only as **reconstructed**, never as the engine's real
multi-parent DAG.

### 27.8 Journey: how the search narrowed and pivoted

Journey is a temporal projection of the same frame: X = experiment order; Y = axes. Its axis selector is
explicitly `exact` or `descendants · release R`, and every rollup deduplicates by experiment/assignment UID.
Consecutive activity is aggregated into stable segments, with markers for candidate-gap appearance/clearance,
current narrowing, `expand`, L4/L5 dedup override and verifier tie-break. An explicit **Show attempts** / Expand
control reveals individual experiments; wheel or pinch changes geometry only. One multi-label experiment
remains one selectable object spanning/touching several lanes; it is not cloned and is counted once in run
totals.

Journey answers “when did the search narrow or pivot?”; Landscape answers “what is observed now?”; Focused
Map answers “how is this concept related?” The three are coordinated projections, not redundant chart modes.

### 27.9 Contextual Inspector and safe actions

Inspector changes subject cleanly: Experiment / Direction / Axis / Concept / Gap. Concept detail includes
definition, all parents/relations, assignments, active hypotheses, first/last touch, associated outcomes,
freshness, source and proof. Gap detail starts with epistemic type — suggested, reference, untouched axis,
unknown — then shows why it matters, provenance, denominator/completeness and prior evidence.

Allowed research actions are read-only navigation, compare, copy a snapshot-fenced link, and **Draft
hypothesis**. “Explore this gap” creates a cited draft for confirmation; it never mutates the active run or
steers the next proposal immediately. Taxonomy promote/merge/split lives only in Governance with preview and
audit. The scientific view cannot turn a convenient visual edit into new engine truth.

### 28.1 Shell and default composition

Atlas is a dedicated scientific workspace, not a mode of Map, Sweep or Memory. Its desktop composition is:

```text
[Saved Scope predicate] [widening policy] [as-of snapshot] [freshness / index health]

[Concept navigator]   [Landscape: virtualized concept × scope matrix]   [Proof drawer]
                      [Findings | Evidence | Changes]                    [Compare tray]
```

The shell states visible/eligible/indexed counts separately: `N visible runs · M measurements · K comparison
groups · profile 93% · assignments 88%`. It pins corpus watermark, taxonomy release and access-filter digest.
A Saved Scope stores the full predicate plus membership preview; “folder/supertask” is never substituted for
scope. When the index advances, the displayed snapshot stays fixed and offers refresh/diff.

### 28.2 Landscape is matrix-first

The default scientific overview is the sparse, virtualized `concept × scope` matrix from doc 17 §21.20.7.
Rows expand explicitly axis → concept → variant; geometric zoom never changes the row set. Axis chips and
rows say `exact` or `descendants · release R`, and rollups deduplicate by experiment/evidence UID across
multi-parent paths. Columns are Study, comparison group, or a selected scope slice — never thousands of raw
runs. The first screen shows gaps, contradictions, partial indexing and recent change before decorative
topology.

A cell announces and displays concept, scope, coverage phase, evidence design, distinct evidence-family/run
count, frame quality and snapshot. Clicking it filters **Evidence** to the exact intersection. Exact
Matrix→Findings filtering additionally requires a revisioned applicability edge:
`ClaimApplicability {claim_uid, claim_revision, concept_uid, taxonomy_release, assignment_revision,
scope_snapshot_uid, comparison_contract_uid, access_digest, relation, evidence_digest,
applicability_revision, provenance}`. Until that projection
exists, Findings remains a separately scoped search and may only offer clearly labelled suggested matches; it
is not part of the cell's proof chain. A zero is allowed only when the eligible CoverageFrame is complete and
its deduplicated count is zero; unknown, partial, stale, not applicable and restricted remain different values.
Restricted counts themselves may be hidden if policy treats existence as sensitive.

The bounded Focused Concept Map from §27.7 is available beside the matrix for a selected concept. Scope affects
badges/evidence, not taxonomy geometry. No action renders every run and concept in one global force graph.

### 28.3 Findings, Evidence and proof

**Findings** presents scoped ClaimAssessments and role-routed lessons: statement, applicability, maturity,
supported/mixed/refuted/inconclusive, independent support versus opposition, direct/ablation/confounded
composition, freshness and completeness. The deterministic default is an explicit **Needs review** order:
current pinned rows, then non-rejected mixed, inconclusive, supported and refuted rows; within each group use
operator maturity, evidence freshness and stable `claim_uid`. Any task-specific re-ranking requires a visible
`DecisionContext` plus a ranking receipt; raw LLM confidence is never the hidden sort.

Two lean data-model facts landed 2026-07-15 that Findings can reuse carefully. (a) Claim evidence refs are
**run-qualified** `"run:node"` strings (`"?:node"` when the run is unknown). A known-run ref addresses a run
and node but still lacks generation fencing; `"?:node"` is unresolved and cannot be an exact proof link.
Moreover `n_support`/`n_oppose` count refs, not independent evidence families; correlated attempts can dominate
and consolidated `evidence_count` is currently lost. The UI must label these as attempt references until the
independence contract lands. (b) Lesson reads have role filters, but automatic Researcher/Strategist advisory
is portfolio-wide and the Repo Developer provider is not state-bound. UI capability copy must not claim
scope-safe role isolation until the fail-closed scope gate in §33 passes.

(c) As of `b0be675`, every claim now carries an operator **`maturity`** (`machine-proposed` |
`operator-ratified` | `operator-rejected` | `operator-pinned`), served on `GET /api/cross-run/claims` and the
Atlas payload. Findings must render these four states as first-class, distinct badges — a `machine-proposed`
claim is never shown as settled and `operator-ratified` is only a latest overlay marker — the current HTTP
write records actor `ui` and blank time, so it is not yet an attributable/audited human vouch. A rejected row
may be struck/collapsed in the human view; chronological history requires a separate API. The incoming fixes
now exclude rejected claims from the active context pack, Atlas contradictions, agent tools and hybrid
retrieval, closing the previously confirmed steering leak. The POST still has no clear/reset action, evidence
revision/expected-version check or history response, so the future UI must not promise undo, attribution or
conflict safety.

**Evidence** is cursor-paged by comparison group/evidence family before individual run/attempt. It shows
contract compatibility, objective direction/unit, exploratory vs confirmatory role, trust exclusions,
cost/convergence/reproducibility and exact links to run/node/code/log. The browser never fetches every run
state to compute the table.

The proof drawer is shared across matrix, concept and claim selection, but it never invents the missing
claim↔concept applicability edge:

```text
statement / selected concept×scope
→ applicability and CoverageFrame
→ aggregation / comparison contract
→ supporting and opposing independent evidence families (for a claim subject only)
→ exact measurements and experiment attempts
```

“Why recalled?” adds exact/adjacent facet matches, concept path, quality/freshness, exclusions, retrieval
model/version and snapshot. A summary with no valid proof stays metrics-only/inconclusive rather than filling
the drawer with generated prose. **CR2a ships only a lean precursor (2026-07-15, `41ef1e6`):**
`engine/claims.py::cross_run_retrieve` runs a hybrid query (lexical + BM25 + default hash-vector channel) over
claims + concepts, exposed as the `cross_run_search` agent tool and `cross-run-search` CLI. The engine/CLI
receipt contains query/requested-k, corpus/hit counts and a channel list, and result rows carry a fused score;
the agent tool renders hit text but drops both scores and receipt. The engine result does **not** carry
per-hit channel contributions/reasons, evidence refs or source, scope/comparison, snapshot/index health,
eligibility/exclusions, model/version or persisted consumer attribution. A bound agent search also passes
scoped lessons but currently reloads portfolio-wide capsules. Therefore this is experimental recall, not the
proof-drawer “Why recalled?” contract; the full intent/eligibility planner and derivation receipt remain gated.

### 28.4 Compare without false ranking

Compare holds at most three subjects on desktop and two below 900 px and uses aligned columns/cards, not
overlaid graphs. It has two valid modes: **one concept across several scopes**, or **several concepts inside
one scope**. A comparison that changes both axes belongs in a separate factorial view with an explicit
`ComparisonContract`; the normal tray rejects it. Each column fixes definition/revision, CoverageFrame,
support/opposition, independence, freshness and cost. Raw
metric rank appears only under one compatible ComparisonContract; normalized effect appears only under an
explicit shared baseline/effect definition. Otherwise the view remains qualitative and names the exact
incompatibility (metric, direction, split, evaluator, dataset lineage, constraints or objective).

Concept filter selection is explicit OR/AND. Study/Scope comparison does not silently widen. “Use in new run”
creates a draft hypothesis/context pin with citations and source snapshot; it never imports a portfolio
conclusion as established local truth.

### 28.5 Changes and Governance

Changes is the temporal answer to “why does the Atlas look different?” It groups:

- taxonomy release, rename/merge/split/re-parent/retirement, plus separate `assignment_revision` changes;
- added/removed/revised assignments and changed completeness;
- run re-evaluation, attempt generation, data/environment/trust drift;
- claim-assessment changes and newly conflicting evidence;
- current versus prior snapshot with affected concepts/scopes/claims.

Taxonomy/claim/scope **Governance** remains a separate role-gated workspace. Merge/split/re-parent/definition
operations create a taxonomy release and require dry-run impact (assignments, coverage, claims, routes,
backfill cost), optimistic base revision, accept/reject, compensating publish and audit. “Revert” always
publishes a new compensating release that points to the superseded release; it never edits or deletes history.
Correcting an assignment to an existing concept creates an independent `assignment_revision`, not a taxonomy
release. A split may leave explicit `unresolved historical assignment` rows for review rather than rewriting
history automatically. Atlas exposes health and links to Governance but never embeds raw index controls or lets
a drag gesture rewrite ontology. Assignment CAS, Changes and route subjects all use the same
`assignment_revision` term.

**Lean claim-decision overlay shipped (2026-07-15, doc 17 §22.4).** Agents have no mutation tool and the POST
is covered by the configured UI-token middleware. The persisted mechanism is a portfolio sidecar,
`claim_decisions.jsonl`, not a per-run `CONTROL_EVENTS` event. That is a valid first seam, but it is not yet a
governance workbench: the key is normalized statement text truncated to 160 characters; decisions can target
nonexistent/future claims and cross scopes. The append now uses the shared interprocess lock, but it remains a
non-fsynced last-write-wins overlay with caller-supplied blank time/anonymous `ui` actor; there is no evidence
revision, CAS, idempotency, reset, history API, or stale
decision state. `pinned` changes a badge but has no defined ordering behavior. Product copy must call this a
**lean decision overlay**, not reversible/audited governance. Full Governance requires structured `claim_uid`,
revisioned decisions, durable fsync, actor/time/audit history, impact preview and separate human-history versus
agent-safe projections.

### 28.6 Theme/Direction boundary across runs

Legacy Direction is run-local provenance, not a cross-run join key. Atlas may show it in an exact run drawer
or as migration diagnostics, but portfolio aggregation uses stable concept UIDs, versioned revisions, Studies
and scopes. Similar strings from two runs remain unrelated until an explicit assignment/alignment says
otherwise. This prevents a “DCL” label collision from silently merging different definitions or scopes.

## 29. Visual, state, accessibility, responsive and scale contract

### 29.1 Orthogonal state grammars

The UI must preserve separate state axes in text, icon/pattern and accessible name — never hue alone and never
one overloaded `status` field:

| Grammar | Values | Applies to | Required presentation |
|---|---|---|---|
| **CoveragePhase** | not-observed · assigned · attempted · evaluated | a concept×scope measurement frame | highest reached phase plus counts; never a success or truth implication |
| **EvidenceDesign** | N/A-not-yet-evaluated · direct · ablation · confounded · unknown-design (multi-valued when mixed) | eligible evidence in the frame | design badges/counts, separate from phase and epistemic stance; unknown means design metadata is missing, not that evaluation has not happened |
| **FrameQuality** | complete · unknown · partial · stale · N/A · restricted | the selected CoverageFrame | reason, bounds/freshness/access without count leakage; zero is not a quality value |
| **ClaimAssessment** | supported · mixed · refuted · inconclusive | one scoped `claim_uid` | stance glyph/text and supporting/opposing proof; never applied to the Concept itself |
| **OperatorMaturity** | `machine-proposed` · `operator-ratified` · `operator-rejected` · `operator-pinned` | one claim revision | display labels may shorten to Machine/Ratified/Rejected/Pinned; wire values stay canonical and independent of epistemic stance |
| **DecisionFreshness** | N/A-no-operator-decision · current · stale-evidence · conflicted · unknown | one operator decision versus current claim revision | separate warning/action; machine-proposed with no decision is N/A, never unknown; freshness never changes maturity or claim stance implicitly |
| **ConceptLifecycle** | active · aliased · merged · split · retired | one taxonomy revision | kind/lifecycle shape/fill; no green/red outcome colour |
| **RelationLayerStatus** | not-produced · partial · current · stale · failed · restricted | one typed-relation producer/release | feature-status message/toggle availability; never masquerades as concept lifecycle or “no relations” |

Suggested and reference gaps are annotations over CoveragePhase + EvidenceDesign + FrameQuality: suggestion
names source/why/completeness; reference gap names the governing release. They are not additional epistemic
truth states. A displayed zero is the derived predicate `FrameQuality=complete ∧ eligible denominator known ∧
deduplicated count=0`; no other state renders `0`.

“Failed-only” means every **associated** evaluated experiment failed under its own multi-change configuration;
it does not prove the concept is refuted. Verifier `0.73` means “advisory tie-break score 0.73” under the named
effective mode (exact metric tie by default; leader CI-band tie when enabled), never “73% correct.” A missing
score means not invoked. `novelty_graded` reads “dedup override allowed: new
implementation” or “failed direction retried,” never a universal novelty grade; absence is not low novelty.

### 29.2 Visual channels

- Experiment status retains the existing semantic colour budget. Concept focus uses outline, glyph and
  dimming rather than repainting the card.
- Axis/concept identity is primarily label + kind icon. Low-chroma axis tint may aid scanning but carries no
  unique meaning.
- Selection uses a focus ring; current work/champion retains its existing marker; gap/partial/stale use
  different icon/pattern/border combinations.
- Node area and edge thickness never imply importance unless a visible legend names the transform and exact
  value. Evidence counts use bounded badges or a documented log scale.
- Tooltip content has focus/touch equivalents and the proof drawer contains every exact value. The graph is
  never the only representation.

### 29.3 Accessible equivalent and keyboard model

Every Concept Map has a synchronized typed relationship table listing every parent/child/relation; only its
hierarchy subset is a DAG. A tree/treegrid may repeat identity-linked references under multiple parents, but
each reference retains one `concept_uid`, shared selection and “also under …” context and is never counted
twice. Every matrix is a semantic grid with sticky headers, `aria-rowcount`/`aria-colcount` and correct
`aria-rowindex`/`aria-colindex` for virtualized cells, plus a card/list alternative. Journey has a synchronized
chronological segment/event table. Each
alternative is first-class: same filters, selection, links, actions and proof drawer.

Keyboard requirements: typeahead search; arrows between semantic neighbours; Home/End and Page movement in
virtualized lists/grids; Enter opens Inspector; Escape returns focus; OR/AND and relation-layer controls are
native controls. Graph selection mirrors in the linked list. Live updates announce only meaningful deltas
(gap changed, current narrowing changed, refresh available), not every assignment event. Focus remains stable
when rows virtualize or a snapshot refreshes.

Acceptance includes full-app axe, keyboard-only, real screen-reader, switch/voice spot checks, focus/touch
equivalence, 200%/400% zoom and reduced motion. Reduced motion disables animated transitions and forced
pan/re-fit; no force layout runs visibly.

### 29.4 Responsive behaviour

| Width | Default composition |
|---|---|
| ≥1280 px | navigator + matrix/canvas + docked proof drawer; compare tray |
| 900–1279 px | collapsible navigator; full canvas/matrix; proof drawer overlays with focus trap |
| <900 px | list/tree first; graph is opt-in full-screen; proof is full-screen sheet; compare is bottom bar |
| ~360 px | Directions become accordions, Concept Map becomes relationship list, Atlas becomes Concepts → scopes/cells → proof; no page-level horizontal overflow |

The signal strip collapses to one summary button on narrow screens. Any internally scrollable matrix retains
sticky labels and a labelled scroll region. Touch targets are at least 44 px in compact mode; no MiniMap or
hover-only legend consumes the small canvas.

### 29.5 Scale and live stability

- Focused graph: about 30 nodes initially, at most 40 visible concept nodes / 80 typed edges; deterministic
  per-relation quotas, `+N omitted` and the logically complete relationship table are required. That table is
  server-filtered/cursor-paged and virtualized with explicit row/byte caps; “complete” never means load-all.
- Landscape: roughly 10–40 initial axes/concepts. Default to 12 selected columns and enforce a hard maximum of
  24 per view/request; hidden columns are not prefetched and require server-side selection or cursor paging.
  Rows are server-filtered/cursor-paged; publish response-row, response-byte, DOM-node and overscan ceilings.
  Two-axis virtualization must preserve stable keyboard focus and the ARIA row/column indices above; evidence
  is cursor-paged.
- Atlas queries are server-filtered and never fetch all run states. Graph/Atlas code remains a lazy chunk.
- Pin the reference host/browser/network and corpus dimensions. Targets: time-to-first-data and query p95,
  response byte/row caps, p95 local select/filter <100 ms, no input/scroll main-thread task >50 ms, bounded
  DOM/canvas objects and an explicit memory ceiling. The 500-run/50k-evidence tier must publish thresholds,
  not merely be described as “interactive.” Debounce/AbortController cancels obsolete requests; every response
  is keyed by `{scope_snapshot,snapshot,taxonomy,assignment_revision,comparison_contract_uid,access_digest,
  request_subject_or_query_digest}` and any response for a prior tuple is discarded.
- Live assignment/snapshot events update a pending-delta badge. Existing graph coordinates and focused object
  do not move until explicit refresh; refresh preserves stable ids and focus where possible.
- No automatic fit-on-every-SSE, global force settle, or semantic reordering by changing metric is allowed.

## 30. Data-readiness and truth contract

### 30.1 What the current state can and cannot support

Without backend work, Horizon A can safely render raw/canonicalized chips, qualitative latest-snapshot facts,
`expand`, verifier copy naming exact-default versus optional leader-CI-band mode, dedup-override events, assignment-based filters and a clearly labelled
current experiment/hypothesis↔concept-slug table. It can derive a provisional prefix-axis crosswalk only if it
says **reconstructed** and exposes its limitations.

It cannot truthfully render canonical multi-parent topology, untouched canonical concepts, definitions/kinds,
typed edges, graph history, assignment rationale/confidence, taxonomy revision, complete tag/coverage
denominators, cross-run identity or causal concept effectiveness. Copying the Python dense-retrieval skeleton
into the browser is explicitly rejected: it would omit dynamic concepts, drift from the engine and create a
second taxonomy implementation.

Two qualifications (2026-07-15). (a) The per-run **cross-run-prior** signal — which similar prior runs touched
a concept and with what outcome — *is* served today via `RunState.cross_run_priors` and is Horizon-A-renderable.
As of the later 2026-07-15 CR slice, a cross-run *aggregate* also exists as an engine/CLI/HTTP lean read-model
(`portfolio_concept_overview` per-concept run lists, `portfolio_atlas` explored/thin/contradictory payload,
`claim_assessments` support/oppose with run-scoped evidence) — so the aggregate is no longer *unbuilt*. The
incoming CR1a/CR1b/CR2a slice adds a deterministic slug-derived `concept_uid` helper, append-only alias/purge
overlay, persisted D8 research claims, opt-in fuzzy grouping, bounded hybrid retrieval and a one-level prefix
digest. Alias-aware reads make a better **diagnostic** view than raw-slug-only joins, but they return canonical
display slugs rather than the UID and their run cards remain raw. The helper encodes normalized spelling rather
than a versioned taxonomy release; fail-closed scope/comparison, eligible evidence, snapshot/health/completeness,
pagination and evidence-family receipts remain absent. Thus “cannot render cross-run identity” means the
release-pinned, scope-safe product graph, not the diagnostic summary, alias overlay or advisory chip. (b) The
engine currently emits **no typed lateral concept relations** (`uses`/`requires`/`alternative_to`/…) — only the
multi-parent *axis* hierarchy — so the `relation_type` edges of the §30.2 `ConceptFrame` and the typed-relation
rows of §27.7 require a **new producer** (a tagger extension), not merely serialization of existing state.

### 30.2 Minimum per-run `ConceptFrame`

This is a conceptual read model, not a new source of truth or an implementation in this review:

```text
ConceptFrame
  identity
    run_id, run_generation, state_sequence, as_of_node
    graph_revision, graph_digest, taxonomy_release, task_type

  feature_status
    effective_flags, disabled_reason, last_success, last_error
    assignment_completeness, current_vocab_size, backfill_pending

  concepts[]
    concept_uid/raw_id/canonical_id, label, definition, kind, status
    parent_uids[], aliases[], reference_kind, provenance

  edges[]
    source_uid, target_uid, relation_type, lifecycle, provenance

  assignments[]
    subject_type, subject_id, node_attempt/generation
    concept_uid/raw_id/canonical_id, taxonomy_release
    source, tagger/model/version, confidence?, rationale/receipt?
    at_vocab, stale, event_sequence

  importance[]
    concept_uid, suggested/reference/operator source, why, derived_at

  consolidations[]
    raw_id, canonical_id, reason?, confidence?, event_sequence, release

  measures
    named numerator/denominator/universe/completeness for each measure
    current and historical narrowing with threshold/fired provenance
```

Optional confidence stays absent/“not calibrated” unless the producer actually records it. Stable route
identity uses UID + release; raw slug remains provenance. Current aliases resolve to one canonical node without
double-counting, while historical labels and the release under which an assignment was made remain inspectable.

The per-run projection is sequence/generation-fenced like Inspector/Report history. A node `propose` reset can
invalidate its tags; an `implement`/`eval` rerun may keep them; tombstoned nodes remain historical. Exact
assignment identity therefore includes node attempt/generation and event sequence rather than only numeric
node id. Graph evolution and consolidation need release/history records; vocabulary size alone cannot detect a
same-size semantic change.

### 30.3 Measures must remain separate

| Measure | Honest definition | Current compact snapshot? |
|---|---|---|
| tag completeness | eligible experiments with ≥1 assignment / all eligible experiments | no (`tagged` absent) |
| axis reach | applicable axes touched / applicable axes in a named graph release | no totals |
| reference reach | declared important concepts touched / concepts in that reference set | gap list only; provenance/why incomplete |
| top-concept concentration | experiments touching the top concept / tagged experiments | fraction recorded, denominator count absent |
| current narrowing | consecutive/latest activity on `recent_axis` under a named rule | `current_streak` present; threshold/fired absent |
| historical concentration | longest same-axis streak and top tag over the run prefix | present as summary facts |
| intersection concentration | co-tagged distinct experiments / named tagged denominator | not in compact snapshot; derivable only with explicit frame |
| cross-run evidence coverage | eligible evidence under Scope/Comparison/CoverageFrame contracts | not available; requires full CR contracts, and the lean summary is insufficient |

The UI never collapses these into one “research coverage score.” If product later needs a composite, it must be
a separately versioned, explained advisory model with every component still visible.

### 30.4 Part-IV provenance caveats

- `node_concepts` are raw pre-consolidation ids; current display applies `concept_consolidation`, keeps aliases
  in proof, and deduplicates after canonicalization.
- `node_concepts_at_vocab` / hypothesis equivalent are vocabulary-size freshness hints, not taxonomy revisions.
- A heuristic aggregate snapshot may exist without per-node assignments; the UI calls this “aggregate only.”
- `fired` in the current snapshot belongs to the uncovered-region signal, not lock-in. `locked_axis/streak` is
  longest-ever; `recent_axis/current_streak` is current narrowing.
- An uncovered key may be curated or LLM-suggested. The current snapshot loses `important[].why`; the UI cannot
  invent a reason or call it proven.
- `Node.verifier_score` folds the result while `n_samples`/agreement/evidence digest stay in the atomic
  `verifier_group_scored` event; full proof must project that treatment. Replay accepts only the complete
  selector-reachable group under the pinned sample/contract policy. It affects selection only inside exact
  metric ties by default or the conservative leader/pooled-SE band when `verifier_ci_tie` is effective; it
  never overrides a significant better metric.
- `novelty_grades` contains only L4/L5 allow overrides with prospective node ids, not every novelty decision.
  A durable node badge requires exact proposal/attempt linkage; otherwise the record stays in Journey/Timeline.
- `expand` records a proposal operator. Success must be assessed from later evidence, never inferred from the
  operator label.

## 31. State matrix and concept-validation contract

### 31.1 Required empty/degraded states

| State | User-facing explanation / action |
|---|---|
| disabled | feature is disabled for this run; show where defaults apply if the user can configure them |
| awaiting first cadence | enabled, no snapshot yet; explain when the first frame can appear |
| unsupported / no vocabulary | task has no available graph and no model-built frame; keep Directions/Lineage |
| tagger degraded/failed | preserve last valid frame, show timestamp/error/retry policy; never substitute empty |
| heuristic fallback | label assignment source and limitations; no model/operator confidence implied |
| aggregate only | snapshot summary exists but exact experiment assignments do not; disable graph drill-down |
| partial/backfill | show tagged/eligible completeness and pending work; absence remains unknown |
| stale | show displayed vs current release/sequence and explicit refresh/backfill action |
| no eligible experiments | explain lifecycle and keep relevant settings/history actions |
| complete, no candidate gaps | positive neutral state with frame/release; not “100% research complete” |

Absence of `concept_coverage_snapshots` cannot identify which row applies because folded state does not persist
all effective flags or producer failures. Horizon A therefore says “No concept snapshot recorded” and links to

### 31.2 Core usability questions

The concept is successful only if a user can answer, without interpreting raw JSON:

1. What did this run try, and what is only proposed/unassigned?
2. How does one legacy Direction decompose into concepts?
3. Which exact experiments/hypotheses are assigned to this concept, under which revision and source?
4. Where is current narrowing versus a historical streak, and did an `expand` attempt actually help later?
5. Which gap is suggested/reference/unknown, why is it important, and how complete is tagging?
6. Which axes co-occurred, and is an apparent outcome direct or confounded?
7. Cross-run, is this concept supported/opposed in an exact or adjacent scope, under a compatible contract?
8. Why was this evidence recalled, what is withheld/stale, and what changed between snapshots?

### 31.3 Acceptance scenarios

Validate the concept against fixtures containing: a 100-node live run with multi-label/multi-parent concepts;
a successful pivot after a historical lock-in; aggregate-only heuristic fallback; partial/stale re-tagging and
consolidated aliases; old/disabled/unsupported runs; an L4/L5 override with prospective identity; an exact
metric tie with verifier provenance; tombstoned/re-run attempts; restricted evidence; incompatible min/max
contracts; contradictory scopes; and 50-run/5k-node plus synthetic 500-run/50k-evidence Atlas tiers.

Acceptance criteria:

- current narrowing and the highest-priority qualified gap are findable in ≤10 seconds in a moderated
  prototype task;
- Concept → exact experiment evidence and Gap → source/why/proof take at most two interactions;
- a multi-tag experiment is reachable from every assigned concept but counted once in run totals;
- no zero appears without a complete eligible frame; unknown, untouched, partial, stale, N/A and restricted
  are understood distinctly in comprehension testing;
- no participant interprets top-concept concentration as breadth/coverage after reading the label;
- legacy Direction is never called a concept, and concept filtering never changes lineage causality;
- owner/audit historical links reproduce the same generation/sequence/release for at least the source-data
  retention period; share-capability expiry/revoke and source retirement are explicit and never substitute
  current state; editing a selection/tray/scope creates a new immutable route-object UID;
- the bounded graph, table/list equivalent, 360 px flow, keyboard route and screen-reader output expose the
  same facts and actions;
- verifier is always described as advisory: exact metric ties by default, or leader-CI-band ties when effective, and never overrules a statistically significant better metric; suggested gap is never called a proven winner;
- scale/performance budgets in §29.5 pass without all-run-state fetches or unbounded DOM/canvas growth.

### 32.2 Decisions fixed by this review

- Keep Experiment Lineage, Direction lanes and Concept Map separate but coordinated.
- Default per-run to Lineage; default Atlas to matrix-first Landscape; make topology bounded/focused.
- Use Direction×Concept crosswalk rather than a new theme force graph.
- Do not add concept as a single-membership DAG grouping mode.

### 33.6 Refined information architecture and graph concept

#### Canonical routes and return context

The existing run **Search** becomes the single canonical **Research** container; old Search links redirect/alias
to it. Its secondary lens replaces the central canvas and lens-specific controls — it does not add another
top-level tab or second lineage canvas. Lineage / Directions / Concepts share one selection subject, Inspector
and return context. In Concepts, DAG-grouping/WhyStrip controls are hidden and Timeline starts collapsed, so the
new lens does not stack more chrome onto the browser-observed dense screen.

The UI needs a versioned, snapshot-fenced route schema before drawing a graph:

```text
#/run/{run_id}?view=research
  &lens=overview|lineage|directions|crosswalk|landscape|intersections|journey|focus
  &gen={run_generation}&seq={state_sequence}&subject={opaque_subject_uid}
  &frame={concept_frame_uid}&release={taxonomy_release}&assignment_revision={assignment_revision}
  &direction_raw={escaped_legacy_label}
  &axis_membership=exact|descendants&concepts={opaque_ordered_selection_uid}
  &op=or|and&layout={layout_version}

#/atlas
  ?scope_snapshot={opaque_scope_snapshot_uid}&snapshot={corpus_watermark}
  &taxonomy={taxonomy_release}&assignment_revision={assignment_revision}&access={access_digest}
  &tab=landscape|findings|evidence|changes
  &subject={opaque_subject_uid}&compare={opaque_compare_uid}
```

Subject selection and the concept filter set are distinct. `subject` resolves to exactly one typed Inspector
subject (experiment attempt, Direction, Axis, Concept or Gap) and is authoritative; `concepts` only filters the
visible set. Legacy `(node,attempt)` is one composite experiment-subject alias: `attempt` qualifies `node` and
is invalid without it. That composite competes with exactly one `direction`/`axis`/`concept` alias; multiple
subject kinds recover to no subject with a visible notice. If `subject` is present it wins. `direction_raw` is
the named temporary fallback only when no
`direction_uid` producer exists and is never a cross-run key. Opaque IDs resolve server-side; raw predicates,
claim text and restricted identifiers never enter the URL. An ad-hoc dirty scope still gets an immutable opaque
snapshot UID after Apply; it is not forced to masquerade as a Saved Scope.

This query-form route is deliberate: the current `App.jsx` parser would treat `/run/{id}/research` as part of
the run ID, while `runRouteState.js` currently accepts only `dag|report` and names its generation parameter
`gen`. R0/R1 must update App parsing,
`runRouteState`, review links and history-route sanitizers **before** emitting Research links. Existing empty
or `view=dag` links canonicalize to `view=research&lens=lineage`; `view=report` remains unchanged. Unknown or
incompatible lens state fails to Lineage with a visible recovery notice rather than losing the run context.

`subject_uid`, `selection_uid`, `compare_uid` and `scope_snapshot_uid` are immutable, revisioned,
lifecycle-managed server objects, not mutable shorteners. Every tray/selection/scope change creates a new UID;
each object binds owner/access policy, canonical content digest, creation/expiry, retention class and revoke
state. Owner/audit objects referenced by persisted reports, comments or governance history are retained at
least as long as their source evidence; expiring/revocable TTL applies to the share capability, not the audit
snapshot. A transient unsaved selection may have a shorter disclosed TTL. Expired/revoked shares resolve to an
explanation plus an explicit fork/rebuild action, while source-data retention expiry says “source retired” and
never substitutes current state. The
`direction_uid` likewise requires a producer contract (run/generation/sequence, immutable raw-label digest and
migration behavior); until then routes carry the escaped raw Direction only as non-cross-run provenance.

Every transition preserves a return target. Report → Research Space returns to the same report section;
Run → Atlas carries the run/task as a proposed scope and displays membership before applying it; an Atlas
evidence row → Run opens the exact node/generation and returns to the prior scope/snapshot. Links with an
expired or inaccessible snapshot show an explicit explanation and offer a new-snapshot fork, never silently
substitute current data.

#### Per-run Research Space

- **Overview:** one decision-oriented strip and three cards — current concentration, qualified gaps, recent
  concept changes — with completeness and `as of` in the same reading line.
- **Direction×Concept crosswalk:** sticky Direction rows, canonical Concept columns, count/state cells and
  a right-side evidence inspector. Axis selections always expose `exact` versus `descendants · release R`, and
  counts deduplicate by experiment/assignment UID. This is the bridge from familiar theme lanes to multi-label
  concepts.
- **Landscape:** multi-parent hierarchy with explicit Expand/Collapse. Start at axes; expand one branch; never render all
  labels. A canonical Concept row exists once in the searchable flat list and exposes all-parent chips. Axis
  branches may show identity-linked references with “also under …”; they share one UID/focus and never double-
  count. Axis rollups overlap and therefore never sum to a grand total. Fill/shape denotes taxonomy kind or
  lifecycle only; focus is a ring; scope-dependent coverage/freshness/access use badges/patterns. A Concept
  never becomes green/red “true/false” from mixed outcomes.
- **Journey:** stable time/attempt axis for first touch, pivot, consolidation and gap-change events. It answers
  “what changed when,” not “which concept is best,” and has a synchronized chronological semantic list/table
  with the same selection, proof links and actions for keyboard/screen-reader/360 px parity.
- **Focused graph:** one selected concept plus parents, children and explicitly typed concept relations. The
  canvas contains concept nodes only; experiments remain badges/list/drawer evidence, with at most one selected
  non-topological context card. Budget about 30 visible nodes by default, 40 nodes / 80 edges hard;
  deterministic per-relation quotas expose `+N omitted`, while the logically complete relationship table is
  server-filtered/cursor-paged, row/byte-capped and virtualized rather than loaded in full. Only
  the hierarchy layer is a DAG; lateral typed relations may cycle. Click/Enter inspects, while **Focus here**
  explicitly re-roots and preserves breadcrumb/back stack, camera, filters and frame. Expand/Collapse fetches
  topology; wheel/pinch zoom never does. Assignment, hierarchy, lineage and suggested similarity never share
  one edge style. Until a versioned lateral-relation
  producer passes its fixture, the state is **relation layer not produced** (not “no relations”): hide that
  toggle and render hierarchy/assignments only.

Assignment repair is a governed review flow, not graph dragging: **Flag assignment** → queue → approve/reject/
reassign with reason → impact diff → new `assignment_revision` and audited backfill.
Taxonomy releases are reserved for merge/split/re-parent/definition changes; a split may leave unresolved
historical assignments for review rather than rewriting them. **Draft hypothesis** /
**Use in new run** opens an editable target-board/new-run preview with snapshot+citation bundle, applicability and
duplicate checks, permission check and explicit confirmation; it is disabled in review-only, expired, stale or
restricted contexts.

#### Cross-run Atlas desktop shell

Use a three-region layout: a sticky scope/snapshot bar; a left concept/saved-scope navigator; a central
virtualized matrix or current tab; and a contextual proof drawer that overlays at medium widths. Default to
**Landscape matrix**, not graph. Keep persistent, human-readable query chips for project/task/dataset/metric/
direction/trust/time/access. A “Why these runs?” receipt opens membership and exclusion reasons.

Saved Scope has a complete draft/apply lifecycle: edit an ad-hoc predicate → policy-filtered membership preview
→ Apply to create an immutable scope snapshot → Save/Save as, rename/delete and owner/share management with
revision/CAS. Editing never changes the applied snapshot until Apply. Refresh shows membership delta; invalid,
expired and conflicted revisions are explicit. “Why these runs?” never reveals restricted existence or counts.
An owner deep link grants no access. A share/review link is a scoped, revocable, expiring capability with
allow-list/redaction, corpus/snapshot/access digest and cache purge on revoke/expiry.

One matrix instance fixes exactly one column dimension (Study **or** comparison group **or** scope slice) and
one CoverageFrame eligibility/comparison definition. Every cell exposes its own named eligible denominator;
ratios compare only when the `ComparisonContract` declares those denominators compatible. Selecting a cell changes one subject
(`concept_uid × scope_snapshot_uid`) and coordinates Evidence and proof without navigation loss. Findings joins
that exact subject only after a revisioned `ClaimApplicability` projection exists; before then it is a separate
scoped search with suggested-match copy. Compare holds at most three subjects on desktop and two below 900 px,
but only as one concept across scopes **or** several concepts inside one scope. Changing both axes requires a
separate factorial view and explicit ComparisonContract. A new subject replaces the oldest only after
confirmation. Scope change revalidates compatibility and removes/marks invalid subjects rather than leaving
them visually comparable. A focused concept map is secondary and never changes taxonomy geometry when scope
changes; only evidence badges change.

The matrix itself has a hard resource contract: 12 selected columns by default, 24 as the hard maximum per
view/request, and no preload of hidden columns; column selection/paging and row filtering/cursor paging happen
server-side. Bounded overscan and measured response-row/byte plus DOM-node ceilings apply at every corpus tier.
Virtualization exposes total/index metadata through ARIA and keeps logical focus stable while cells mount/
unmount; a card/list view is the equivalent, not a reduced fallback.

At <900 px, replace the matrix with an accessible row list grouped by concept, make the drawer a full-screen
detail route, keep scope chips horizontally scrollable and use a bottom-bar subject switcher. Compare becomes
a stacked aligned diff; at 360 px it never relies on side-by-side columns. Canvas topology has a synchronized
relationship table and never becomes the sole route to evidence.

- State axes remain orthogonal: **CoveragePhase** (not-observed/assigned/attempted/evaluated),
  **EvidenceDesign** (N-A-not-yet-evaluated/direct/ablation/confounded/unknown-design), **FrameQuality**
  (complete/partial/stale/unknown/N-A/restricted),
  **ClaimAssessment** (supported/mixed/refuted/inconclusive for a selected `claim_uid` only), and
  **OperatorMaturity** (`machine-proposed`/`operator-ratified`/`operator-rejected`/`operator-pinned`).
  **DecisionFreshness** (N-A-no-operator-decision/current/stale-evidence/conflicted/unknown), ConceptLifecycle,
  RelationLayerStatus and ExperimentStatus
  remain separate again. A concept×scope cell never calls the Concept itself
  supported/refuted. Zero is derived only from a complete eligible denominator with deduplicated count zero;
  unknown/partial/restricted never show 0. Every axis has text/icon/pattern, not colour alone.
- Destructive or meaning-changing governance actions live in a role-gated workbench. The confirmation shows

- Scope/snapshot refresh atomically marks or clears proof, Findings and Evidence before loading the new tuple.
  Responses are accepted only for the current `{scope_snapshot,snapshot,taxonomy,assignment_revision,
  comparison_contract_uid,access_digest,request_subject_or_query_digest}`; stale requests cannot repopulate
  old proof below new scope chips or a newly selected subject.

### 33.8 Final global verdict and highest-leverage UI changes

The concept is ready for design prototyping, but implementation must split into the independent per-run and
cross-run tracks in §33.7. The largest UX improvement is **not a prettier graph**; it is giving each question an
honest surface, one stable selection model and immediate proof without making the operator decode the engine's
storage model.

| Product move | Why it changes the UI materially | Hard boundary |
|---|---|---|
| Rename the overloaded run **Search** area to one **Research** container with Overview / Lineage / Directions / Crosswalk / Landscape / Intersections / Journey / Focus lenses | removes competing pages and keeps node/concept selection, Inspector and return context coherent | do not stack concept topology onto the dense 131-node lineage canvas |
| Reserve one canonical **Atlas** IA/route now; enable it as a first-class global destination only at G5, defaulting to a virtualized concept×scope matrix | prevents diagnostic-route sprawl now and makes gaps, contradictions, partial indexing and comparison groups scannable once the truth contract exists | before G5 expose only a separate Experimental portfolio diagnostic; never default to a global force/DAG graph |
| Use a bounded **Focused Concept Map** only after selecting one concept | preserves topology value without hairballs; typed parent/child/relation evidence stays inspectable | about 30 nodes default, 40/80 node/edge hard budget; relationship table is the overflow path |
| Put a shared **proof drawer** behind every matrix cell, claim and concept | converts visual confidence into traceable evidence and keeps “why” within two actions | no generated explanation without scope/snapshot/comparison/source receipts |
| Treat scope as an applied immutable snapshot with a visible draft/apply lifecycle | prevents filters, refresh and shares from silently changing the evidence universe | no “Why these runs?”, compare or share claim until fail-closed backend scope exists |
| Keep Coverage phase, Evidence design, Frame quality, Claim assessment, Operator maturity, Decision freshness, Concept lifecycle and Relation-layer status visually orthogonal | prevents “green concept = true,” “0 = unknown” and “not produced = no relations” errors | zero is derived only from a complete eligible frame; every state needs text/icon/pattern and an accessible table/list equivalent |
| Add effective-state Settings and contextual launch points from List/Report | makes Part-IV capabilities discoverable without exposing unsafe switches as harmless toggles | show dependency, apply timing, cost/risk and disabled reason; unsafe cross-run paths stay off |
| Make mobile/accessibility an alternate composition, not a squeezed canvas | keeps evidence and governance usable at <900 px/360 px and by keyboard/screen reader | matrix→grouped list, drawer→detail route, compare→stacked diff, graph→relationship table |

The essential missing product functionality is therefore explicit: a sequence-fenced per-run `ConceptFrame`;
server-filtered/cursor-paged `CoverageFrame` queries with scope/snapshot/access/health receipts; Saved Scope
create/apply/refresh/share/revoke; evidence-family proof and compatible compare; Changes/history; governed
claim/taxonomy revisions with impact preview and conflict handling; and a cited **Draft hypothesis / Use in new
run** hand-off. Until those exist, the shipped cross-run slice should appear only as an **Experimental portfolio
diagnostic**, never as the Research Atlas or as verified organizational memory.

### 34.2 Shipped Atlas preview contract

The canonical owner route is `#/atlas`; the legacy `#/research-atlas` alias is replaced in place. Public review
and shared-assistant routes return before the preview can mount, and the owner shell/auth gate wraps the lazy
route. Runs home exposes an explicitly named **Research Atlas preview** entry.

The screen fetches four independent, bounded, `no-store` sources with one abort signal:

1. `GET /api/cross-run/atlas?limit=24`;
2. `GET /api/cross-run/claims?limit=40&offset=0`;
3. `GET /api/cross-run/curation-log?limit=20`;
4. `GET /api/cross-run/claim-curation-log?limit=20`.

It renders summary counts, bounded explored concepts and run links, single-run **thin coverage**, active
contradictions, claim support/oppose/unverified references, and merged concept/claim steward history. Copy says
“Experimental · bounded · read-only,” explains that thin means observed in one run rather than untried, and
states that refresh may observe new live data but cannot mutate claims, taxonomy, runs or selection.

The resource-state contract is materially stronger than a happy-path dashboard:

- loading, memory-not-configured, true unavailable, valid empty, malformed-record and section-empty states are
  distinct;
- `Promise.allSettled` allows a partial first load to render successful slices with an explicit degraded notice;
  a partial refresh replaces successful slices while retaining last-good failed slices, and a total refresh
  failure keeps the prior view marked stale;
- every collection and persisted text field is normalized and capped before React receives it; control
  characters are collapsed, invalid rows are counted, omitted rows/references are disclosed, and run links are
  encoded;
- scrollable evidence regions are keyboard-focusable and route/main/loading/error semantics are named.

This is intentionally a summary/cards view, not a global force graph. The matrix-first target remains correct:
the preview has no Saved Scope lifecycle, frozen corpus/access snapshot, CoverageFrame, compatible Compare,
server-paged exploration, evidence-family proof, interactive concept topology or governance controls.

### 34.3 Backend truth the UI may now state — and what it still may not claim

| Contract | Safe current copy | Still prohibited copy |
|---|---|---|
| D8 evidence | Whenever shared memory is configured, new v2 rows persist independently of the concept-capsule flag and retain task/run/direction identity, run-qualified references, URLs and verifier verdict/method/note; unsupported/legacy/malformed rows stay unverified. | “Every reference is support,” “independent replication,” or “complete evidence.” |
| Agent reads | Bound pull tools require compatible direction. Lessons/capsules allow exact task or a strict two-term/coverage goal fingerprint; v2 D8 is exact-task-only. Task facets are metadata reserved for future ranking and currently affect neither visibility nor order. Genesis is goal/direction-bound and Repo Developer is task-bound. | “Tenant isolation,” “ACL scope,” “same dataset/evaluation,” or “no irrelevant transfer.” |

Concept HTTP actions validate live canonical sources and merge targets; split children may be new provisional entities. The preview correctly chooses not to expose the stronger mutation endpoints. Claim decisions now fence both
claim existence and the reviewed evidence projection. Revision CAS prevents blind governance-ledger overwrite;
neither guarantee by itself supplies taxonomy/assignment or evidence-family releases, release-stable concept
identity and migration/backfill semantics, impact analysis, history/workbench or authorization.

Claim-decision lookup is deliberately layered: exact scope+metric wins, followed by scope-only, global metric
and global. Thus an unscoped operator decision is a portfolio-wide fallback; only a scoped decision is isolated
to its task. This precedence is governance behavior, not an ACL.

### 34.4 Second-review verdict and remaining UX priority

## 35. Reconciled UI concept and final mega-review authority (2026-07-16)

ordering in §§33–34 without erasing their review chronology. It reconciles the independent concept audit with

force graph is not the default for either journey.

### 35.1 One question, one surface

| Operator question | Canonical surface | Visual model | Product boundary |
|---|---|---|---|
| “Which runs exist and where did they come from?” | **Runs · List/Map** | filterable run inventory and project/seed lineage | not a theme, concept, claim or evidence graph |

| “What meaning should change?” | future **Governance workbench** | review queue, impact diff, conflict/history and explicit confirmation | requires release-pinned identity, validated migration/backfill and role/access policy |

The stable mental model is therefore **run lineage → per-run research explanation → scoped portfolio
comparison → governed meaning change**. Navigation may preserve return context between those surfaces, but it
must not merge their node types or imply that one projection answers all four questions.

### 35.3 Preview truth corrections

These wording changes are part of the data contract, not cosmetic polish:

| Misleading shortcut | Current safe label | What the user may conclude |
|---|---|---|
| “Coverage” | **Concept observations / Concepts seen across runs** | the bounded concept-capsule response contains these rows; no eligible denominator or completeness is implied |
| “Thin coverage” | **Observed in one run** | one run is present in this response; it does not mean untried, underexplored or absent elsewhere |
| “Contradictions” | **Mixed-evidence claim records** | both support-labelled and opposition-labelled references are present; applicability and proposition truth are undecided |
| “Supported / Refuted” | **Support-only / opposition-only evidence** | this is a machine-derived evidence balance, not a verdict; operator maturity remains an independent badge |
| “Audit trail / Governance” | **Steward invocation log / Recent proposals + outcomes** | the rows are bounded recorded invocations and outcomes, not the current taxonomy/claim state or a complete audit history |
| an unqualified source total | **showing N of reported N** plus omission/malformed/degraded state | the client rendered a capped live response; the number is not a frozen corpus total |

Direction summaries require the same restraint. A displayed best metric or signed difference is descriptive of
the current run's observations. Until a `ComparisonContract` proves compatible populations and eligibility,
the UI must say **best observed** and **difference from run baseline**, avoid causal “gain/lift/effect” copy,

### 35.5 Graph, mobile and accessibility acceptance

#### Current run DAG

- The 80/48/24 composition budgets remain deterministic and reversible; deep links, selected nodes/groups and
  history context always win over the overview heuristic. Live growth through the threshold never moves the
  camera or changes grouping by surprise.
- Filtered collapsed groups show the active subset and `matched / total`, not an aggregate computed from hidden
  nodes. Empty filtered groups have an explicit zero-result state and a clear way back.
- At 360 px, the page itself has no horizontal overflow. Direction/setting chip rows may scroll locally; long
  run, node and trace identifiers wrap or ellipsize with an accessible full value. Primary targets are at least
  44 × 44 CSS px.

#### Future concept topology

- Default to about 30 visible concept nodes; hard-stop at 40 nodes / 80 edges. Deterministic relation-type
  quotas disclose `+N omitted`; the server-paged relationship table remains the logically complete path.
- **Inspect** and **Focus here** are separate keyboard-operable actions. Re-rooting preserves breadcrumb/back
  stack, filters and camera. Wheel/pinch zoom never fetches or changes scope.
- Hierarchy, assignment, experiment lineage and suggested similarity never share an edge grammar. Colour is
  redundant with label/icon/pattern, cycles are allowed only for explicitly typed lateral relations, and “not
  produced” is distinct from a known empty relation layer.
- Below 900 px the graph is optional; the synchronized relationship list/detail route is first-class. Evidence
  and governance actions are never canvas-only.

#### Cross-surface accessibility gate

- Every selected filter/toggle uses native semantics or an accurate `aria-pressed`/selected equivalent; named
  groups and headings define the reading order. All controls are reachable in a stable keyboard sequence with a
  visible focus indicator.
- Loading, refresh, degraded, stale, invalid and saved/error states use appropriately scoped status/alert live
  regions without announcing every polling update. Focus moves to a route heading or blocking validation
  error, and returns to the invoking control when a drawer/dialog closes.
- Every matrix/graph/chart has an equivalent ordered list or table with the same subject, counts, proof links
  and actions. Unknown, zero, partial, stale and restricted are conveyed by text plus icon/pattern, never colour
  alone.
- Acceptance requires automated semantic/axe tests and manual keyboard + screen-reader checks at desktop and
  360 px. A desktop screenshot or DOM-presence assertion alone is insufficient.
