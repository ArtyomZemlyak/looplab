looplab autonomous ML/DS research engine · defaults view

The agent that invents, builds, tests & improves — on a loop.

Give LoopLab a goal. It runs a closed research loop — a Researcher proposes an idea, a Developer writes the code, a sandbox runs it, an evaluator scores it — then refines and merges the best candidates until the budget runs out, and returns the best verified result. Every step is one line in an append-only event log, so a run is fully reproducible and crash-resumable by replay.

A run, end to end

Six stages from a goal to a verified champion

The middle stage — The Loop — is where the work happens; it repeats until the node budget is spent. Everything to its left sets the run up; everything to its right hardens and remembers the result.

Propose → implement → run → score → gate → pick next. Repeats up to max_nodes (default 8). The heart of the engine.

Inside the loop: four roles, one turn of the wheel

A candidate solution is a node. Each turn expands one node into a child, colored here as it moves through the four roles. Two edges break the circle: a failed run loops back to the Developer to be repaired, and the merge operator fuses two strong lineages into a multi-parent child.

What flows around the wheel

The color advances with the candidate. The Policy hub at the center selects which node to expand next; the optional Strategist ✧ re-tunes policy, operators and fidelity every few nodes.

Repair ↺A crash or timeout is fed back with its stderr. Mechanical faults are fixed in place — no node spent; deeper ones get a "reproduce, then fix" directive.

The four roles, in detail

What each role actually does

By default one LLM identity — the unified agent — plays every role across stages, choosing its model and tools per stage and driving the next action within a legal-action gate that keeps pipeline discipline.

01

Researcher

proposes an Idea = operator + params + hypothesis

Reasons about the goal and everything tried so far, then proposes the next experiment. It's conditioned on injected memory — and can actively pull more with tools.

  • Sees fingerprint-matched lessons, exact-task meta-notes, open hypotheses, and digests of winners & failures
  • Can call tools to read its own experiments, sibling runs, cases, lessons, skills & the knowledge base (researcher_tools · cross_run_tools)
  • Picks an operator: draft · improve · debug · ablate · merge · sweep
02

Developer

turns the idea into runnable code

Emits a complete solution (or applies params to a repo), restricted to the task's allowed edit surface. On a live model it writes the whole script; the toy backend fills a template.

  • Edit-surface gate is reject-not-strip: a patch touching a protected file or escaping the surface is rejected wholesale
  • Every accepted child is a commit — lineage is always reconstructable
  • Can be an external agent (OpenCode / Aider / Goose) in a worktree, diff-gated back to the surface
03

Sandbox

runs the candidate in isolation

Executes the solution under a hard budget and captures everything. The tier is chosen by trust_mode, not your environment.

  • Default trusted_local: a subprocess — process isolation + 30 s timeout + whole-tree kill + ~64 KB output caps. No Docker
  • Runs n_seeds=3 so a score isn't a single sample
  • Auto-installs a known-missing library and re-runs; untrusted tier swaps in Docker --network none
04

Evaluator

decides if the run is a real improvement

Turns a run into a gated verdict — never a bare number. Leakage-first, variance-aware, so noise and cheating don't crown a candidate.

  • Validity → objective on a consistent CV protocol: K-fold or purged walk-forward (no look-ahead)
  • Variance gate: beat the incumbent by more than ~1 standard error, or it's noise
  • Held-out partition (holdout 0.25) selects the robust best; leakage & reward-hack monitors are available as audit gates
The supporting cast

The subsystems around the loop

The loop is small on purpose. These subsystems make it honest, resumable, and smarter over time — each is a swappable seam wired from config.

Orchestrator & Event Log

The sole writer of run state. It drives the loop, owns the budget, and appends every step to events.jsonl. Replay folds the log into state with a pure function — identical every time — so a hard-killed run resumes from the exact frontier.

orchestrator.pyeventstore.pyreplay.py

Trust & Evaluation

The leakage-first rigor layer: consistent cross-validation, train/test·temporal·target leakage detectors, the variance gate, and multi-seed confirmation at the frontier. Reward-hack, code-leakage and critic monitors run as audit events.

cv.pyleakage.pygate.pyconfirm.py

Search & Operators

The policy decides where to expand; operators decide what each step does — and operators do the heavy lifting. Default is a greedy tree with multi-parent merge; ASHA, MCTS and evolutionary policies are drop-in.

policy.pyoperators.pysweep.pyarchive.py

Memory & Knowledge

Both stores are on by default. Across runs the engine accumulates cases, causal meta-notes, lessons, skills and a knowledge base — injected into prompts and agentically retrievable. Memora indexes them by abstraction + anchors and consolidates duplicates.

memory.pyretrieval.pymemora.pyvectorstore.py

LLM Backend

One call surface over any OpenAI-compatible endpoint — Ollama, vLLM, SGLang, OpenAI — a base_url change, not a code change. Per-role and per-stage model routing, structured output via tool-calls with a text fallback, and cost accounting.

llm.pyparse.pyunified_agent.py

Observability & UI

Every step emits a trace span to spans.jsonl (LLM prompts + completions captured by default). A live React control plane and a terminal TUI read the same files and append command intents — no engine coupling, no write races.

tracing.pyserver.pyhtmlview.pytui.py

Two properties everything rests on

Why the event log is the keystone

↻ Reproducible by replay

looplab replay RUN_DIR folds the event log into the current state with a pure function — no side effects, identical result every time. Because the task and config are snapshotted into the run dir, a run is fully self-describing.

⏻ Crash-resumable, exactly

Hard-kill a run at any point; looplab resume RUN_DIR replays the log, reconstructs the exact frontier, and continues — no duplicated or lost work. The append-only writer even tolerates a torn final line from a mid-write kill.

Out of the box

What's on by default — and what's dormant

A plain looplab run ships lean and honest. The full agent stack is enabled but dormant until you switch backend=llm; the heavier trust gates stay off until you ask for them with profile=thorough.

The honest read: a stock run is deterministic and offline — it proves the loop, the event log, replay and crash-resume without a single API call. Point it at backend=llm and the Researcher, Developer, Strategist, agentic retrieval and self-repair all come alive; add profile=thorough and the frontier rigor (multi-seed confirmation + trust gates) switches on in one word.
